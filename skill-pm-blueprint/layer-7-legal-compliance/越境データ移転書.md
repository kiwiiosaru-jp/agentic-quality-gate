# 越境データ移転書

## 対応する品質評価ID

- **LEGAL-CROSSBORDER-001 (Major)** - 越境データ移転の判定と移転根拠が未整備
- 関連: LEGAL-PII-001、COMPLY-PIPA-JP-001、LEGAL-DATACLASS-001

## 概要

プロジェクトで取扱う個人情報・機密データの**越境移転**を網羅的に可視化し、個情法第28条 (外国にある第三者への提供の制限) および GDPR / CCPA 等への適合性を整備する。

このスキルの目的:

1. **データフローの可視化** - 国境を越える経路の全洗い出し
2. **越境根拠の特定** - 「本人同意」「基準適合体制」「十分性認定国」のいずれか
3. **移転先の安全管理措置の確認** - 移転先国の個情法/SCC/BCR 等の確認
4. **メタデータ越境への対応** - ログ、監査記録、暗号鍵の越境チェック
5. **DPO 承認** - データ保護責任者による事前承認

## いつ使うか

- 矢羽①業務要件定義の中盤 (Week 4-6)
- クラウドサービスのリージョン選定時
- 新規 SaaS ベンダー導入時
- データ転送経路変更時

## 個情法第28条の要件

外国にある第三者へ個人データを提供する場合、以下のいずれかが必要:

### 要件 A: 本人同意

- 提供先の国名
- 提供先国の個人情報保護制度の内容
- 提供先で講じる個人情報の保護のための措置
- 上記を本人に情報提供してから同意取得

### 要件 B: 基準適合体制

- 提供先が **OECD プライバシーガイドライン相当**の体制を有すること
- 標準的契約条項 (SCC: Standard Contractual Clauses) などで担保
- 個情委届出は不要だが、契約書類の準備が必要

### 要件 C: 十分性認定国

- 個情委が認定した国に所在する第三者への提供
- 2025年時点: EU 加盟国、英国
- 認定があれば、本人同意・SCC は不要

### 要件 D: 法令・委託・事業承継・共同利用

- 本人同意なくとも例外として認められる場合
- 詳細は個情委ガイドライン参照

## 「越境」の判定基準

### 越境とみなされるケース

- データのリージョン (保管場所) が外国
- データへの**アクセス可能性**が外国にある第三者にある
- 例: Snowflake のリージョンは Tokyo でも、サポートが US から接続可能ならば越境
- 例: SaaS のテナントは Japan East でも、ベンダー社員が US から DB アクセスするケース

### 越境とみなされないケース

- 同一法人内 (本社⇄海外支社) は「第三者提供」に該当せず、別途「外国にある親法人」のルール
- データを暗号化し、復号鍵を国内に保持する場合 (実質的アクセス不可)
- 個人を特定できないよう仮名化したデータ (一定条件下)

## 手順

### ステップ 1: データフロー図の作成

全データフローを以下の表記で可視化:

```
[取得元 (国)] → [一次保管 (国)] → [加工 (国)] → [配信先 (国)] → [削除]
```

各矢印に**通信経路** (TLS/SSL/専用線) を併記。

### ステップ 2: 越境ポイントの抽出

データフロー図から、国境を越える矢印を全て抽出。各矢印について:

- 越境元国 / 越境先国
- 越境するデータ項目 (PII inventory ID と紐付け)
- 越境の必要性 (なぜ国内処理ではダメか)
- 越境根拠 (要件 A/B/C/D のいずれか)

### ステップ 3: メタデータ越境のチェック

本体データ以外で越境しがちなもの:

| メタデータ                  | 越境発生例                                      |
| --------------------------- | ----------------------------------------------- |
| アクセスログ                | クラウドベンダーの中央集約ログ (US リージョン) |
| 監査ログ                    | SaaS のグローバル監査ログ                       |
| 暗号鍵                      | KMS のグローバル鍵管理                          |
| バックアップ                | DR 用クロスリージョンレプリケーション           |
| パフォーマンステレメトリ    | APM SaaS のグローバル集約                      |
| サポートチケット            | サポート対応中の画面共有・ログ送付              |

これらは見落としがちなので、ベンダーの**データ処理付録 (DPA: Data Processing Addendum)** を必ず確認。

### ステップ 4: クラウド設定の確認

| クラウド | 確認項目                                                   |
| -------- | ---------------------------------------------------------- |
| Azure    | リージョン (Japan East/West)、Storage の geo-redundancy    |
| AWS      | リージョン (ap-northeast-1)、S3 Cross-Region Replication   |
| GCP      | リージョン (asia-northeast1)、Multi-region storage         |
| Snowflake | アカウント所在リージョン、Replication 設定                |
| Databricks | Workspace リージョン、Unity Catalog 設定                  |

### ステップ 5: DPO 承認 → 法務承認

- データ保護責任者 (DPO) が技術的妥当性を確認
- 法務部が法的根拠を確認
- 経営層 (CISO/CPO) 最終承認

### ステップ 6: 本人同意取得が必要な場合の対応

- プライバシーポリシーへの追記
- Web フォーム/契約書の同意条項
- 同意取得記録の保管 (個情法第29条)

## 例示

### データ分析基盤 (DWH/Lakehouse) のケース

| 経路                                                  | 越境             | 移転データ      | 根拠         | 備考                            |
| ----------------------------------------------------- | ---------------- | --------------- | ------------ | ------------------------------- |
| Web フォーム (JP) → Azure SQL (Japan East)            | なし             | -               | -            | 国内完結                        |
| Azure SQL (JP) → Snowflake (US-West-2)                | あり             | 顧客 PII 集計用 | 不適合 (要対応) | 同意取得 or リージョン変更    |
| Snowflake (US) → BI ダッシュボード (Japan East)        | あり (戻り)      | 集計済み (PII なし) | 適合 (PII なし) | 個人特定不可なら問題なし     |
| Azure SQL (JP) → DR バックアップ (East US 2)           | あり (メタ含)   | 全データ        | 不適合 (要対応) | リージョンを West Japan に変更 |
| Azure Monitor → Microsoft 中央集約ログ (Global)        | あり (メタ)      | アクセスログ    | 適合 (DPA で担保) | 標準的契約条項で OK         |

### コールセンター (VoC 分析) のケース

| 経路                                                            | 越境       | 移転データ            | 根拠           | 備考                           |
| --------------------------------------------------------------- | ---------- | --------------------- | -------------- | ------------------------------ |
| 通話 (JP) → 通話システム (Japan East)                           | なし       | -                     | -              | 国内完結                       |
| 通話システム → 外部 ASR ベンダー (US East)                      | あり       | 音声データ            | 不適合         | 国内ベンダーに変更検討         |
| ASR 結果 → AI 要約 (Anthropic Claude API、US-Region)            | あり       | 文字起こし (PII 削除済) | 適合 (DPA)    | 入力データの PII 削除が前提   |
| 要約結果 → CRM (Japan East)                                     | あり (戻り)| -                     | -              | -                              |
| サポートチケット (US) → サポート画面共有                        | あり       | 不定期で機微情報含む  | 注意           | サポート時の画面共有を制限     |

### 物流業 (運送業 SaaS) のケース

| 経路                                                          | 越境           | 移転データ              | 根拠     | 備考                                  |
| ------------------------------------------------------------- | -------------- | ----------------------- | -------- | ------------------------------------- |
| GPS 端末 (JP) → IoT Hub (Japan East)                         | なし           | -                       | -        | 国内完結                              |
| 国内配送データ → DWH (Japan East)                            | なし           | -                       | -        | -                                     |
| 国際輸送 配送先 → 通関業者 (現地 各国)                       | あり           | 配送先氏名・住所        | 要 同意 | 荷主から事前同意取得                  |
| GPS データ → Cloud Service の Global Telemetry (US)          | あり (メタ)    | 端末ID と稼働時間       | 適合     | 個人特定なし (端末 ID と運転手は別管理) |

## DR/バックアップでの越境注意点

```
本番: Japan East
   ↓ 自動レプリケーション
DR: East US 2  ← 越境発生!
```

このパターンは Azure SQL の geo-redundant Backup や、AWS S3 の Cross-Region Replication で多発。

**対策**: ペアリージョン (Japan East ↔ Japan West) を使う。両方とも国内のため越境なし。

## 移転先の制度確認 (主要国)

| 国/地域       | 個情法相当 | 十分性認定 | SCC が有効 | 留意点                               |
| ------------- | ---------- | ---------- | ---------- | ------------------------------------ |
| EU            | GDPR       | 相互認定   | -          | -                                    |
| 英国          | UK GDPR    | 認定済     | -          | Brexit 後は別認定                    |
| 米国          | 州法 + 連邦法 | 認定なし | 必要       | CCPA/CPRA、HIPAA 等                  |
| 中国          | PIPL       | 認定なし | 必要       | データ越境法、安全評価必須          |
| 韓国          | PIPA       | 認定なし   | 必要       | -                                    |
| シンガポール  | PDPA       | 認定なし   | 必要       | -                                    |

## 注意事項

- 「クラウドベンダーが日本にデータセンターを持っていれば OK」は不正確。グローバル管理機能経由のアクセス可能性も判定対象
- ベンダー側の障害対応・サポートで越境が発生するケースが多い (例: US の SRE がクエリ実行)
- DR/バックアップ設計の段階で越境を見落とすことが多い
- 中国向け展開は PIPL の安全評価義務があり、特に注意
