# AI生成物著作権ポリシー

## 対応する品質評価ID

- **LEGAL-AIGEN-001 (Critical)** - AI 生成物の著作権帰属・責任所在が未整備
- 関連: LEGAL-LICENSE-001、LEGAL-PII-001

## 概要

生成 AI (LLM、画像生成等) を業務利用する際の**著作権・帰属・責任所在**を、文化庁「AIと著作権に関する考え方について」(2024年3月) に準拠する形で整備する。

このポリシーの目的:

1. **学習データ段階の整理** - 著作物の AI 学習利用 (著作権法第30条の4) の要件確認
2. **生成物の著作権帰属の明確化** - 利用者・AI ベンダー・モデル提供者のいずれに帰属するか
3. **第三者著作権侵害リスクの管理** - 生成物が既存著作物に類似した場合の責任
4. **業務利用の事前合意** - 顧客への提供物に AI 生成物を含む場合の通知/同意
5. **ハルシネーション責任の所在** - 誤情報生成時の責任分界

## いつ使うか

- 矢羽①業務要件定義の中盤 (Week 4-6) - AI 利用方針の確定
- 新規 AI ベンダー導入時
- AI 生成物を顧客に提供する新サービス開始時
- 文化庁ガイドライン改訂時

## 文化庁「AIと著作権」(2024) の主要論点

### 学習データ段階 (著作権法第30条の4)

- AI 開発・学習に著作物を利用することは、原則として**著作権法上問題ない** (情報解析の目的)
- ただし、**著作権者の利益を不当に害する場合**は許諾が必要 (例: 学習用データセットの不正コピー販売)
- **追加学習 (ファインチューニング)** で特定の作家の作風を模倣する目的の場合、注意が必要

### 生成・利用段階

- AI 生成物そのものは、**人間の創作的寄与がない場合は著作物にならない**
- 生成物が既存著作物と**類似性**および**依拠性**を持つ場合、著作権侵害に該当しうる
- 生成物の利用は、**通常の著作物利用と同じルール**で判断 (引用、私的使用等の例外規定)

### 著作権侵害の判断要素

```
類似性: 既存著作物との表現上の類似度
   +
依拠性: 既存著作物に AI が依拠して生成した蓋然性
   ↓
両方が認められると著作権侵害
```

### 著作物性の判断

- AI が自動生成しただけ → 著作物性なし
- 人間がプロンプトを工夫し、生成物を選別・加工 → 創作的寄与があれば著作物
- 創作的寄与の例: 複数案からの選別、AI 生成物への手動加筆、構図・配置の意図的調整

## 手順

### ステップ 1: AI 利用パターンの分類

プロジェクトで使う生成 AI の利用パターンを以下の 4 類型に分類:

| 類型     | 例                                             | リスク度 |
| -------- | ---------------------------------------------- | -------- |
| 内部支援 | コード補完、社内ドキュメント要約               | 低       |
| 顧客対応 | チャットボット、コールセンター応対サマリ       | 中       |
| 顧客提供 | レポート自動生成、生成画像の納品               | 高       |
| 外部発信 | 記事・ブログの公開、広告クリエイティブの公開   | 最高     |

### ステップ 2: 各利用パターンの著作権整理

各パターンについて以下を整理:

1. 学習データの出所 (パブリック OK / 自社データ / 顧客データ / Web スクレイピング)
2. 生成物の利用範囲 (社内 / 顧客 / 公開)
3. 著作権の帰属 (利用者 / 自社 / ベンダー)
4. ハルシネーション責任の所在
5. 第三者著作権侵害が判明した場合の対応

### ステップ 3: AI ベンダー契約の条項確認

主要ベンダーの契約条項のチェック観点:

- **学習への利用**: 入力プロンプトがモデル再学習に使われるか
- **データ秘匿性**: 入力データの保管場所、保管期間
- **生成物の権利譲渡**: ベンダーが生成物の権利を主張しないか
- **侵害賠償保証 (Indemnification)**: ベンダーが第三者著作権侵害の賠償を保証するか
- **コンテンツフィルタ**: 不適切な生成物のフィルタ機能の有無

### ステップ 4: 利用ガイドラインの策定

社員/関係者向けの利用ガイドラインを作成:

- 使ってよい AI ツール (allowlist)
- 入力してはいけないデータ (極秘情報、個人情報、契約上の機密)
- 生成物の確認義務 (ハルシネーション、著作権チェック)
- 生成物のクレジット表記 (例: 「本記事は AI 支援で作成」)
- インシデント発生時の報告ルート

### ステップ 5: 法務承認 + ガバナンス委員会承認

- 法務部レビュー (1 週間)
- 情報セキュリティ委員会承認
- 経営層 (CISO/CPO) 最終承認

## 例示

### データ分析基盤 (DWH/Lakehouse) のケース

| 利用パターン                  | 著作権帰属             | ハルシネーション責任        | 備考                                        |
| ----------------------------- | ---------------------- | --------------------------- | ------------------------------------------- |
| SQL 自動補完 (内部支援)       | 利用者 (自社)          | 利用者 (確認義務)           | コード補完は通常リスク低                    |
| 自然言語クエリ (顧客提供)     | 利用者 (顧客)          | 自社 (誤集計リスクの注記)   | 結果数値のハルシネーションに注意            |
| 自動レポート生成 (顧客提供)   | 利用者 (顧客)          | 自社 (人間レビュー必須)     | 文章生成は事実誤認リスクあり                |
| ダッシュボード説明文 (顧客)   | 利用者 (顧客)          | 自社 (事前テンプレ承認制)   | カスタム文章は法務確認必須                  |

### コールセンター (VoC 分析) のケース

| 利用パターン                | 著作権帰属             | ハルシネーション責任         | 備考                                          |
| --------------------------- | ---------------------- | ---------------------------- | --------------------------------------------- |
| 通話要約 (内部支援)         | 利用者 (自社)          | 応対者 (確認義務)            | 要約は実質的な創作的寄与なし、著作物性なし    |
| FAQ 自動生成 (顧客対応)     | 利用者 (自社)          | 法務確認後に公開             | FAQ 公開前に必ず人手レビュー                  |
| 応対スクリプト提案 (内部)   | 利用者 (自社)          | 応対者                       | 入力プロンプトのモデル学習禁止を契約で確認    |
| 顧客向け Q&A 回答 (顧客対応)| 利用者 (顧客対応者)    | 自社 (応対品質責任)          | リアルタイム生成回答はチェックが困難          |

通話録音内容の AI 学習利用は、本人同意が原則必要。

### 物流業 (運送業 SaaS) のケース

| 利用パターン                  | 著作権帰属         | ハルシネーション責任 | 備考                                          |
| ----------------------------- | ------------------ | -------------------- | --------------------------------------------- |
| 配送ルート最適化提案 (顧客)   | 利用者 (顧客)      | 自社 (誤判断責任)    | 最適化結果は事実上ロジック計算、著作物性なし  |
| 運転手向けアドバイス (内部)   | 利用者 (自社)      | 安全運行管理者       | 安全関連の誤情報は重大事故につながる懸念       |
| 配送遅延の説明文 (顧客対応)   | 利用者 (顧客)      | 自社 (事前テンプレ)  | 自由文生成は法務確認後のみ                    |

## 入力ホワイトリスト/ブラックリスト

### 入力 OK

- 公開済み Web 情報 (ライセンスを確認済)
- 自社が著作権を持つ社内文書
- 集計済み・匿名化済みデータ
- 疑似データ・テストデータ

### 入力 NG (絶対禁止)

- 極秘クラスのデータ (M&A 情報等)
- 機密クラスのデータ (顧客リスト等)
- 顧客の個人情報 (氏名・住所・通話内容等)
- API キー、暗号鍵、認証情報
- 他社の機密文書 (NDA で受領したもの)

### 入力 条件付き

- 機密レベルの自社情報 → ベンダーとの契約で学習禁止が確認されている場合のみ
- 公開予定の前情報 → ベンダーがオンプレ/プライベートデプロイの場合のみ

### Layer 5 リスクレジスタとの紐付け

AI 利用に関するリスクは Layer 5 リスクレジスタに登録:

```yaml
- id: R-LEG-31
  title: "AI 生成物の第三者著作権侵害"
  category: legal
  probability: low
  impact: severe
  description: |
    顧客提供レポートの自動生成で、既存著作物と類似した文章が含まれる可能性。
    依拠性が認められると著作権侵害となり、損害賠償+信用失墜のリスク。
  triggers:
    - 著作権者からの侵害通知
    - 類似性検知ツールでの一致警告
  mitigations:
    - 生成物の人間レビュー必須化
    - 類似性検知ツールの導入
    - ベンダーの侵害賠償保証 (Indemnification) 確保
  owner: "法務部長"
  kill_criterion: |
    著作権侵害が認定された場合、当該機能の即時停止 +
    謝罪 + 影響顧客への通知 + 再発防止策提出
```

### 品質ゲートID対応

- **LEGAL-AIGEN-001**: 利用パターン×著作権帰属×責任所在のマトリクスがあれば Critical 解消
- **LEGAL-LICENSE-001**: 学習データのライセンスは OSS ライセンス管理.md と相互参照
- **LEGAL-PII-001**: AI への入力データは個人情報取扱台帳と整合させる

## 注意事項

- 文化庁ガイドラインは**法令ではなく考え方の整理**。最終的な判断は裁判所
- 2024-2026 年は AI と著作権の議論が急速に変化している。年次でレビュー必須
- 米国・EU・中国でルールが異なるため、海外展開時は別途検討
- 「人間の創作的寄与」の判定は曖昧で、安易に著作物性を主張しない方が安全な場合もある
