# データ分類台帳

## 対応する品質評価ID

- **LEGAL-DATACLASS-001 (Critical)** - データ分類が未整備でアクセス制御の根拠が曖昧
- 関連: LEGAL-PII-001、COMPLY-PIPA-JP-001、DATA-DELETE-001

## 概要

プロジェクトが扱う**全てのデータ資産**を 4 つの機密度クラス (公開/内部/機密/極秘) に分類し、各クラスに応じたアクセス制御・暗号化・削除戦略を定義する。

このスキルの目的:

1. **機密度の可視化** - 全データ資産を一覧で把握
2. **アクセス制御の根拠化** - なぜ特定の人だけアクセスできるかを明示
3. **データオーナーの明確化** - データに関する判断責任者を指名
4. **暗号化・マスキング戦略の整合化** - 機密度に応じた一貫した保護
5. **インシデント対応の優先順位化** - 「極秘」漏えいは最優先対応

## いつ使うか

- 矢羽①業務要件定義の初期 (Week 1-3)
- 新規データソース追加時 (随時更新)
- データ統合 (ETL) 設計時
- セキュリティ監査時の根拠資料として

## 4 分類の定義

| クラス     | 機密度 | 例                                                 | アクセス制御原則         |
| ---------- | ------ | -------------------------------------------------- | ------------------------ |
| **公開**   | 最低   | 公開済みプレスリリース、自社 Web の公開情報       | 誰でも閲覧可能           |
| **内部**   | 低     | 社内マニュアル、組織図、社内 Slack の通常会話     | 全社員 + 一部委託先      |
| **機密**   | 中〜高 | 顧客リスト、購買履歴、給与情報、取引先契約書       | 該当部門のみ + NDA       |
| **極秘**   | 最高   | M&A 情報、個人情報の大量データ、API キー、暗号鍵   | 役員 + 必要最小限の特定者 |

### 各クラスの追加定義

| クラス | 暗号化         | マスキング       | 越境           | 保管期間            | 削除方式             |
| ------ | -------------- | ---------------- | -------------- | ------------------- | -------------------- |
| 公開   | 不要           | 不要             | 制限なし       | 必要に応じて        | 通常削除             |
| 内部   | 通信時         | 不要             | 国内推奨       | 業務必要期間        | 通常削除             |
| 機密   | 保管時+通信時  | 出力時に部分マスク | 越境制限あり   | 法令+契約に従う     | 物理削除 (期限超過)  |
| 極秘   | 保管時+通信時+列暗号化 | ログ出力時に完全マスク | 越境原則禁止   | 必要最小限          | 物理削除 + 削除証明書 |

## 手順

### ステップ 1: データ資産の全洗い出し

以下のソースを参照し、全データ資産をリストアップ:

- データベース内の全テーブル
- DWH/Data Lake の全 schema/dataset
- BI ダッシュボードの全データソース
- ファイル共有 (S3 バケット、Azure Blob、SharePoint)
- 各 SaaS アカウント内のデータ
- メール/Slack チャネルの保存メッセージ
- 機密書類 (契約書、人事資料)

### ステップ 2: 4 分類への割当

各データ資産を 4 分類のいずれかに割当てる。判断基準:

```
Q1: 第三者 (社外) に漏えいした場合の影響度は?
   → 致命的     → 極秘
   → 重大       → 機密
   → 軽微       → 内部
   → 影響なし   → 公開

Q2: 法令で機密扱いを義務付けられているか?
   → Yes → 機密以上 (個情法、不競法、機密保持契約)

Q3: 顧客との契約で機密扱いを義務付けられているか?
   → Yes → 機密以上
```

迷った場合は**1 段階上のクラス**に分類するのが原則。

### ステップ 3: データオーナー指名

各データ資産に**データオーナー** (1 名) を必ず指名する。データオーナーの責務:

- データの利用目的の妥当性判断
- アクセス権限付与の最終承認
- データ品質の保証
- 削除/保持期間の判断
- インシデント時の一次対応者

データオーナーは原則として**業務部門の管理職以上**。情シス部門が肩代わりするのは NG (= 無責任体制になる)。

### ステップ 4: アクセス制御の設計

4 分類とロール (Role) のマトリクスを作成:

| ロール         | 公開 | 内部 | 機密 | 極秘 |
| -------------- | ---- | ---- | ---- | ---- |
| 全社員         | R    | R    | -    | -    |
| 該当部門社員   | R    | R    | R    | -    |
| 該当部門管理職 | R    | R    | RW   | R    |
| データオーナー | R    | R    | RW   | RW   |
| 経営層         | R    | R    | R    | R    |
| 委託先         | R    | -    | -    | -    |
| 一般 (社外)    | R    | -    | -    | -    |

R = Read、RW = Read+Write、- = アクセス不可

### ステップ 5: ガバナンス委員会への上申

「極秘」クラスのデータについては、四半期ごとにガバナンス委員会で棚卸しを実施し、新規追加・削除を報告。

## 月次/四半期レビューでの確認項目

### 月次

- 新規データ資産の追加状況
- アクセス権限の棚卸し (在職者リストとの突合)
- 異常アクセスのアラート確認

### 四半期

- 「極秘」クラスの全資産棚卸し
- データオーナーの異動有無
- 分類の見直し (機密→内部 など、機密度低下)

### 年次

- 全データ資産の網羅性確認 (新規システム取込み漏れ)
- 分類スキーム自体の妥当性レビュー
- 暗号鍵ローテーション

## pm-blueprint 連携

### Layer 5 リスクレジスタとの紐付け

「機密」「極秘」クラスのデータには、必ず Layer 5 リスクレジスタにセキュリティリスクを登録:

```yaml
- id: R-SEC-03
  title: "極秘クラスデータの不正アクセス"
  category: technical
  probability: low
  impact: critical
  description: |
    payment_methods テーブルへの不正アクセスが発生した場合、
    PCI-DSS 違反 + ブランドへの賠償 + 顧客告知義務が発生する。
  triggers:
    - 決済担当外の SQL クエリ実行ログ
    - HSM 鍵の異常アクセスログ
  mitigations:
    - 列レベル暗号化
    - HSM での鍵管理
    - 二要素認証必須化
    - 月次アクセス棚卸し
  owner: "CISO"
  kill_criterion: |
    payment_methods への不正アクセスが 1 件でも検知されたら、
    即時アクセス停止 + フォレンジック調査開始
  linked_data_assets: [DA-007]
```

### 品質ゲートID対応

- **LEGAL-DATACLASS-001**: 全データ資産に分類+オーナー+アクセス制御がある状態で Critical 解消
- **LEGAL-PII-001**: PII を含む資産は機密以上に分類することを台帳で確認
- **DATA-DELETE-001**: 各資産の削除戦略が定義済みで、データ保持_削除戦略.md と整合

### Layer 6 WBS への展開

- 1.4.1 全データ資産棚卸しワークショップ (情シス×業務部門、半日 × 2 回)
- 1.4.2 4 分類割当ドラフト (PM、5 営業日)
- 1.4.3 データオーナー指名 (各部門長と調整、2 週間)
- 1.4.4 アクセス制御マトリクスの定義
- 1.4.5 「極秘」クラスのガバナンス委員会承認
- 1.4.6 IAM システムへの反映タスク (実装フェーズ)

## 注意事項

- データ分類は**業務部門との合意**が最重要。情シス側が一方的に決めると現場で守られない
- 「極秘」を増やしすぎると現場のオペレーションが回らないため、本当に必要な資産に限定する
- 分類の変更 (機密→公開 など) は**ガバナンス委員会承認**を必須とし、安易な格下げを防ぐ
- データ加工後 (集計後) はクラスが下がるケース (機密→内部) があるため、変換ロジックも記録
