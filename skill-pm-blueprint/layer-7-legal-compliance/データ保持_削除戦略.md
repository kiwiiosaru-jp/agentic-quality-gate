# データ保持・削除戦略

## 対応する品質評価ID

- **DATA-DELETE-001 (Major)** - データ削除戦略 (物理 vs 論理) と本人請求対応の SOP が未整備
- **COMPLY-RETAIN-001 (Major)** - 法令保管期間とビジネス保管期間の整合性確認が未整備
- 関連: LEGAL-PII-001、COMPLY-PIPA-JP-001、LEGAL-DATACLASS-001

## 概要

プロジェクトで取扱う**全データ資産**について、保管期間・削除方法・本人請求対応を体系化する。

このスキルの目的:

1. **保管期間の根拠化** - 法令、契約、ビジネス必要性を明示
2. **削除戦略の選択** - 物理削除 vs 論理削除の使い分け
3. **自動削除ジョブの設計** - 期限超過データの自動削除
4. **本人請求対応 SOP** - 個情法第33-35条への対応手順
5. **削除証明** - 削除実施記録の保管

## いつ使うか

- 矢羽②PoC+システム要件 (Week 7-12)
- 新規データ資産追加時
- 法定保管期間の改正時
- 本人請求 (削除依頼) 受付時の対応開始

## 物理削除 vs 論理削除

### 物理削除 (Physical Delete)

データを実際にストレージから消去する。

| メリット                                | デメリット                              |
| --------------------------------------- | --------------------------------------- |
| 個情法の削除請求への完全対応            | 一度消すと復元不可 (誤削除リスク)       |
| ストレージコスト削減                    | バックアップからの復元手順が必要        |
| GDPR の右への忘れられる権利に対応       | 監査・調査時にデータが残らない          |

### 論理削除 (Soft Delete)

`is_deleted` フラグや `deleted_at` カラムで論理的に削除を表現する。

| メリット                                  | デメリット                              |
| ----------------------------------------- | --------------------------------------- |
| 誤削除からの復元が容易                    | 個情法削除請求では「削除」と認められない |
| 削除前後のクエリで整合性を保ちやすい      | ストレージコストは下がらない            |
| 監査ログ・調査用にデータが残る            | 個人特定可能データのまま保管する形      |

### 使い分けの原則

```
個人情報・要配慮個人情報    → 物理削除 (期限超過)
業務上の参照履歴データ       → 論理削除 (一定期間後に物理削除)
ログデータ                   → 物理削除 (自動 TTL)
集計済み・匿名化済みデータ   → 削除不要 (個人特定不可なら)
```

組合せ例:

```
[活動データ]
   ↓ ユーザー退会・期限超過
[論理削除] (30 日間、復元可)
   ↓
[物理削除] (期限超過後)
   ↓
[削除証明書発行] (DPO 用)
```

## 手順

### ステップ 1: データ資産ごとの保管期間決定

データ分類台帳の各資産について以下を決定:

| 項目             | 内容                                             |
| ---------------- | ------------------------------------------------ |
| 法定保管期間     | 該当する法令とその期間                           |
| 契約上の保管期間 | 顧客との契約による必須保管期間                   |
| ビジネス保管期間 | 業務必要性 (例: 不正検知のため過去 1 年)         |
| 統合保管期間     | 上記の最大値 (= 必要最小限の保管期間)            |
| 削除トリガー     | 顧客退会時、期限超過時、本人請求時等             |

### ステップ 2: 削除方式の決定

各データ資産に物理/論理いずれかの削除方式を割当:

| データタイプ                  | 推奨方式                            |
| ----------------------------- | ----------------------------------- |
| PII (個人情報)                | 物理削除 (退会+5年後)               |
| 通話録音                      | 物理削除 (3年後)                    |
| 注文履歴                      | 論理削除 (退会+5年) → 物理削除      |
| アクセスログ                  | TTL 自動物理削除                    |
| 監査ログ                      | TTL 自動物理削除 (長期保管後)       |
| 集計済みデータ                | 削除不要 (匿名化済み)               |

### ステップ 3: 自動削除ジョブの設計

期限超過データを自動削除する仕組みを構築:

#### 設計パターン

```
[毎日 03:00]
  ↓
[Job: identify expired records]
  ├ orders WHERE deleted_at < NOW() - INTERVAL '5 years'
  ├ recordings WHERE created_at < NOW() - INTERVAL '3 years'
  └ access_logs WHERE created_at < NOW() - INTERVAL '90 days'
  ↓
[Dry run] (削除対象件数を経営層 dashboards に報告)
  ↓
[人間承認 (週次)]
  ↓
[Actual physical delete + 削除証明書発行]
```

#### 安全装置

- **二重確認**: 削除前に対象件数を表示し、想定範囲内か確認
- **ステージング**: 物理削除前に隔離テーブルへ移動
- **削除上限**: 1日の削除件数に上限 (例: 10,000 件) → 上限超過は手動承認
- **誤削除復旧**: 隔離期間 (30 日) 中は復旧可能

### ステップ 4: 本人請求対応 SOP

個情法第33-35条への対応手順を確立:

#### 開示請求 (第33条)

```
1. 本人確認 (運転免許証等のコピー)
2. 該当データの収集 (個人情報取扱台帳に基づく)
3. 開示用フォーマットへの整形
4. 開示の実施 (書面 / 電子データ / 視聴)
5. 記録の保管 (請求受付日、対応者、開示内容)
```

開示期限: 法令で2週間以内、自社目標で1週間以内。

#### 訂正・追加・削除請求 (第34条)

```
1. 本人確認
2. 訂正・追加・削除の必要性判定 (事実誤認の有無)
3. 訂正等の実施
4. 訂正後の再開示 (任意)
5. 拒否する場合の理由通知
```

#### 利用停止・消去請求 (第35条)

```
1. 本人確認
2. 請求理由の確認 (利用目的範囲外、不正取得、漏えい懸念等)
3. 一時停止 (調査期間中)
4. 必要性判定 (法令上保管必要かの確認)
5. 利用停止または消去の実施
6. 結果通知
```

### ステップ 5: 削除証明書の発行

特に「機密」「極秘」クラスのデータは、削除実施を**書面化**して残す:

```
削除証明書

データ資産: dwh.bronze.orders (顧客 ID: 12345)
削除依頼: 顧客本人からの第35条請求 (2026-04-15 受付)
削除日時: 2026-04-22 03:00:00 JST
削除担当: (担当者 ID)
削除方式: 物理削除 (テーブル + バックアップ + ログ含む)
削除対象件数: 348 行
削除確認: SELECT COUNT(*) = 0 確認済み
DPO 承認: (氏名) 2026-04-22
```

### ステップ 6: 保持・削除戦略書の最終承認

- 法務部レビュー (1 週間)
- DPO 最終承認
- 経営層 (CISO/CPO) 承認

## 監査・モニタリング

### 月次レビュー

- 自動削除 Job の実行成功率
- 削除件数の傾向
- 本人請求対応件数とリードタイム
- 例外的に削除しなかったデータ件数 (法令上保管必要)

### 年次レビュー

- 全データ資産の保管期間棚卸し
- 法定保管期間の改正反映
- 自動 Job の二重確認ルールの妥当性
- 削除証明書の保管状況

## バックアップとの整合性

削除対象データは**バックアップ**にも残ることが多い:

| バックアップ種別      | 削除対応                                            |
| --------------------- | --------------------------------------------------- |
| 日次バックアップ      | バックアップ自体の retention (例: 30 日) で自然消滅 |
| 週次・月次バックアップ| 個別データを削除する仕組みは困難 → 全体retention  |
| 個人情報を含む長期保管 | 暗号化 + アクセス制限で運用、retention で自然消滅 |
| 法的保全データ        | 削除不可 (訴訟保留中の法令対応)                    |

GDPR 等の右への忘れられる権利では「合理的な努力」で OK と解釈されることが多いが、本人請求があった場合の対応指針は事前合意必須。

- **DATA-DELETE-001**: 物理/論理削除戦略 + 自動 Job + 本人請求 SOP が定義されていれば Major 解消

## 注意事項

- 「削除した」の証明は意外と難しい。削除前の COUNT、削除後の COUNT、ログを 3 点セットで残す
- バックアップからの完全削除は技術的に困難なケースあり (テープバックアップ等) → 暗号化+鍵削除で対応する案
- 「論理削除しただけ」を「削除しました」と本人に回答するのは個情法上 NG (削除請求への適切な対応ではない)
- 法的保全 (Litigation Hold) 中のデータは保管期間を超過しても削除不可
- 子供の個人情報 (16 歳未満等) は別途厳格な扱いが必要なケースあり
