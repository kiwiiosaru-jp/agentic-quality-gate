# 個人情報取扱台帳

## 対応する品質評価ID

- **LEGAL-PII-001 (Critical)** - 個人情報の項目別取扱台帳が未整備
- 関連: COMPLY-PIPA-JP-001、LEGAL-CROSSBORDER-001

## 概要

プロジェクトで取扱う**全ての個人情報**を、項目・取得元・目的・保管期間・第三者提供・越境有無の 6 軸で台帳化する。これは個人情報保護法 (個情法) における**保有個人データの開示・訂正・削除請求**への対応基盤になる。

台帳の目的:

1. **網羅性確保** - 取扱う PII の見落としを防ぐ
2. **合法性立証** - 利用目的・取得経路の妥当性を立証可能にする
3. **本人請求対応** - 開示・訂正・削除請求があった際の対応工数を削減
4. **DPIA (Data Protection Impact Assessment) の基盤** - 影響評価の前提情報

## いつ使うか

- 矢羽①業務要件定義の初期 (Week 1-3)
- 新規 PII 項目を取扱うことが判明した時 (随時更新)
- 個情委への報告/相談が必要になった時
- M&A/事業譲渡で他社からデータ受領する時

## 手順

### ステップ 1: PII の網羅的洗い出し

業務フロー図/データフロー図を見ながら、以下を全て洗い出す:

- **顕在 PII**: 氏名、住所、電話、メール、生年月日
- **潜在 PII**: 端末ID、Cookie ID、IP アドレス、位置情報、購買履歴 (個人特定可能なら個人情報)
- **要配慮個人情報**: 病歴、犯罪歴、人種、信条 (個情法第2条第3項)
- **個人識別符号**: マイナンバー、運転免許証番号、パスポート番号

### ステップ 2: 6 軸への分解

各 PII 項目について、下記 6 軸を埋める:

| 軸             | 例                                                             |
| -------------- | -------------------------------------------------------------- |
| 項目           | 顧客氏名、メールアドレス、購買履歴、通話録音、運転手位置情報   |
| 取得元         | 自社会員登録、他社からの提供、Webスクレイピング、IoT センサー |
| 目的           | 商品配送、メルマガ配信、AI モデル学習、不正利用検知           |
| 保管期間       | 5年、契約終了+3年、削除依頼まで、無期限 (要見直し)            |
| 第三者提供     | あり/なし、提供先、提供条件、本人同意の有無                    |
| 越境有無       | 国内のみ / EU内 / 米国 / 中国 / 複数国 (国名を全て列挙)        |

### ステップ 3: リスクスコアリング

各項目に簡易的なリスクスコアを付ける:

```
リスク度 = 機微性 × 規模 × 越境度

機微性:  1=非機微 / 2=通常PII / 3=要配慮
規模:    1=数百件 / 2=数万件 / 3=数十万件以上
越境度:  1=国内のみ / 2=十分性認定国 / 3=非認定国
```

スコア 9 以上は重点監視対象。

### ステップ 4: 法務 + DPO 承認フロー

以下の承認を経て確定:

```
プロジェクト責任者 起票
   ↓
データオーナー 確認
   ↓
情報セキュリティ部門 レビュー
   ↓
法務部門 (個情法整合性確認)
   ↓
DPO (Data Protection Officer) 最終承認
   ↓
台帳確定 → 個情委への保有個人データ公表対応
```

承認は**書面 (または電子サイン)** で記録する。口頭承認は不可。

## 月次レビューでの更新項目

- 新規追加された PII 項目はないか
- 利用目的の変更が必要な項目はないか
- 保管期間を超過しているデータはないか
- 第三者提供の追加変更はないか
- 越境ポイントの変更はないか
- 本人請求 (開示/訂正/削除) の対応件数

## pm-blueprint 連携

### Layer 5 リスクレジスタとの紐付け

各 PII 項目のリスクスコアが 6 以上の場合、Layer 5 のリスクレジスタに以下のように登録:

```yaml
- id: R-LEG-01
  title: "顧客 PII の漏洩"
  category: legal
  probability: low
  impact: critical
  description: |
    顧客氏名、購買履歴を含む大量データが Azure SQL に格納されている。
    DBA 権限の悪用、アプリ側の SQL インジェクション等で漏洩可能性。
  triggers:
    - 認証ログの異常検知アラート
    - 大量クエリ実行検知
  mitigations:
    - TDE 暗号化 + 列レベル暗号化 (購買履歴のみ)
    - DBA 操作の二重承認
    - SQL インジェクション静的解析の自動化
  owner: "DPO"
  kill_criterion: |
    PII 漏洩事故が発生した場合、即時サービス停止 +
    個情委への 3-5 営業日以内の報告 + 全顧客への通知
  linked_pii_inventory: [PI-01, PI-02]
```

### 品質ゲートID対応

- **LEGAL-PII-001**: 本台帳の存在自体で Critical Fail を解消
- **COMPLY-PIPA-JP-001**: 「利用目的」「第三者提供」の項目が個情法整合性チェックの基盤
- **LEGAL-CROSSBORDER-001**: 「越境有無」の項目が越境データ移転書の基盤
