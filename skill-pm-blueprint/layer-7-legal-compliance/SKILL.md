## このレイヤーの目的

プロジェクト青写真を描く段階で、以下を一気通貫で押さえ込み、後工程で「気付いたら違法」「気付いたら賠償リスク」という致命的事故を防ぐ:

1. **個人情報の網羅的洗い出し** - 個情法第18-31条への適合性を可視化
2. **業法該当性の判定** - 運送業/電気通信事業法/下請法/景表法/消費者契約法
3. **データ分類とアクセス制御** - 公開/内部/機密/極秘の4分類
4. **AI生成物の権利関係整理** - 文化庁2024年見解への準拠
5. **越境データ移転の事前合意** - Azure/AWS/GCP のリージョン別判定
6. **OSSライセンス汚染の予防** - GPL/AGPL等のコピーレフト系の混入防止
7. **データ保持・削除戦略** - 物理削除と論理削除の明確化、本人請求対応

## 推奨実行順序

```
1. データ分類台帳.md            ← まず対象データを分類
2. 個人情報取扱台帳.md          ← PII を抽出して台帳化
3. 個情法対応マトリクス.md      ← 個情法第18-31条のチェック
4. 業法該当性判定書.md          ← 業種ごとの法令適合性
5. 越境データ移転書.md          ← クラウドリージョン判定
6. OSSライセンス管理.md         ← SBOM とライセンス整合性
7. AI生成物著作権ポリシー.md    ← 生成AI を使う場合に追加
8. データ保持_削除戦略.md       ← 最後に保管期間と削除戦略
```

各スキルの成果物は **法務承認 + DPO (Data Protection Officer) 承認** を経て確定させる。

## 入力 (前レイヤーから受け取るもの)

- Layer 2 の前提リスト (特にステークホルダーマップ、業務領域)
- Layer 3 の ADR (使用クラウド、SaaSベンダー、AI/LLM ベンダー)
- Layer 4 の機能要件 (個人情報を扱う処理の特定)
- Layer 5 のリスクレジスタ (法的リスクのカテゴリ抽出)

## 出力 (次レイヤーへ渡すもの)

- **個人情報取扱台帳** (PII Inventory) - 項目/取得元/目的/保管期間/第三者提供/越境有無
- **個情法対応マトリクス** - 第18-31条の適合状況一覧
- **業法該当性判定書** - 該当する業法と必要な届出/許認可
- **データ分類台帳** - 全データ資産の機密度分類
- **AI生成物著作権ポリシー** - 帰属/責任所在/学習データの扱い
- **越境データ移転書** - データフロー図 + 越境ポイントの法的根拠
- **OSSライセンス SBOM** - 全 OSS 部品の license 一覧と商用可否
- **保持_削除戦略書** - 保管期間表 + 自動削除ジョブ仕様

これらは Layer 5 (リスクレジスタへ法的リスク登録)、Layer 6 (WBSに法務タスク追加) の入力となる。

## 年間1億円規模での軽量化方針

- **法務専門家は外部弁護士の月次4時間を確保** (年間予算 100万円程度を確保)
- **DPO は内部兼任**で OK だが、ガバナンス委員会への定期報告は必須
- **個人情報取扱台帳** は 50項目以内、**業法判定書** は 1業種に絞り、深掘りする
- **AI生成物著作権ポリシー** は雛形 + プロジェクト固有差分のみ作成
- **越境データ移転書** はクラウド 1社・リージョン 1〜2 つに絞って作成
- **OSSライセンス管理** は CycloneDX/SPDX の自動生成ツール (`syft`, `trivy`) を採用してマニュアル工数を削減
- データ分類は **4分類 (公開/内部/機密/極秘)** に固定し、業務側との合意を優先

## ドメイン別の重点ポイント

### データ分析基盤 (DWH/Lakehouse)

- **PII の集中**: Bronze 層に PII が集約されるため、マスキング設計が重要
- **越境**: Snowflake/Databricks のリージョン選定が越境判定の中核
- **OSS**: dbt-core (Apache 2.0) は OK だが、関連 adapter の license を要確認
- **保持**: BI 用キューブと生データで保持期間が異なるケースが多い

### コールセンター (VoC 分析)

- **PII の特殊性**: 通話録音は要配慮個人情報 (個情法第2条第3項) になる場合あり
- **業法**: 電気通信事業法 (録音・通信内容の取扱)、消費者契約法、景表法
- **AI生成物**: 通話要約のハルシネーション責任所在の事前合意が必須
- **保持**: 通話録音の法定保存期間 (3〜7年、業界による) と GDPR 削除請求の衝突

### 物流業 (運送業 SaaS)

- **業法**: 貨物自動車運送事業法、下請法、独占禁止法 (荷主との力関係)
- **PII**: 運転手の運行データは個人情報になりうる (位置情報×時刻×個人ID)
- **越境**: 国際輸送では複数国の個情法に同時準拠する必要
- **保持**: 配送記録の法定保存期間 (運送約款第14条等)

## pm-blueprint 連携

### Layer 5 リスクレジスタとの紐付け

本レイヤーで識別した法的リスクは **Layer 5 リスクレジスタ** にカテゴリ `legal` として登録する。Kill基準の例:

- 「法務が PII 取扱について NG 判定を出したら即時停止」 (パターン C)
- 「個情委から行政指導が入ったら全機能停止 + 是正計画書提出」 (パターン A)
- 「OSS ライセンス汚染が発見されたら 7 営業日以内に切替検討」 (パターン B)

### 品質ゲートID対応

本レイヤーの全成果物は、品質評価レポート (`2026-04-30_blueprint_review.md`) の判定項目と 1:1 でマッピングする。レビュー時の確認手順:

1. 品質ゲートID (例: LEGAL-PII-001) を引く
2. 対応する成果物 (例: 個人情報取扱台帳) を開く
3. レビュー観点 (例: 第三者提供の有無が明記されているか) でチェック
4. 不適合があれば Layer 5 のリスクレジスタに R番号を発行して追跡

### Layer 6 WBS への展開

各スキルの末尾に「WBS追加タスク例」を記載している。Layer 6 ではこれを矢羽①業務要件定義のサブタスクとして組込む。

## 注意事項

- **本スキルは法務助言ではない**。最終判断は必ず弁護士・社内法務に確認すること
- 個情法・業法は改正頻度が高いため、参照時点のバージョンを台帳に明記する
- AI生成物著作権の議論は 2024-2026 で急速に変化中。文化庁見解の最新版を参照
- データ越境の判定は単純な「サーバー所在地」ではなく「アクセス可能性」で判定される
