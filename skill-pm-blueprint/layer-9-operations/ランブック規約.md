# ランブック規約

## 品質ゲート対応
- **DOC-RUNBOOK-001** (high): 運用対象別ランブック群、演習記録、3ヶ月以内最終更新

## 概要

ランブック (Runbook) は、運用作業の手順書。運用対象別に整備し、定期的に演習・更新を行う。

## いつ使うか

- 矢羽⑥ 非機能・運用 6.6 (障害対応フロー + ランブック) で初版作成
- 矢羽⑦ パイロットで演習実施
- 矢羽⑧ 全拠点展開期間中、随時更新
- 本番運用以降、3ヶ月毎レビュー

## 品質チェックリスト

ランブックレビュー時の確認事項:

- [ ] 最終更新が 3ヶ月以内
- [ ] トリガーが具体的 (「障害発生時」など曖昧でない)
- [ ] 前提条件 (権限・ツール・時間) が明記
- [ ] 手順が再現可能 (「適切に対応」など曖昧でない)
- [ ] 失敗時の代替手順あり
- [ ] エスカレーション経路あり
- [ ] 演習履歴あり (年 2 回以上)
- [ ] 関連リスク ID と紐付け済
- [ ] 関連 SLO/NFR と紐付け済

## 演習サイクル

| 種類 | 頻度 | 対象 |
|------|------|------|
| 個別ランブック演習 | 四半期 | 各ランブック × 担当者 |
| 統合演習 (障害シナリオ) | 半期 | 主要 5-10 ランブック |

## サンプル: Azure OpenAI クォータ枯渇時 (R-002 対応)

```markdown
# RB-LLM-001: Azure OpenAI クォータ枯渇

## トリガー
- アラート: Azure Monitor "OpenAI 429 Errors > 1%/5min"
- ユーザ報告: 分析パイプラインのエラー多発

## 前提
- 権限: Azure OpenAI Resource Contributor
- 想定所要: 30 分

## 手順
### Step 1: 現状確認 (5分)
1. Azure Portal → OpenAI → メトリクス → 429 エラー率を確認
2. クォータ使用率を確認 (目標 80% 未満)

### Step 2: 一次対応 (15分)
1. EARS-W04 に従い、キュー退避を確認
   - Azure Function 「llm-batch-queue」で滞留確認
2. 緊急ならクォータ増額申請 (Azure Support 経由)
   - 通常 1-3 営業日
3. 並行して低コストモデル (GPT-4o-mini) へ一時切替検討

### Step 3: 復旧確認 (5分)
1. 429 エラー率 < 0.5% を 5 分連続で確認
2. キュー残量が減少傾向

### Step 4: 事後処理 (5分)
1. インシデント記録 (Jira INC-XXX)
2. Slack #ops-incident に通知
3. 翌営業日: クォータ予約交渉ミーティング設定

## エスカレーション
- 6 時間解決しない: AI Lead + 経営層通知 (R-002 Kill基準警戒)
- 3 ヶ月連続発生: モデル切替判断 (R-002 Kill基準発動)

## 関連
- リスク: R-002
- SLO: NFR-005 可用性
```
