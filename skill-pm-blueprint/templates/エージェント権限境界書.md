## 1. 用語定義

| 用語 | 定義 |
|------|------|
| エージェント | LLM を中核とし、ツール呼出やデータアクセスを伴う自律的システム |
| ツール | エージェントが呼出可能な API・関数・コマンド |
| 権限 (Capability) | エージェントが実行可能な操作種別 |
| Human-in-the-Loop (HITL) | 人間の確認・承認を要する処理段階 |
| Sandbox | 隔離された実行環境 (DB/外部 API への直接影響なし) |
| Read-only | 読み取りのみ許可 (副作用なし) |

## 2. エージェント分類と権限ティア

### 2.1 ティア定義

| ティア | 名称 | 権限 | 例 |
|------|------|------|----|
| T0 | Sandbox-only | 隔離環境のみ。本番データなし | 開発・実験エージェント |
| T1 | Read-only | 本番データ参照のみ | レポート生成・分析エージェント |
| T2 | Limited Write | 限定された書込 (低リスク) | チケット下書き作成、推奨レコード生成 |
| T3 | Approved Action | 書込/外部 API 呼出可だが Human 承認必須 | 顧客向けメール送信、配車計画提案 |
| T4 | Autonomous | 自律的書込・実行可 (定型業務) | 内部 KPI 集計、ダッシュボード自動更新 |

### 2.2 ティアごとの統制

| 統制項目 | T0 | T1 | T2 | T3 | T4 |
|--------|----|----|----|----|----|
| 監査ログ | 必須 | 必須 | 必須 | 必須 | 必須 |
| RLS/RBAC | 必須 | 必須 | 必須 | 必須 | 必須 |
| プロンプトログ | 任意 | 必須 | 必須 | 必須 | 必須 |
| 出力ログ | 任意 | 必須 | 必須 | 必須 | 必須 |
| Human 承認 | 不要 | 不要 | 任意 | 必須 | 不要 |
| クォータ上限 | 緩 | 中 | 中 | 厳 | 厳 |
| インシデント時 Kill Switch | 任意 | 必須 | 必須 | 必須 | 必須 |

## 3. ハイリスク操作と承認フロー

### 3.1 ハイリスク操作リスト (例)

| 操作 | リスク | 承認要否 |
|------|------|--------|
| 顧客向けメール/SMS の送信 | 誤送信 → 顧客信頼失墜 | Human 必須 (T3) |
| 個人情報を含む大量レコードの参照 | PII 漏洩 | Human 承認 (T2 以上) |
| 業務系 DB への書込 | データ整合性破壊 | Human 必須 (T3) |
| 決済 API 呼出 | 金銭損失 | 二重 Human 承認 |
| 内部システムへのコード実行 | システム破壊 | 原則禁止 |
| 外部公開コンテンツの生成 | レピュテーション影響 | Human 必須 + 法務確認 |
| 大量レコードの一括削除 | データ消失 | 二重 Human 承認 + DBA 立会い |
| LLM への大量データ送信 | コスト/越境/学習混入 | クォータ事前設定 |

### 3.2 Human-in-the-Loop フロー

```
[エージェント提案]
    ↓
[1. 提案内容を承認 UI に提示]
    ↓
[2. 承認者がレビュー (差分・根拠・影響範囲)]
    ↓
[3. 承認/否認]
    ↓ (承認時)
[4. エージェントが実行]
    ↓
[5. 監査ログに承認者・実行結果を記録]
```

**SLA:**

| ハイリスク種別 | 承認 SLA |
|------------|--------|
| 顧客向け送信物 | 1 時間以内 |
| 内部書込 | 4 時間以内 |
| 公開コンテンツ | 1 営業日 |

### 3.3 二重承認 (Two-Person Rule)

- 決済・大量削除・公式ステートメント等は **異なる 2 名** の承認必須
- 同一部署内の上長と他部署 (CISO/法務) の組合せが望ましい

## 4. ツール許可リスト (Allowlist)

### 4.1 標準許可リスト構造

```yaml
agent_id: voc_analyzer_v1
tier: T1
allowed_tools:
  - id: query_voc_silver
    type: read
    scope: "SELECT only on schema voc_silver.*"
    rate_limit: "100 requests/min"
  - id: generate_summary
    type: llm_call
    model: "azure_openai:gpt-4o"
    scope: "summary generation only"
    rate_limit: "50 requests/min"
denied_tools:
  - id: write_to_production_db
    reason: "T1 read-only ティア"
  - id: send_email
    reason: "Human approval required (T3+)"
```

### 4.2 デフォルトポリシー: Deny by Default

許可リストに **明示されていない操作はすべて拒否**。

### 4.3 許可リスト変更プロセス

```
[追加/変更要求]
    ↓
[AI ガバナンス委員会レビュー (リスク評価)]
    ↓
[セキュリティレビュー (CISO)]
    ↓
[ステージング環境で 1 週間検証]
    ↓
[本番反映]
```

## 5. データアクセス境界

### 5.1 ティア別データアクセス

| ティア | アクセス可能データ | 機密度上限 |
|------|---------------|---------|
| T0 | テストデータのみ | (限定なし) |
| T1 | 本番データ (機密度内部以下 推奨) | confidential (RLS+マスキング前提) |
| T2 | 本番データ (限定スキーマ) | confidential |
| T3 | 本番データ (業務系含む) | restricted (Human 承認必須) |
| T4 | 集計データのみ (自律運用) | internal |

### 5.2 PII フィルタリング

エージェント入力前に **PII 自動検出 + マスキング** を必須化。

```
[ユーザ入力]
    ↓
[PII 検出器 (例: Presidio, 自社辞書)]
    ↓
[マスキング (氏名→***、電話→末尾4桁のみ)]
    ↓
[LLM へ送信]
```

### 5.3 データ分類との連携

`データ分類台帳.yml` の classification を参照し、`restricted` データは原則エージェント禁止。

## 6. Sandbox 環境仕様

### 6.1 Sandbox の必須要件

- 本番データへの直接接続 **不可**
- ネットワーク隔離 (内部システムへの outbound 制限)
- 計算リソース制限 (CPU/Memory/Time)
- ファイルシステム書込先の限定
- 実行ログの完全保管

### 6.2 推奨実装

- Azure Container Instances + 制限ネットワーク
- Databricks サンドボックスクラスタ
- 専用 サブスクリプション + Hub-and-Spoke 隔離

## 7. 監査・ログ要件

### 7.1 必須記録項目

| 項目 | 記録粒度 |
|------|--------|
| エージェント ID | 都度 |
| 起動者 (User ID) | 都度 |
| 入力 (プロンプト全文) | 都度 |
| 出力 (生成結果) | 都度 |
| 呼び出されたツール | 都度 |
| ツール実行結果 | 都度 |
| Human 承認者 | 該当時 |
| 実行時間・トークン消費 | 都度 |

### 7.2 保管期間

- 監査ログ: **7 年** (規制対応)
- ハイリスク操作: **永続** (Immutable Storage)
- 通常操作: 3 年

### 7.3 監査クエリ例

- 「特定ユーザの過去 30 日のエージェント呼出全件」
- 「Human 承認なき書込操作の検出」
- 「許可リスト外のツール呼出試行」

## 8. Kill Switch (緊急停止)

### 8.1 発動条件

| 条件 | 自動/手動 |
|------|--------|
| プロンプトインジェクション検出 (信頼度 > 0.9) | 自動 |
| 異常な出力パターン (規定外 API 呼出) | 自動 |
| 監査ログから不正利用判明 | 手動 |
| 法的・倫理的問題発生 | 手動 |

### 8.2 停止プロセス

```
[Kill Switch 発動]
    ↓
[全エージェント実行を即時停止]
    ↓
[新規呼出を遮断]
    ↓
[CISO + AI リード + 法務に通知]
    ↓
[原因調査 → 再開判断]
```

### 8.3 段階的再開

- 完全停止 → サンドボックスで検証 → 限定ユーザで段階再開 → 全面再開
- 各段階で AI ガバナンス委員会の承認

## 10. レッドチーム評価

### 10.1 評価頻度

- 新規エージェント本番投入前 (必須)
- 既存エージェント: 四半期 1 回
- モデル変更時 (必須)

### 10.2 評価項目

詳細は `プロンプトインジェクション評価セット.yml` 参照。

- プロンプトインジェクション耐性
- 権限境界の越境試行
- PII 漏洩試行
- ハルシネーション率
- 不適切な出力 (差別・暴力等)

## 11. 違反時の対応

| 違反度 | 例 | 対応 |
|------|----|------|
| 軽微 | 許可リスト外ツール試行 (拒否成功) | ログ記録のみ |
| 中 | プロンプトインジェクション検知 | エージェント停止 + 調査 |
| 重大 | データ漏洩・不正書込 | 全エージェント停止 + インシデント対応 |
| 致命 | 法的問題発生 | 全停止 + 法務 + 経営判断 |

## 14. pm-blueprint 連携

- **関連品質ゲート ID:**
  - LLM-AGENCY-001 (Critical) ... 本書の存在
