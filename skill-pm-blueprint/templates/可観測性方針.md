## 1. 可観測性の三本柱

### 1.1 ログ (Logs)

- **目的:** イベント発生の事実記録、原因究明
- **粒度:** リクエスト/トランザクション単位
- **代表ツール:** Azure Monitor Logs (Log Analytics), Datadog Logs

### 1.2 メトリクス (Metrics)

- **目的:** 集計値による傾向把握、アラート発火
- **粒度:** 時系列の集計値 (count/avg/p95 等)
- **代表ツール:** Azure Monitor Metrics, Prometheus, Grafana

### 1.3 トレース (Traces)

- **目的:** 分散システム横断のリクエスト追跡
- **粒度:** リクエスト全工程の Span 構造
- **代表ツール:** Application Insights, Jaeger, OpenTelemetry

### 1.4 連携モデル

```
[ユーザリクエスト]
    ↓
[traceparent 生成 (W3C Trace Context)]
    ↓
[各サービスでログに trace_id 添付]
    ↓
[メトリクスは集計値、ログは詳細、トレースで横断検索]
```

## 2. 分散トレース ID 規約

### 2.1 W3C Trace Context 準拠

`traceparent` ヘッダ形式: `00-<trace_id>-<parent_id>-<trace_flags>`

例: `00-0af7651916cd43dd8448eb211c80319c-b7ad6b7169203331-01`

| フィールド | 桁数 | 説明 |
|---------|------|------|
| version | 2 | "00" 固定 |
| trace_id | 32 | 全工程で同一 |
| parent_id | 16 | 直近 Span ID |
| trace_flags | 2 | サンプリングフラグ等 |

### 2.2 全サービスでの伝搬必須

- すべての HTTP リクエストで `traceparent` を付与・伝搬
- 非 HTTP (Service Bus 等) でもメッセージプロパティに搬送
- LLM API 呼出時もログに `trace_id` 残す
- ETL/バッチでは job_run_id を別途生成 + trace_id を上位コンテキストに

### 2.3 業務 ID との関連付け

| 種別 | 例 | 用途 |
|------|----|------|
| trace_id | W3C 32桁 | 技術トラブルシュート |
| user_id (sub) | UUID | ユーザー横断検索 |
| tenant_id | UUID | マルチテナント切り分け |
| business_event_id | UUID | 業務工程追跡 (例: shipment_id, contact_id) |

## 3. ログ仕様

### 3.1 構造化ログ (JSON 必須)

すべてのアプリケーションログは構造化 JSON 形式で出力する。

**最低限のフィールド:**

```json
{
  "timestamp": "2026-04-30T12:34:56.789Z",
  "level": "INFO",
  "logger": "com.example.api.TicketService",
  "message": "ticket created",
  "trace_id": "0af7651916cd43dd8448eb211c80319c",
  "span_id": "b7ad6b7169203331",
  "service": "ticket-api",
  "version": "1.2.3",
  "env": "prod",
  "tenant_id": "11111111-2222-3333-4444-555555555555",
  "user_id": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
  "business_event_id": "ticket:tk_abc123",
  "duration_ms": 42,
  "result": "success"
}
```

### 3.2 ログレベル

| レベル | 用途 | 例 |
|------|------|----|
| TRACE | デバッグ詳細 (本番では出さない) | 関数呼出パラメータ |
| DEBUG | 開発時調査 (本番では出さない) | 条件分岐の経路 |
| INFO | 主要イベント | リクエスト受付、起票完了 |
| WARN | 異常だが業務継続 | リトライ発生、フォールバック |
| ERROR | 業務影響あり | 起票失敗、外部 API エラー |
| FATAL | サービス停止級 | DB 接続不可、暗号鍵欠損 |

### 3.3 PII マスキング

ログには PII を出力しない。やむを得ず必要な場合は以下のポリシー。

| 対象 | 処理 |
|------|------|
| 氏名 | マスク (例: 山田 太郎 → 山田 ***) |
| 電話番号 | 末尾 4 桁を除きマスク (例: 090-XXXX-1234) |
| メールアドレス | ローカル部の先頭 2 文字以外マスク (例: ab****@example.com) |
| 住所 | 都道府県・市区町村まで |
| クレジットカード番号 | 完全禁止 |

### 3.4 ログ保管

| 種別 | 保管期間 | 保管場所 | 根拠 |
|------|--------|--------|------|
| アプリケーションログ | 90 日 | Log Analytics + Blob (Cool) | 一般的トラシュ用 |
| 監査ログ (アクセス・操作) | 7 年 | Log Analytics + Immutable Blob | 個情法・社内規定 |
| セキュリティログ | 3 年 | Log Analytics + 別アカウント | インシデント対応 |
| 通信ログ (電気通信事業者要件) | 6ヶ月 〜 1年 | 専用領域 | 関連法令 |

## 4. メトリクス仕様

### 4.1 RED メソッド (リクエストサービス向け)

| 略号 | 意味 | 例 |
|------|------|----|
| Rate | リクエスト数/秒 | api_requests_total |
| Errors | エラー率 | api_errors_total |
| Duration | レイテンシ分布 | api_latency_seconds_bucket |

### 4.2 USE メソッド (リソース向け)

| 略号 | 意味 | 例 |
|------|------|----|
| Utilization | 使用率 | cpu_usage_percent |
| Saturation | 飽和度 | queue_depth |
| Errors | エラー数 | disk_io_errors |

### 4.3 ビジネスメトリクス

| メトリクス名 | 説明 | ドメイン例 |
|----------|------|---------|
| contacts_per_hour | 時間あたり問合せ件数 | コールセンター |
| ticket_creation_rate | チケット起票率 | コールセンター |
| shipment_delivery_success_rate | 配送成功率 | 物流 |
| etl_batch_duration_minutes | ETL 所要時間 | データ分析基盤 |
| llm_token_consumption | LLM トークン消費量 | AI |

### 4.4 メトリクスタグ規約

すべてのメトリクスに以下のタグ付け必須。

- `service` (例: ticket-api)
- `env` (例: prod / stg / dev)
- `region` (例: japaneast)
- `version` (例: 1.2.3)

カーディナリティ爆発を防ぐため、`user_id` 等の高基数値は **メトリクスタグに使わない** (ログ側で対応)。

## 5. トレース仕様

### 5.1 必須計装範囲

- HTTP 受付・送信
- DB アクセス (SQL ステートメント要約付き)
- メッセージブローカ (Send/Receive)
- 外部 API 呼出 (LLM, SaaS)
- ETL ジョブ (Span = 各タスク)

### 5.2 Span 命名規則

`<verb> <resource>` 形式。

例:
- `POST /v1/tickets`
- `SELECT customer`
- `azure-openai.chat.completions`
- `databricks.run.task.silver_normalize`

### 5.3 Span 属性

OpenTelemetry セマンティック規約に準拠。

| 属性 | 例 |
|------|----|
| http.method | "POST" |
| http.url | "/v1/tickets" |
| http.status_code | 201 |
| db.system | "mssql" |
| db.statement | "SELECT * FROM CUSTOMER WHERE id = ?" |
| messaging.system | "servicebus" |
| ai.provider | "azure_openai" |
| ai.model | "gpt-4o" |
| ai.tokens.input | 250 |
| ai.tokens.output | 50 |

### 5.4 サンプリング

| 環境 | レート | 戦略 |
|------|------|------|
| 本番 | 10% | Head-based + エラー時 100% |
| ステージング | 100% | All |
| 開発 | 100% | All |

## 7. アラート設計

### 7.1 重大度

| レベル | 通知方法 | 対応 SLA |
|------|--------|--------|
| P1 (page) | PagerDuty + 電話 | 15 分以内一次対応 |
| P2 (critical) | Slack + メール + チケット | 1 時間以内 |
| P3 (warning) | Slack | 1 営業日以内 |
| P4 (info) | ログのみ | 都度確認 |

### 7.2 アラートポリシー

- **対応可能** であること (Actionable)
- **誤報率 < 5%** を目標
- 30 日経過しても未発火のアラートはレビュー対象 (削除候補)
- 1 リソース 1 アラートを目指す (重複しない)
