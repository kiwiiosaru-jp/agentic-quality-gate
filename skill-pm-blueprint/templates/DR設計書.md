## 1. RTO / RPO 目標値

### 1.1 ティア定義

| ティア | RTO | RPO | 適用例 |
|------|-----|-----|------|
| Tier 0 (致命的) | 5 分 | 0 (RPO ≈ 0) | 通話受付・配車・決済 |
| Tier 1 (基幹) | 15 分 | 5 分 | チケット管理・配送追跡 |
| Tier 2 (重要) | 1 時間 | 15 分 | BI ダッシュボード |
| Tier 3 (一般) | 4 時間 | 1 時間 | レポーティング・分析 |
| Tier 4 (低優先) | 24 時間 | 24 時間 | 内部用ナレッジサイト |

### 1.2 主要システムのティア割り当て

| システム | ティア | RTO | RPO | 根拠 |
|---------|------|-----|-----|------|
| Genesys Cloud (通話) | Tier 0 | 5 分 | ≈ 0 | 顧客応対停止は致命的 |
| 配送追跡 API | Tier 1 | 15 分 | 5 分 | B2C/B2B 顧客影響大 |
| 顧客サービス基幹 DB | Tier 1 | 15 分 | 5 分 | UC-005 起票影響 |
| Azure Databricks Gold | Tier 2 | 1 時間 | 15 分 | BI 利用者影響 |
| 通話録音 Blob | Tier 2 | 1 時間 | 0 (immutable) | 法定保存義務 |
| 内部ナレッジ Wiki | Tier 4 | 24 時間 | 24 時間 | 業務影響限定 |

## 2. 冗長構成パターン

### 2.1 比較表

| パターン | 説明 | RTO 目安 | RPO 目安 | 月額コスト目安 (倍率) | 推奨ティア |
|--------|------|--------|--------|----------------|---------|
| **Active-Active** | 複数リージョンで稼働、同期レプリ | < 5 分 | ≈ 0 | 2.0x | Tier 0 |
| **Active-Passive (Hot)** | 待機側を Hot で常時起動 | 5-15 分 | 1-5 分 | 1.7x | Tier 1 |
| **Active-Passive (Warm)** | 待機側は最小構成、必要時スケール | 15-60 分 | 5-15 分 | 1.3x | Tier 2 |
| **Pilot Light** | 待機側は DB 同期のみ、コンピュートは停止 | 1-4 時間 | 15-60 分 | 1.1x | Tier 3 |
| **Backup & Restore** | バックアップから手動復旧 | 4-24 時間 | 1-24 時間 | 1.0x | Tier 4 |

### 2.2 採用構成

#### 2.2.1 Genesys Cloud (Tier 0) — Active-Active

- **プライマリ:** Genesys クラウド (米国西海岸 + 東京リージョン マルチリージョン)
- **設定:** 2 リージョン同時利用、自動フェイルオーバー
- **検討事項:** 越境前提のため契約上の越境合意必須

#### 2.2.2 顧客サービス基幹 DB (Tier 1) — Active-Passive (Hot)

- **プライマリ:** Azure SQL Database (Japan East)
- **セカンダリ:** Azure SQL Database (Japan West) — Auto-failover Group
- **同期方式:** 同期レプリケーション (RPO ≈ 0)
- **フェイルオーバー:** 自動 (10秒以内検知)

```
[Japan East: Active]
   ↕ 同期レプリ
[Japan West: Hot Standby]
       ↓ 自動フェイルオーバー (10秒)
[Japan West: Active]
```

#### 2.2.3 通話録音 Blob (Tier 2) — Active-Passive (Warm)

- **プライマリ:** Azure Blob Storage (Japan East / RA-GZRS)
- **セカンダリ:** Japan West (Read-Access Geo-Zone-Redundant)
- **同期方式:** 非同期レプリ (RPO 15分以下)
- **不変ストレージ (Immutable Blob):** 法定保存5年間ロック

#### 2.2.4 Databricks Gold (Tier 2) — Pilot Light

- **プライマリ:** Databricks ワークスペース (Japan East)
- **セカンダリ:** ワークスペース定義のみ (Japan West)、データは ADLS Gen2 GRS で保護
- **復旧手順:** Terraform でセカンダリにワークスペース起動 → ADLS から取込

## 4. フェイルオーバー / フェイルバック手順

### 4.1 自動フェイルオーバー

**条件:**

- ヘルスチェック 3 連続失敗
- リージョン全体障害検知 (Azure Service Health)

**フロー:**

```
[障害検知]
    ↓
[Front Door: トラフィック切替 (DNS TTL 60秒)]
    ↓
[Azure SQL: Auto-failover Group 切替]
    ↓
[App Service: Slots 切替]
    ↓
[復旧確認 (15分以内)]
```

### 4.2 手動フェイルオーバー (Pilot Light など)

ランブック `RB-DR-001 DR フェイルオーバー手順` 参照。

要点:

1. インシデント司令官 (IC) 任命
2. 経営層・顧客通知 (発生から 30 分以内に第一報)
3. Terraform で Pilot Light → Active 起動 (約 60 分)
4. データ整合性確認
5. トラフィック切替
6. ステークホルダー報告

### 4.3 フェイルバック (復旧後の本番戻し)

- 業務影響の少ない時間帯に実施 (週末夜間)
- データ差分の同期 → 切戻し
- 詳細は `RB-DR-002` 参照

## 5. 演習計画

### 5.1 演習種別と頻度

| 演習種別 | 頻度 | 参加者 | 業務影響 |
|--------|------|-------|--------|
| デスクトップ演習 (Tabletop) | 半期 1 回 | 経営+SRE+業務 | なし (机上) |
| パーシャルフェイルオーバー演習 | 四半期 1 回 | SRE+開発 | 最小 (深夜) |
| フルフェイルオーバー演習 | 年 1 回 | 全社 | 計画的 (週末) |
| バックアップリストア検証 | 月 1 回 | SRE | なし (DR 環境) |
| ランブック実走 | 月 1 回 | SRE | なし (DR 環境) |

### 5.2 演習成功基準

- RTO 達成 (実測値が目標 RTO の 80% 以内)
- RPO 達成 (データロス < 目標 RPO)
- ランブック逸脱 0 件
- ステークホルダー連絡 SLA 達成

## 7. データバックアップ戦略

### 7.1 バックアップ種別

| データ | 方式 | 頻度 | 保管場所 | 保管期間 |
|------|------|------|--------|--------|
| Azure SQL | PITR | 連続 (5分) | リージョン内 + Geo | 35日 + 月次 1年 |
| Blob (録音) | Soft Delete + Versioning | 都度 | Same + Geo | 5年 (法定) |
| Databricks Delta | Time Travel | 都度 | リージョン内 | 30日 |
| 設定 (Terraform/Bicep) | Git タグ | 都度 | GitHub + アーカイブ | 永続 |

### 7.2 バックアップ取得確認

- 月次: 全バックアップから無作為に1件リストア検証
- 結果はダッシュボードに自動記録
