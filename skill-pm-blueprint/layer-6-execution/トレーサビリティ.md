# トレーサビリティマトリクス

## 目的

プロジェクトの各成果物を**双方向に辿れる**ようにする。

- **上から下** (Top-down): 「このビジネス目標はどのタスクで実現されるか」
- **下から上** (Bottom-up): 「このタスクは何のために存在するか」

青写真段階でトレーサビリティを確立すると:

- スコープ変更時の影響範囲が即座に判る
- 「この機能は何のために?」の質問に即答できる
- 矢羽完了時の受入判定が客観的になる
- 監査対応が容易 (規制業界で特に重要)

## マトリクス構造

```
[L1 ビジネス目標]
    ↕
[L2 仮説 / JTBD]
    ↕
[L4 ユースケース (要件定義側のL4, WBS側のL4と区別)]
    ↕
[L3 ADR]
    ↕
[L5 リスク]
    ↕
[L6 WBS L4 タスク]
    ↕
[スプリントチケット (矢羽③④⑤のみ)]
```

矢印は**双方向参照**を意味する。各オブジェクトは自分が紐づく相手の ID を持つ。

## 実装形式

### 軽量版: Excel / Notion / Google Sheets

年間 1 億円規模では**Excel or Notion で十分**。専用ツール (Jama, Polarion 等) は不要。

列:

```
| ID | 種別 | 内容 | 紐付け先ID | ステータス | 最終更新日 |
```

種別: goal / hypothesis / useCase / adr / risk / nfr / wbs / ticket

### 中量版: GitHub Issues + YAML

リポジトリに全成果物を YAML で持ち、ID でリンクする方式。`ccpm` の PRD → Epic → Task → Issue → Code パターンに近い。

```yaml
# goals/G-001.yml
id: G-001
type: goal
title: "経営会議の意思決定速度を 2 倍にする"
owner: "CEO"
status: active

# hypotheses/H-001.yml
id: H-001
type: hypothesis
parent: G-001
title: "経営層はKPIダッシュボードを週1回閲覧する"
evidence: "役員3名インタビュー 2026-05-10"
status: validated

# useCases/UC-001.yml
id: UC-001
type: useCase
parents: [H-001, H-005]
title: "経営層向けKPIダッシュボード閲覧"
nfr: [NFR-001, NFR-005, NFR-010]
adr: [ADR-0001, ADR-0005]
risks: [R003]

# risks/R-003.yml
id: R-003
type: risk
linkedUseCases: [UC-001, UC-003]
title: "PII 漏洩"
killCriterion: "PII関連インシデント発生時即停止"

# wbs/WBS-4.3.2.yml
id: WBS-4.3.2
type: wbsL4
parent: L3-売上Mart開発
title: "売上 Mart の RLS (Row-Level Security) 実装"
useCase: [UC-001]
risk: [R-003]
nfr: [NFR-010]
estimate: 0.3  # 人月
assignee: "データエンジニアB"
```

### 重量版: 専用RTM (Requirements Traceability Matrix) ツール

Jama Connect、Polarion、IBM DOORS。年間 1 億円では過剰投資。

## 列の定義

| 列 | 意味 | 記入例 |
|---|------|-------|
| ID | 一意識別子 | G-001, UC-001 |
| 種別 | 上記 8 種別のいずれか | useCase |
| 内容 | 1 行サマリー | 経営層向けKPIダッシュボード閲覧 |
| 紐付け先ID | 上位・下位オブジェクトの ID (カンマ区切り) | H-001, ADR-0001, NFR-005, R-003 |
| ステータス | 提案中 / 承認 / 実装中 / 完了 / 非推奨 | 承認 |
| 最終更新日 | YYYY-MM-DD | 2026-05-15 |
| オーナー | 責任者 | 山田 |

## トレーサビリティの典型パス

### パス 1: ビジネス目標からタスクへ (Top-down)

```
G-001 経営会議の意思決定速度 2 倍
 └── H-001 経営層はKPIを週1閲覧する (検証済)
      └── UC-001 経営層向けKPIダッシュボード
           ├── ADR-0001 DWH = Databricks
           ├── ADR-0005 認証 = Entra ID
           ├── NFR-001 P95 応答 3秒
           ├── NFR-005 可用性 99.5%
           ├── NFR-010 MFA必須
           ├── R-003 PII漏洩 (Kill基準: 即停止)
           └── WBS-4.3.2 売上Mart RLS 実装
                └── Ticket JIRA-456 (Sprint 4)
```

### パス 2: リスクから影響範囲へ (Risk-driven)

```
R-003 PII漏洩
 ├── 影響UC: UC-001, UC-003, UC-005
 ├── 関連NFR: NFR-010, NFR-015, NFR-020
 ├── 関連ADR: ADR-0003 (LLM選定)
 └── 関連WBS: WBS-3.2.1, WBS-4.3.2, WBS-5.1.3
```

R-003 が顕在化した場合、上記の UC/NFR/ADR/WBS すべてに波及。

### パス 3: 変更要求の影響分析 (Change Impact)

業務部門から「ユーザー UC-001 に新機能 X を追加したい」と要求があった時:

- UC-001 の紐付けを辿る
  - 既存の NFR-001 で応答時間 3 秒以内が要求されている → 新機能が性能に影響しないか要検証
  - ADR-0005 認証方式に変更の可能性 → ADR を再オープン
  - R-003 PII 漏洩リスクが悪化する可能性 → リスクレジスタ更新
  - WBS-4.3.2 の工数見直し
  - 関連 Sprint チケットへの反映

## 運用方法

### 更新頻度

- **矢羽間の遷移時**: 全項目の棚卸し
- **月次**: ステータス確認
- **変更要求時**: 都度更新 (影響範囲分析の起点)

### オーナー

- **全体のマトリクス運用**: PM
- **各オブジェクトの内容**: 種別ごとの担当者
  - goal / hypothesis: プロダクトオーナー
  - useCase / NFR: 業務 + 開発リード
  - ADR: アーキ
  - risk: リスクオーナー (各担当)
  - WBS / ticket: 開発リード

## ツール選択ガイドライン

| プロジェクト規模 | 推奨ツール | 理由 |
|----------------|-----------|------|
| 3〜5 人, 6ヶ月 | Notion or Excel | 軽量で十分 |
| 6〜10 人, 12ヶ月 | GitHub Issues + YAML | Git 管理で変更履歴も残る |
| 10 人超, 18ヶ月 | GitHub Issues or Linear + 独自スクリプト | Issue の階層管理を補強 |
| 金融・医療・組込 | Jama / Polarion | 規制対応で必要 |

年間 1 億円規模は通常 6〜10 人 / 12〜18ヶ月なので、**GitHub Issues + YAML** を推奨。

## 参考パターン (automazeio/ccpm)

ccpm は以下のフローを提案:

```
PRD (プロダクト要件)
 └── Epic (大きな機能群)
      └── Task (実装単位)
           └── GitHub Issue (個別作業)
                └── Code (実装)
```

本スキルの L2〜L6 と対応:

| 本スキル | ccpm |
|---------|------|
| Layer 1 (経営) | - |
| Layer 2 (仮説) | PRD |
| Layer 4 (要件) | Epic |
| Layer 6 L3 | Epic |
| Layer 6 L4 | Task |
| Sprint チケット | Issue |
