# 経営層 意思決定テンプレート

## 3 種のテンプレート

- **A. 投資判断** (Go / No-Go) — プロジェクト開始前、各矢羽ゲート前
- **B. ピボット判断** (Pivot / Continue) — 方向転換を検討する際
- **C. 撤退判断** (Exit / Continue) — Kill基準超過時、事業環境激変時

各テンプレートは**問いのリスト**であり、問いに答えていくことで経営層が判断できる状態になる。

---

## A. 投資判断テンプレート

### 目的
このプロジェクトに投資する価値があるか?

### 答えるべき問い

#### A-1. 期待される成果 (定量・定性)

- [ ] このプロジェクト完了時に実現する**定量的**成果は?
  - 例: 月次集計時間が 2 人日 → 0.5 人日 (年間 180 人日 = 4,500 万円相当の削減)
  - 例: 機会創出 (クロスセル分析による売上 +2%)
  - 例: コンプライアンス対応コスト削減

- [ ] 定性的成果は?
  - 例: データドリブン文化の浸透
  - 例: 意思決定スピードの向上
  - 例: 属人化の解消

- [ ] 成果の**確からしさ**は? (確実 / 80% / 50% / 不確実)

#### A-2. 投資額の内訳

| 項目 | 想定額 | 備考 |
|------|-------|------|
| 内部人件費 (社内メンバ) | X 円 | PM, アーキ, データエンジニア |
| 外部人件費 (パートナー) | Y 円 | dbt 開発, BI 開発 |
| ツール・ライセンス費 | Z 円 | Databricks, Power BI, dbt Cloud 等 |
| クラウド従量費 | W 円 | 開発環境 + 本番 初年度 |
| 教育・コンサル費 | V 円 | ハンズオン, 外部レビュー |
| 予備費 (全体 15%) | U 円 | バッファ |
| **合計** | **(初期) + 運用** | 運用費は年次計上 |

#### A-3. 回収期間 (Payback Period)

- [ ] 単純回収: 初期投資 / 年間効果額 = 何年?
- [ ] 3 年後、5 年後の累積効果は?
- [ ] 運用コストの成長率 (データ量増 → コスト増) を考慮した現実的な回収は?

#### A-4. 3 つのシナリオと期待値

| シナリオ | 発生確率 | 成果 | 投資コスト | 純効果 |
|---------|---------|------|----------|-------|
| 楽観 | 20% | 想定の 120% | 想定の 100% | +20% |
| 現実 | 60% | 想定の 100% | 想定の 110% | -10% |
| 悲観 | 20% | 想定の 60% | 想定の 130% | -40% |
| 期待値 | - | - | - | -8% |

- [ ] 悲観シナリオでも撤退ラインを下回らないか?
- [ ] 期待値が投資額を超えるか?
- [ ] どのシナリオが最もありうるか (ステークホルダー間で合意形成)

#### A-5. Go/No-Go 判断基準

以下すべて Yes なら Go、1 つでも No なら要再検討:

- [ ] 期待値 (3 年合計) が投資額を超える
- [ ] 悲観シナリオでも大損失 (事業継続不可) にならない
- [ ] Layer 5 で洗い出された最優先リスクすべてに緩和策がある
- [ ] ステークホルダー (経営・業務・情シス) 全員がスコープに合意
- [ ] Kill基準 (撤退条件) が設定済み

#### A-6. 代替案 (やらない選択肢の機会コスト)

- [ ] このプロジェクトをやらない場合、どんな損失が発生するか?
- [ ] 同じ予算で他にどんな投資ができるか? (Opportunity cost)
- [ ] 「やらない」が最適な場合、代わりに何をすべきか?

### 出力フォーマット

```markdown
# プロジェクト投資判断書

## エグゼクティブサマリー
- プロジェクト: [名称]
- 投資額: [初期] + [運用/年] × N 年
- 期待値 (3年累計): [金額]
- Go/No-Go 推奨: [Go / No-Go / 条件付きGo]
- 主要リスク: [Top 3]

## 期待される成果
(A-1 への回答)

## 投資額内訳
(A-2 の表)

## 回収シナリオ
(A-3, A-4)

## Go/No-Go 判断基準との照合
(A-5 のチェックリスト状態)

## 代替案
(A-6 への回答)

## 推奨事項
- 推奨: [Go / No-Go / 条件付きGo]
- 条件: (もしあれば)
- 次ステップ: キックオフ日、初期メンバ、矢羽①開始
```

## B. ピボット判断テンプレート

### 目的
現在の方向性を続けるか、方向転換するか?

### 答えるべき問い

#### B-1. Kill基準の照合 (Layer 5 との連携)

- [ ] Kill基準 (リスクレジスタの各リスクに設定済み) と現状を照合
- [ ] Kill基準に「近づいている」指標は? (ワーニングサイン)
- [ ] Kill基準を超えたリスクがあるか?

| リスクID | タイトル | Kill基準 | 現状 | 距離 |
|---------|---------|---------|------|------|
| R001 | データソース仕様変更 | 月3回以上 & 成功率<90% | 月1回、成功率95% | 余裕あり |
| R007 | Databricks コスト超過 | 月次 150% 超が 3ヶ月連続 | 月次 120% が 2ヶ月連続 | 警戒 |

#### B-2. ピボット選択肢 (Eric Ries の分類)

| パターン | 内容 | データ分析基盤での例 |
|---------|-----|---------------------|
| Zoom-in | 機能の 1 つを新プロダクト化 | 「セグメント抽出」だけを独立サービス化 |
| Zoom-out | 機能を大きな製品の一部へ | 分析基盤を全社データガバナンスの一部に |
| Customer segment | ターゲット顧客変更 | マーケ部向けから経営企画向けへ |
| Customer need | 解決する課題変更 | KPI可視化 → データ品質ガバナンスへ |
| Platform | プラットフォーム変更 | Lakehouse → Warehouse 回帰 |
| Technology | 技術変更 | 自社開発 → SaaS (Tableau等) |
| Channel | 提供経路変更 | Web UI → Slack ボット |

#### B-3. ピボットに要する追加投資と時間

- [ ] ピボットに必要な追加コスト?
- [ ] ピボットに必要な追加期間?
- [ ] 既存成果物で再利用可能なもの?
- [ ] 既存成果物で捨てるもの?

#### B-4. ピボットしない場合のリスク

- [ ] このまま続行した場合の期待ROI (B より下がる見込み)
- [ ] 悪化傾向は継続するか、反転する可能性があるか
- [ ] ステークホルダーの信頼を失うリスク

#### B-5. 推奨判断

- Pivot A (選定): 理由、追加投資、期待 ROI
- Pivot B (次点): 理由、比較
- Continue (継続): 理由、警戒ポイント
- Exit (撤退): C のテンプレートへ

### 出力フォーマット

```markdown
# ピボット判断書

## 現状サマリー
- プロジェクト: [名称]
- 現フェーズ: 矢羽X
- 進捗: N%
- 主要懸念: [Kill基準近接リスク]

## Kill基準照合
(B-1 の表)

## ピボット選択肢
- 選択肢 A: [内容、コスト、期間、期待ROI]
- 選択肢 B: [内容、コスト、期間、期待ROI]
- 継続: [コスト、期待ROI]

## 推奨判断
[Pivot A / Pivot B / Continue / Exit へ]
根拠: [...]
次ステップ: [...]
```

## C. 撤退判断テンプレート

### 目的
撤退するか、継続するか?

### 答えるべき問い

#### C-1. Sunk Cost の明示 (無視する原則)

- [ ] 現時点までに投資した額: X 円 (時間は戻らない、投資は戻らない)
- [ ] この金額は**今後の判断に影響させてはいけない**
- [ ] 「ここまで来たんだから…」は撤退判断の最大の敵

#### C-2. 残存価値の評価

- [ ] 現時点で完成している成果物は?
  - 例: 業務要件定義書、ADR、PoC結果
- [ ] これらを転用・再利用できるか?
  - 例: 別プロジェクトへの流用
  - 例: 社内資産としての利用
  - 例: 業界知見として公開

#### C-3. 撤退コストの算出

- [ ] 契約解除の違約金? (クラウド, ベンダー, ライセンス)
- [ ] パートナー精算?
- [ ] データの処分・移行コスト?
- [ ] ステークホルダーへの説明コスト?
- [ ] 関連プロジェクトへの影響?

#### C-4. ステークホルダー説明ロジック

経営層以外 (業務部門, 情シス, 外部パートナー) への説明を準備:

- [ ] 撤退の理由を 3 行で説明できるか
- [ ] 「何を学んだか」を言語化できるか
- [ ] 「次どうするか」(代替投資, やり直し) が準備されているか
- [ ] 責任論にならないように説明できるか (組織学習に昇華)

#### C-5. 撤退 or 継続の最終判断基準

以下を総合して判断:

- [ ] 今後の投資で回収できるか (完了までの追加投資額 vs 回収見込み)
- [ ] 同じ予算を他に回した方がリターンが高いか (Opportunity cost)
- [ ] 関係者の士気は?
- [ ] 戦略的意義は?

### 出力フォーマット

```markdown
# 撤退判断書

## 経緯
- プロジェクト: [名称]
- 開始: [日付]
- 現状: [フェーズ, 進捗]
- 累計投資: [金額] (判断には含めない)

## 撤退判断の背景
(トリガー: Kill基準超過, 外部環境変化, 等)

## 残存価値
- 再利用可能な成果物: [...]
- 組織学習: [...]

## 撤退コスト
(C-3 の内訳)

## 継続 vs 撤退 比較
| 選択 | 追加投資 | 期待ROI | リスク |
|------|---------|--------|-------|
| 継続 | X | Y | ... |
| 撤退 | 撤退コストZ | 残存価値 + 他投資 | ... |

## 推奨判断
[撤退 / 継続]
根拠: [...]

## ステークホルダー説明ロジック
- 業務部門向け: [...]
- 情シス向け: [...]
- パートナー向け: [...]
```

## データ分析基盤での適用例 3 件

### 例 1 (A 投資判断): 小売業のデータ分析基盤

**サマリー:**
- 投資額: 初期 1.2 億円 + 運用 年 3000 万円
- 期待成果: 分析工数削減 年 5000 万円相当、機会創出 年 3000 万円
- 3 年累計期待値: +1.0 億円 (投資回収)

**推奨: 条件付き Go**
- 条件: 矢羽①完了時に業務部門承認を得ること
- 条件: 矢羽②PoC で主要3ユースケースの実現可能性を確認すること

### 例 2 (B ピボット判断): 中盤での SaaS 化検討

**状況:**
- 矢羽④ AI分析 Sprint 中に、類似機能の SaaS 製品が急速に成熟
- 自社開発コストが高騰、SaaS 採用で 30% コスト削減の試算

**推奨: Technology Pivot (自社 → SaaS)**
- 追加投資: 3000 万円 (既存投資の一部放棄)
- 期間: +3ヶ月
- 期待: 運用費 30% 減、保守負担減

### 例 3 (C 撤退判断): データソース契約破綻

**状況:**
- 主要データソースである POS ベンダーが倒産
- 代替データソース整備に 9ヶ月、追加 5000 万円必要
- 代替期間中の業務損失は計り知れない

**推奨: 撤退**
- Sunk cost: 7000 万円 (無視)
- 撤退コスト: 契約解除金 1000 万円 + パートナー精算 500 万円
- 残存価値: 業務要件定義書 (将来のリトライで流用可能)
- 代替案: 現行 Excel 体制継続 + 別ベンダー再選定検討 (6ヶ月後)

## 作成時のチェックリスト

- [ ] 3 テンプレートのどれを使うべきか明確になっている
- [ ] 数値は現実的 (楽観バイアス排除)
- [ ] 悲観シナリオが十分厳しい
- [ ] Kill基準との照合が客観的
- [ ] Sunk cost に引きずられていない
- [ ] ステークホルダー説明のストーリーが準備できている
- [ ] 代替案が複数検討されている
