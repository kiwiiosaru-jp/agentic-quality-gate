# アーキスタイル選定 (Topology / Internal Style / Hosting Tier)

## 対応する品質ゲート (Phase 2)

| ID | 観点 | 既定の Fail/Conditional 条件 |
|----|------|------|
| ARCH-MONOLITH-001 | 配置トポロジ | モノリス / モジュラーモノリス / マイクロサービスの選択根拠が不在 |
| ARCH-STYLE-001 | 内部スタイル | レイヤード / ヘキサゴナル / オニオン / クリーン の宣言なし |
| ARCH-LAYER-001 | ホスティング階層 | IaaS / PaaS / SaaS の責任分界点が宣言されていない |

## 概要

「アーキテクチャ」と一括りにされがちだが、3 つの異なる軸がある:

1. **配置トポロジ (Topology)** : 何個のプロセス・デプロイ単位に分けるか
2. **内部スタイル (Internal Style)** : 1 つのコードベース内をどう層化するか
3. **ホスティング階層 (Hosting Tier)** : 自社で運用する範囲と他社 (クラウド) に委ねる範囲

3 軸はそれぞれ独立に選択でき、組み合わせは多数。**選定根拠の明文化** と **ARB (Architecture Review Board) 承認** を経て、ADR として固定する。

本スキルは Phase 2 (本格スケール期) の意思決定を想定。Phase 1 はモノリス + PaaS 中心で開始し、痛点が見えてから本スキルを発動する。

## いつ使うか

- Phase 2 着手前 (利用者数 / データ量が初期計画の 5 倍を超えた、組織が 2 チーム → 6 チームに拡大、等)
- 主要機能を別チームが並行開発するため独立デプロイが要る時
- SaaS 化 (マルチテナント) や海外展開を企画する時
- レガシーモノリスの分割計画 (Strangler Fig パターン) を立てる時

## 軸 ① 配置トポロジ (ARCH-MONOLITH-001)

| トポロジ | 特徴 | 適性 |
|---------|-----|-----|
| **Monolith** | 1 つのプロセス、1 つのデプロイ単位 | 〜10 人、〜中規模、検証フェーズ |
| **Modular Monolith** | 1 プロセスだがモジュール境界が明確 | 10〜30 人、将来の分割を見据える |
| **Microservices** | サービス毎に独立プロセス・独立デプロイ | 30 人〜、組織の自律性が必要 |
| **Service-based** | 数個 (3〜10) の中粒度サービス | マイクロサービスの過剰さを避けたい時 |
| **Serverless / FaaS** | 関数単位、イベント駆動 | スパイク負荷、低頻度処理 |
| **Macro Services (Modular Monolith + Workers)** | コアはモノリス + バッチ/分析だけ別 | 多くの企業の現実解 |

### 選定マトリクス (8 観点スコアリング)

| 観点 | Monolith | Modular Monolith | Microservices |
|-----|---------|----------------|--------------|
| 開発速度 (初期) | ◎ | ○ | × |
| 開発速度 (中規模以降) | × | ○ | ◎ |
| 運用コスト | ◎ | ○ | × |
| 障害分離 | × | △ | ◎ |
| 独立デプロイ | × | △ | ◎ |
| トランザクション一貫性 | ◎ | ◎ | △ (Saga 必須) |
| 必要な組織成熟度 | 低 | 中 | 高 |
| 観測性必要度 | 低 | 中 | 高 |

### 鉄則

- **「マイクロサービスにすればスケールする」は嘘**。組織が分割に耐えないと逆効果
- **Modular Monolith から始めて、痛点ができてから一部分を切り出す**
- 1 サービス = 1 チームの「2 ピザ ルール」。チーム数 < サービス数 になると破綻する

### コールセンター実例 (現実的解)

```yaml
topology_decision:
  current_phase: Phase 1 (検証)
  topology: Modular Monolith + 別バッチ
  modules:
    - inquiry_intake     # Genesys 受信、Postgres 永続化
    - inquiry_search     # 過去問合せ検索
    - voc_analysis_api   # Databricks Gold への問い合わせ
    - admin              # 管理画面
  separate_workers:
    - voc_outbox_worker  # Postgres → Event Hubs
    - dlq_processor      # Dead Letter 再処理
  rationale: |
    検証フェーズで利用者 50 名想定。マイクロサービス化すると運用負荷が予算超過。
    将来 voc_analysis_api だけ分割する余地を残してモジュール境界は明確化。
  trigger_to_split:
    - voc_analysis_api の応答時間が他モジュールに影響を与え始めた時
    - チーム数が 3 → 6 に拡大した時
```

## 軸 ② 内部スタイル (ARCH-STYLE-001)

1 プロセス内をどう層化するか。すべて「ビジネスロジックを外部依存から守る」発想だが、純度が違う。

| スタイル | 特徴 | 採用例 |
|---------|-----|-------|
| **Layered (N-tier)** | プレゼン → アプリ → ドメイン → DB の縦並び | 一般業務系、多数の Web アプリ |
| **Hexagonal (Ports & Adapters)** | ドメインを中心、外部は Adapter で繋ぐ | 外部依存の入替が頻繁 |
| **Onion** | ドメインを中心の同心円、外側ほど不安定 | DDD 採用プロジェクト |
| **Clean Architecture** | UseCase 層を明示し、依存方向を内向きに統一 | 大規模ビジネスロジック |
| **Vertical Slice** | 機能 (Feature) ごとに垂直方向で分割 | Feature Flag / 並行開発 |
| **Event-Driven (Internal)** | 内部メソッドを Event Bus 経由で呼ぶ | Eventual Consistency が前提 |

### 選定の指針

- **依存対象が安定している場合は Layered で十分**
- **DB / 外部 API / メッセージング が取り替わる可能性が高ければ Hexagonal / Onion / Clean**
- **Clean Architecture は学習コストが高い**。チーム全員が理解できないなら Hexagonal で止める
- **Vertical Slice はチーム規模が大きい時の並行開発向け**

### 推奨レイヤ構成 (Hexagonal の現実解)

```
src/
├── domain/                  # ビジネスロジック (依存なし、純粋 Python/TS)
│   ├── entities/
│   ├── value_objects/
│   └── domain_services/
├── application/             # UseCase (Domain を呼ぶ)
│   ├── use_cases/
│   └── ports/               # 外部 IO のインタフェース
├── adapters/                # Port の実装
│   ├── primary/             # API / CLI / Webhook (入力側)
│   │   ├── http/
│   │   └── grpc/
│   └── secondary/           # DB / 外部 API (出力側)
│       ├── postgres/
│       ├── eventhub/
│       └── databricks/
└── infrastructure/          # DI コンテナ、設定、起動コード
```

### 鉄則

- **依存方向は外 → 内のみ**。domain は adapters を import してはいけない
- **Port (interface) を application に置き、実装を adapters に置く**
- **ドメイン層に DB ライブラリ / HTTP クライアントを呼ぶコードを書かない**

## 軸 ③ ホスティング階層 (ARCH-LAYER-001)

責任分界点を「どこまで自分達で運用するか」で決める。

| 階層 | 責任範囲 (自分側) | 例 |
|------|----------------|-----|
| **オンプレ** | ハードウェア + OS + ミドル + アプリ | データセンター、VPN |
| **IaaS** | OS + ミドル + アプリ | EC2 / Azure VM / GCE |
| **CaaS (Container)** | コンテナ + アプリ | EKS / AKS / GKE |
| **PaaS** | アプリ + 設定 | Container Apps / App Service / Lambda |
| **SaaS** | 設定のみ | Genesys / Salesforce / Snowflake |

### 階層別判断軸

```
                  カスタム性 ←─────────────→ 運用負荷の低さ
オンプレ ── IaaS ── CaaS ── PaaS ── SaaS
                     ↑              ↑
                 標準 OSS 動かしたい   ベンダーサービスで十分
```

### 推奨選び方

1. **要件を満たす SaaS があれば最優先で SaaS** (Genesys, Salesforce, Snowflake)
2. **業務ロジック固有性が高ければ PaaS** (Container Apps, Lambda)
3. **PaaS で動かないコンテナだけ CaaS** (AKS, EKS)
4. **OS レベルカスタムが要るものだけ IaaS** (殆ど不要)
5. **オンプレは新規プロジェクトでは原則ゼロ**

### コールセンター実例 (Hosting 配置)

```yaml
hosting_tier:
  - capability: 顧客向け IVR / 音声基盤
    tier: SaaS (Genesys Cloud)
    rationale: 業界標準、運用 0、SLA 99.99%
    boundary:
      genesys_responsibility: SBC, Media Server, ACD, Recording
      our_responsibility: Webhook 受信ロジック, 業務 API 連携
  - capability: 業務 API (受付・検索・分析)
    tier: PaaS (Azure Container Apps)
    rationale: コンテナ運用 0、autoscale, KEDA 対応
  - capability: 分析基盤 (DWH / ETL)
    tier: SaaS (Databricks on Azure)
    rationale: クラスタ管理 0、Unity Catalog
  - capability: 業務 DB
    tier: PaaS (Azure Database for PostgreSQL Flexible Server)
    rationale: パッチ・バックアップ・冗長を Microsoft 管理
  - capability: ID / 認証
    tier: SaaS (Microsoft Entra ID)
    rationale: 既存社員認証と統合、SSO/MFA 標準
  - capability: ファイル保管
    tier: PaaS (Azure Blob Storage)
    rationale: Immutable Policy 対応 (WORM 要件)
```

## 軸 ④ ARB (Architecture Review Board) 承認フロー

トポロジ・内部スタイル・Hosting の選定は **不可逆 (Type 1) 決定** 。ARB を通して承認を得る。

### ARB の役割

- アーキ整合性チェック (社内既存ガイドラインとの整合)
- リスク評価 (将来の負債、ロックイン)
- セキュリティ・コンプライアンス観点レビュー
- コスト影響 (TCO) レビュー

### 構成メンバー (5〜7 名)

| 役割 | 主担当 |
|-----|------|
| Chief Architect | アーキ整合 |
| プロダクトオーナー | 業務観点 |
| セキュリティリード | リスク観点 |
| プラットフォーム/SRE リード | 運用観点 |
| データガバナンス担当 | データ観点 |
| (オブザーバ) ファイナンス | コスト観点 |

### ARB 提出資料テンプレート

```markdown
# ARB 提案 #NNN: <タイトル>

## 1. 概要
- 提案者 / 提案日
- 対象スコープ (どのプロジェクト / どの境界)

## 2. 背景・課題
現状の何が問題で、なぜこの提案が必要か。

## 3. 提案 (3 軸全て記述)
### 配置トポロジ
- 選択: Modular Monolith
- 理由: …

### 内部スタイル
- 選択: Hexagonal
- 理由: …

### ホスティング階層
- 選択: PaaS (Container Apps) + SaaS (Databricks, Genesys)
- 理由: …

## 4. 代替案と却下理由
3 案以上記述

## 5. リスクと対策
- ロックインリスク
- 移行コスト
- 運用スキルギャップ

## 6. TCO (3 年)
| 区分 | Year1 | Year2 | Year3 | 合計 |

## 7. 承認結果
- [ ] 承認 / [ ] 条件付承認 / [ ] 差し戻し
- 条件: …
- 承認日 / 承認者
```

### ARB 議事録テンプレート

```yaml
arb_meeting:
  date: 2026-05-30
  proposal: ARB-PR-007
  decision: 条件付承認
  conditions:
    - 6 ヶ月後にトポロジ再評価ゲートを設置
    - SLO 違反時は分割計画を発動
  participants: [Chief Architect, PO, Security, SRE, DataGov]
  follow_up_adr: ADR-0030
```
