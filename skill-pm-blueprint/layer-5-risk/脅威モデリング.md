# 脅威モデリング

## 概要

セキュリティリスクは通常のリスクレジスタだけでは網羅しにくい。Microsoft が提唱した **STRIDE** フレームワークを用い、システムの各要素に対して 6 種類の脅威を体系的に洗い出す。

STRIDE:

| 記号 | 脅威 | 対応する性質 |
|-----|------|-------------|
| S | Spoofing (なりすまし) | Authentication (認証) |
| T | Tampering (改ざん) | Integrity (完全性) |
| R | Repudiation (否認) | Non-repudiation (否認防止) |
| I | Information disclosure (情報漏洩) | Confidentiality (機密性) |
| D | Denial of service (DoS) | Availability (可用性) |
| E | Elevation of privilege (権限昇格) | Authorization (認可) |

## いつ使うか

- Layer 3 アーキテクチャ (C4 Container図) ができた直後
- PII/機微情報を扱うコンポーネントの設計前
- 外部 API/ベンダーとの連携設計時
- 認証認可方式の決定前
- Kill基準に「セキュリティ事故発生」を含める前

## 手順

### ステップ 1: 対象システムの分解

C4 Container 図を用意し、各コンポーネントを洗い出す。データ分析基盤の典型例:

- データソース (POS, CRM, 会員DB, Web行動ログ, 外部SaaS)
- 取り込み層 (Ingest/ELT)
- データレイク/DWH (Bronze/Silver/Gold)
- 変換層 (dbt, LLMパイプライン)
- 配信層 (BI, API, Semantic Layer)
- 利用者 (経営層, マーケ部, 店舗, 外部連携)

### ステップ 2: 信頼境界の引き方

以下のルールで信頼境界を引く:

1. **組織境界**: 自組織 / 外部ベンダー / 取引先 / エンドユーザー
2. **ネットワーク境界**: VPC / オンプレ / インターネット / クラウド間
3. **認証境界**: 認証済み / 未認証
4. **権限境界**: 管理者 / 一般利用者 / 外部監査者
5. **データ機密度境界**: PII / 機微情報 / 一般業務情報 / 公開情報

信頼境界をまたぐデータフローは全て脅威モデリング対象。

### ステップ 3: データフローの列挙

各信頼境界をまたぐフローに ID を付ける。例:

| ID | From | To | データ内容 | 機密度 |
|----|------|-----|----------|-------|
| F001 | POS | ingest-job | 購買データ (顧客ID含む) | PII |
| F002 | 会員DB | Bronze 層 | 会員情報 (氏名・住所・誕生日) | PII |
| F003 | Bronze 層 | Silver 層 | 匿名化後の購買データ | 機微 |
| F004 | Gold 層 | BIツール | 集計済みKPI | 一般 |
| F005 | LLM API | 変換層 | 要約テキスト (入力に問い合わせ文含む) | PII可能性 |

### ステップ 4: STRIDE 適用

各フローに対して STRIDE を 1 個ずつ当てはめて「起きうるか」を検討。

**フロー F002 (会員DB → Bronze 層) の例:**

| 脅威 | シナリオ | 緩和策 |
|-----|---------|-------|
| S | ingest-job のサービスアカウント資格情報が漏洩し、会員DB が読まれる | Secrets 管理 (KeyVault等)、短命トークン、監査ログ |
| T | Bronze 層書き込み中にデータが改ざんされる | 書き込み後 ハッシュ検証、WORM (Write Once Read Many) ストレージ |
| R | Bronze 層への書き込みログが消されて誰が書いたか分からない | 監査ログを別アカウントへ送信、タンパープルーフ |
| I | Bronze 層の会員情報が誤って開発者に閲覧される | Row-level security、最小権限、マスキング |
| D | ingest-job の同時実行数制限で業務時間帯に本番取り込みが遅延 | レート制御、キュー分離、優先度付け |
| E | ingest-job のサービスアカウントが DBA 権限を取得 | 最小権限、定期的な権限レビュー、Just-In-Time権限 |

### ステップ 5: 多層防御の設計

脅威ごとに緩和策を 1 つだけでなく**多層**で設ける:

- **予防** (Preventive): 認証、認可、暗号化、設計
- **検知** (Detective): 監査ログ、アラート、異常検知
- **対応** (Responsive): インシデント対応、リストア、連絡網

### ステップ 6: 残存リスクの受容判定

全ての脅威を完全に排除することは不可能。**残存リスク**を明示し、以下のいずれかを選択:

- **緩和** (Mitigate): 緩和策を追加実装
- **転嫁** (Transfer): 保険、ベンダー責任範囲に押し込む
- **受容** (Accept): 影響小のため受容。ただし文書化必須
- **回避** (Avoid): 機能やデータそのものを諦める

## PII 取扱経路の重点チェック

個人情報保護法 (2022 改正施行) の観点で、データ分析基盤では以下を必ず確認:

- [ ] 取得目的の明示と同意の取得経路が明確か
- [ ] 第三者提供 (他部門含む) の経路がリスト化されているか
- [ ] 保有期間の上限と削除プロセスが定義されているか
- [ ] 本人開示請求への対応経路があるか
- [ ] 海外移転 (クラウドリージョン) の根拠が説明できるか
- [ ] 匿名加工情報と個人情報の境界が明確か
- [ ] LLM への入力に PII が混入しないか (プロンプトインジェクション対策)

## 多層防御の設計指針

| 層 | 例 |
|---|----|
| 物理 | データセンターアクセス制限 |
| ネットワーク | VPC, Firewall, PrivateLink |
| 認証 | IdP 連携, MFA |
| 認可 | RBAC, ABAC, Row-level security |
| データ | 暗号化 (保存時/転送時), マスキング, トークン化 |
| アプリ | 入力検証, エスケープ, セキュアコーディング |
| 監査 | ログ, モニタリング, アラート |
| 人 | 教育, 権限レビュー, 内部統制 |

## 出力フォーマット

STRIDE の結果は以下 2 形式で保存:

1. **フロー × STRIDE マトリクス** (Markdown 表)
2. **緩和策一覧** (リスクレジスタに `linked_threat_ids` で紐付け)
