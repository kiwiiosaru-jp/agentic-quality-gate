# ゼロトラスト方針

> **品質ゲート対応**: SEC-ZEROTRUST-001 (ゼロトラストアーキテクチャ採否が未明示), GOV-INTERNAL-SYSTEM-001 (内部システム間通信が「VPC 内だから安全」前提で例外扱いされている)

## 概要

「社内 LAN だから安全」「VPC 内だから暗号化不要」「踏み台経由だから多要素認証不要」 - これらは全て古典的な**境界防御モデル**の前提であり、現代のクラウド + リモートワーク + サプライチェーン攻撃環境では破綻している。**Zero Trust Architecture (ZTA)** は「決して信頼せず、常に検証する (Never Trust, Always Verify)」を原則とし、全てのアクセスを「未認証・未認可・潜在的脅威」とみなして都度検証する。

ZTA の 3 原則 (NIST SP 800-207):

1. **明示的に検証** (Verify Explicitly) - 全アクセスに認証・認可・コンテキスト判定
2. **最小権限の原則** (Use Least Privilege Access) - JIT, JEA (Just Enough Access)
3. **侵害前提** (Assume Breach) - セグメンテーション, 暗号化, 継続監視

## いつ使うか

- Layer 3 アーキテクチャでオンプレ + クラウド + SaaS のハイブリッド構成が決まったとき
- Layer 5 STRIDE で「Spoofing」「Elevation of privilege」が頻出する場合
- 内部統制部門・監査法人から「内部例外を撤廃せよ」と指摘されたとき
- 新規プロジェクトの認証認可基盤を全面刷新する前
- リモートワーク前提の運用設計時 (BYOD, 出張, 委託先)
- 重大インシデント (内部不正, 委託先からの侵入) 後の再発防止策策定時

## 手順

### ステップ 1: 既存「内部例外」の棚卸し

既存システムで「内部だから」を理由に省略されているセキュリティ統制を全件洗い出す:

| 内部例外パターン | よくある言い訳 | ZTA 違反 |
|------------------|--------------|---------|
| 内部 API は認証なし | 「踏み台経由だから安全」 | Verify Explicitly 違反 |
| 内部 LAN は暗号化なし (HTTP) | 「VPC 内だから盗聴不可」 | Assume Breach 違反 |
| 管理コンソール = 共用 ID/PW | 「社内 IP からのみ許可」 | 最小権限違反 |
| バッチサーバ = root 常時 | 「ログインできるのは SRE だけ」 | 最小権限違反 |
| 委託先 = 共有 VPN アカウント | 「契約があるから信頼」 | Verify Explicitly 違反 |
| DB 直 SSH | 「踏み台経由だから OK」 | 最小権限 + 侵害前提違反 |
| MFA 未設定 (社内アカウント) | 「IP 制限してるから不要」 | Verify Explicitly 違反 |

### ステップ 2: ZTA の 3 原則をデータ基盤・コールセンタ基盤に当てはめる

#### 原則 1: 明示的に検証

| 検証対象 | 実装 |
|---------|------|
| **ユーザ ID** | Microsoft Entra ID + MFA (FIDO2/Authenticator) |
| **デバイス** | Intune 準拠デバイス + Defender for Endpoint 健全性 |
| **ネットワーク** | Conditional Access で社外 IP は MFA 強化、危険地域は Block |
| **アプリ** | アプリ毎に Entra ID Enterprise App + 属性ベース許可 |
| **時間** | 業務時間外は Step-up 認証 |
| **データ機密度** | PII アクセスは追加 MFA + 利用目的の記録 |

#### 原則 2: 最小権限

```
[標準権限] = 業務 80% を満たす最小セット
       ↓ 必要時のみ
[一時昇格 JIT/PIM] = 残り 20% (時間制限・承認・記録付き)
       ↓ 異常時のみ
[緊急昇格 Break-glass] = 最終手段 (二人承認・全件記録・事後レビュー)
```

JIT (Just-In-Time) 昇格の運用:
- Microsoft Entra ID Privileged Identity Management (PIM) を使用
- 標準: 1 時間の Activation, 業務理由必須
- 重要権限 (Global Admin, Key Vault Admin): 承認者必須
- 全アクティベーションを Sentinel に転送、四半期レビュー

#### 原則 3: 侵害前提

| 対策 | 内容 |
|------|------|
| **マイクロセグメンテーション** | NSG/ASG で「データ層」「アプリ層」「管理層」を分離。同一 VNet 内も deny-by-default |
| **東西通信の暗号化** | サービス間 mTLS、Azure Private Link、サービス専用 ID |
| **ログ集中** | Sentinel + Log Analytics + 90 日以上保管 + Immutable Storage |
| **異常検知** | UEBA (User Entity Behavior Analytics)、Defender for Cloud |
| **インシデント応答** | Playbook 自動化 (Logic Apps + Sentinel)、訓練年 2 回 |

### ステップ 3: Microsoft Entra ID を中心とした ZTA 構成

```
┌──────────────────────────────────────────────────────┐
│ 入口 (任意のネットワーク・任意のデバイス)              │
└──────────────────────────────────────────────────────┘
                          ▼
┌──────────────────────────────────────────────────────┐
│ Microsoft Entra ID                                    │
│  ├─ ユーザ認証 (パスワード or パスワードレス)          │
│  ├─ MFA (Authenticator/FIDO2/SMS※非推奨)             │
│  └─ Conditional Access ポリシー評価                   │
│       ├─ デバイス準拠? (Intune)                       │
│       ├─ リスクスコア? (Identity Protection)         │
│       ├─ 場所? (国/IP)                                │
│       └─ アプリ? (機密度別)                           │
│  → 許可 / Block / MFA 追加 / セッション制限          │
└──────────────────────────────────────────────────────┘
                          ▼
┌──────────────────────────────────────────────────────┐
│ Continuous Access Evaluation (CAE)                    │
│  - セッション中もリスク変化を監視                      │
│  - 異常検知時 → 即時セッション失効                     │
└──────────────────────────────────────────────────────┘
                          ▼
┌──────────────────────────────────────────────────────┐
│ アプリ・データ (Databricks / SAP / Genesys / Power BI)│
│  - サービス毎に PIM (JIT 昇格)                         │
│  - データ層に Row/Column Filter                       │
│  - すべての認可判定を OPA / Unity Catalog で集中       │
└──────────────────────────────────────────────────────┘
                          ▼
┌──────────────────────────────────────────────────────┐
│ 監査 (Microsoft Sentinel / Purview)                   │
│  - 認証ログ, 認可ログ, データアクセスログを相関         │
│  - UEBA で異常検知                                    │
│  - Immutable Storage に 7 年保管                      │
└──────────────────────────────────────────────────────┘
```

### ステップ 4: 内部例外撤廃チェックリスト

ステップ 1 で洗い出した例外を以下チェックリストで個別に処置:

- [ ] **内部 API**: ServicePrincipal/Managed Identity で認証必須化、Token-based authz
- [ ] **内部 HTTP 通信**: TLS 1.3 / mTLS への置換 (Service Mesh または Application Gateway)
- [ ] **共用 ID**: 個別 ID への分割、削除確認、移行期限設定
- [ ] **常時 root/sa**: sudo + 監査ログ化、JIT 化、自動失効
- [ ] **委託先共有 VPN**: 個別 ID 発行 + Conditional Access + 終了時自動 deactivation
- [ ] **DB 直 SSH**: Bastion + JIT + セッション録画
- [ ] **MFA 例外**: 例外申請プロセス化、四半期レビュー、最終的に全廃計画
- [ ] **IP allow list 単独**: MFA 併用必須、IP only は禁止
- [ ] **ハードコード認証情報**: Key Vault 化、Managed Identity 化、ローテーション
- [ ] **ローカル管理者アカウント**: LAPS (Local Admin Password Solution) または PIM
- [ ] **共有メールボックス権限**: 個別委任、Audit Log、四半期レビュー
- [ ] **ベンダーリモート保守**: JIT 接続、セッション録画、終了通知
- [ ] **「特権ユーザは MFA 不要」例外**: 即時撤廃 (むしろ特権ほど MFA 必須)

### ステップ 5: JIT 昇格の運用設計

| 役割 | 標準権限 | JIT 昇格時間 | 承認 | 用途 |
|------|---------|------------|------|------|
| Databricks Workspace Admin | なし | 4h (最大) | SRE Lead | 障害対応 |
| Key Vault Administrator | なし | 1h | CISO 代理 | 鍵ローテーション緊急 |
| Global Administrator | なし | 30 分 | CISO + CIO 二者 | 緊急時のみ |
| SAP Basis | 開発環境のみ | 4h (本番) | SAP Lead | 本番作業 |
| Genesys Admin | 標準 Supervisor | 2h (Admin) | CS Director | 録音抽出等 |
| 委託先エンジニア | プロジェクト RG のみ | 8h | プロジェクト PM | 開発作業 |

### ステップ 6: CISO 承認フロー

ZTA 移行は段階的だが、各段階で CISO 承認を必須とする:

```
Phase 0: 現状棚卸し (内部例外チェックリスト)
   → CISO レビュー → 撤廃計画承認
Phase 1: Quick Win (MFA, Conditional Access 基本)
   → CISO 承認 → 全社展開
Phase 2: マイクロセグメンテーション + 内部 mTLS
   → CISO + CTO 承認 (アーキテクチャ変更)
Phase 3: PIM/JIT 全面適用
   → CISO 承認 → 段階展開 (一般→特権)
Phase 4: 内部例外完全撤廃 (残存例外ゼロ)
   → CISO + 監査委員会 報告
Phase 5: 継続改善 (UEBA 高度化, パスワードレス)
   → 四半期 CISO レビュー
```

承認時の必須提出物:

- 撤廃対象の内部例外リスト (棚卸し結果)
- 移行計画 (期限・担当・代替手段)
- リスク評価 (移行中の一時的リスク)
- ロールバック手順
- 監査・運用への影響評価

### ステップ 7: 残存例外の管理

Phase 4 で完全撤廃を目指すが、業務上の制約で例外が残る場合:

- 例外台帳 (`docs/security/zta-exceptions.yaml`) で管理
- 各例外に**期限・代替策・補償統制 (Compensating Controls)** を必須記載
- 期限切れ前 30 日でアラート、CISO に再承認申請
- 例外件数を四半期で公開し、減少傾向を可視化

```yaml
- id: ZTA-EXC-001
  description: "レガシー POS 端末からの非 mTLS 通信"
  systems: ["pos-store-001 to ingest-job"]
  reason: "POS 端末が TLS 1.3 未対応、機種更改 2027 年予定"
  compensating_controls:
    - "専用 VLAN 隔離"
    - "送信先固定 + IP allow list"
    - "全通信ペイロード暗号化 (アプリ層)"
  expiry_date: "2027-03-31"
  approver: "CISO"
  reviewed_at: "2026-04-15"
  next_review: "2026-07-15"
```

## データ分析基盤・コールセンター向け実例ノート

- **「Genesys 録音は社内ネットワークから取り出すから安全」**: 旧来の論理。録音は PII の塊であり、内部経路でも mTLS + 認可必須。録音 DL ボタンは PIM Activation 必須化。
- **「Databricks Workspace は VPC 内だから IAM 緩く OK」**: 不正。Workspace Admin は PIM JIT、データプレーンと Unity Catalog の権限は分離、サービス用 ID は専用ロールに最小化。
- **「SAP は閉域だから MFA 不要」**: 撤廃対象。SAP Fiori Launchpad に Entra ID 認証、PFCG ロールに対し PIM を Entra ID 側で適用 (HR/Finance Admin 権限は JIT 化)。
- **「委託先は VPN つないでるから内部扱い」**: ZTA 違反の典型。委託先には個別 Entra ID ゲストアカウント + Conditional Access + Cloud App Security でセッション監視 + プロジェクト終了時自動失効。
- **「録音 5-10 年保管のため過去録音 DL を許す」**: 用途を限定し、ダウンロード理由 (`purpose_of_use` 必須)、自動失効 ZIP、ダウンロード件数のレート制限を OPA で実装。
- **「LLM API は本番 VNet 内のみだから API キー固定で OK」**: NG。API キーは Key Vault + 90 日ローテーション、サービスは Managed Identity で取得。

## STRIDE との関係

- STRIDE の Spoofing/Elevation を **アーキテクチャレベルで根絶**するのが ZTA の役割。
- STRIDE が「個別フロー単位の脅威」なら、ZTA は「組織全体のセキュリティ前提条件」。
- STRIDE で「内部だから許容」と判断したフローを ZTA は再評価し、**仮定そのものを否定**する。
- 認可方式選定・暗号化_KMS設計と組み合わせて初めて ZTA が成立する。
