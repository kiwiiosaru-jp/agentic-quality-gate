# 認可方式選定

> **品質ゲート対応**: SEC-AUTHZ-DESIGN-001 (認可方式が STRIDE 内で抽象的に扱われており、独立した設計プロセスが不足)

## 概要

データ分析基盤やコールセンター業務では、**誰が・どのデータに・どの操作を・どの条件下で**実行できるかを厳密に定義しなければならない。STRIDE 脅威モデリングは「権限昇格 (E)」を扱うが、認可"方式"そのものの選定はカバーしない。本スキルは認可モデルの比較選定、ポリシー言語の決定、複数 IdP/データ基盤統合方針、および ADR 化までを単独プロセスとして担う。

主要な認可モデル:

| モデル | 中核概念 | 適している場面 |
|-------|---------|--------------|
| **RBAC** (Role-Based) | 役割に権限を紐付ける | 階層が明確、変更頻度が低い社内システム |
| **ABAC** (Attribute-Based) | 主体・資源・環境の属性で判定 | 動的・文脈依存 (時間/場所/データ機密度) の判断が必要 |
| **ReBAC** (Relationship-Based) | エンティティ間の関係グラフで判定 | 「自分の顧客」「自店舗のデータ」など所有/担当関係 |
| **PBAC / OPA** (Policy-Based) | 宣言的ポリシー言語 (Rego等) で集中管理 | 複数サービス横断で一貫した認可、監査要件強い |

## いつ使うか

- Layer 3 アーキテクチャで複数のデータストア (Databricks/SAP/Genesys等) を束ねると決まったとき
- Layer 5 STRIDE で「権限昇格 (E)」の脅威が顕在化したとき
- 顧客データを扱うコールセンター/CRM 連携で「担当顧客のみ閲覧可」要件が出たとき
- 監査ログ提出を伴う上場準備・SOC2/ISO27001 対応の前
- 内部統制部門から「最小権限の根拠資料を提出せよ」と要求されたとき

## 手順

### ステップ 1: 認可要件の棚卸し

要件を 3 軸で書き出す:

| 軸 | 観点 | 例 (コールセンター VoC 基盤) |
|----|------|----------------------------|
| **主体** | 誰が | オペレータ / SV / マーケ部 / 内部監査 / 外部ベンダー |
| **資源** | 何を | 録音ファイル / 文字起こし / 個人情報 / 集計KPI / LLM プロンプト |
| **操作** | どうする | 閲覧 / 検索 / ダウンロード / 削除 / LLM 投入 / API 経由配信 |

特に「検索 (件数のみ) は OK だが個別レコード閲覧は NG」のような**操作粒度**を見落とさない。

### ステップ 2: モデル比較選定

評価軸 (5 段階):

| 評価軸 | 重み | RBAC | ABAC | ReBAC | OPA/PBAC |
|--------|-----|------|------|-------|----------|
| 表現力 (動的条件) | 高 | 2 | 5 | 4 | 5 |
| 既存 IdP 連携容易性 | 中 | 5 | 4 | 3 | 4 |
| 運用負荷 (ポリシー作成) | 中 | 5 | 3 | 3 | 3 |
| 監査追跡性 | 高 | 4 | 4 | 4 | 5 |
| エンジニア習熟度 | 中 | 5 | 3 | 2 | 3 |
| データ基盤統合性 (Unity Catalog 等) | 高 | 4 | 4 | 3 | 4 |

データ分析基盤+コールセンターの推奨構成:

- **基盤 (Databricks/SAP)**: RBAC を骨格 (ロール = 部署 × 業務役割)
- **PII/録音アクセス**: ABAC を上乗せ (時間帯 / IP / 案件アサイン)
- **「自分の顧客のみ」**: ReBAC を追加 (担当グラフを Unity Catalog Row Filter に流す)
- **横断ポリシー監査**: OPA で集中管理 (Rego ポリシーを Git 管理)

### ステップ 3: ポリシー定義例

**RBAC 例 (Databricks Unity Catalog)**:

```sql
-- ロール定義
CREATE ROLE cs_operator;
CREATE ROLE cs_supervisor;
CREATE ROLE marketing_analyst;

-- 権限付与
GRANT SELECT ON SCHEMA gold.kpi_dashboard TO marketing_analyst;
GRANT SELECT ON TABLE silver.transcripts TO cs_supervisor;
-- オペレータは個別レコードを直接見ない
GRANT EXECUTE ON FUNCTION gold.search_anonymized TO cs_operator;
```

**ABAC 例 (Row Filter + 環境属性)**:

```sql
CREATE OR REPLACE FUNCTION pii_access_filter(record_region STRING)
  RETURN
    CASE
      WHEN current_user_attribute('department') = 'compliance' THEN TRUE
      WHEN current_user_attribute('region') = record_region
        AND hour(current_timestamp()) BETWEEN 8 AND 20
        AND current_user_attribute('mfa_completed') = 'true'
      THEN TRUE
      ELSE FALSE
    END;

ALTER TABLE silver.transcripts SET ROW FILTER pii_access_filter ON (region);
```

**ReBAC 例 (担当グラフ)**:

```sql
-- 担当関係テーブル
CREATE TABLE rebac.assignments (
  user_id STRING, customer_id STRING, valid_from DATE, valid_to DATE
);

CREATE OR REPLACE FUNCTION assignment_filter(customer_id STRING)
  RETURN EXISTS (
    SELECT 1 FROM rebac.assignments a
    WHERE a.user_id = current_user()
      AND a.customer_id = customer_id
      AND current_date() BETWEEN a.valid_from AND a.valid_to
  );
```

**OPA Rego 例 (横断ポリシー)**:

```rego
package vocplatform.authz

default allow = false

# LLM プロンプトに PII を含めるのはコンプライアンス承認済みのみ
allow {
  input.action == "llm_invoke"
  input.resource.contains_pii == true
  input.subject.role == "compliance_officer"
  input.context.purpose_of_use in {"audit", "regulatory_response"}
}

# 録音ダウンロードは SV 以上 + 二要素 + 業務時間内
allow {
  input.action == "download_recording"
  input.subject.role in {"cs_supervisor", "compliance_officer"}
  input.subject.mfa_completed == true
  input.context.hour >= 8
  input.context.hour < 20
}
```

### ステップ 4: 統合方針 (Microsoft Entra ID + Genesys + Databricks + SAP)

```
┌──────────────────────────────────────────────┐
│ Microsoft Entra ID (中央 IdP / Conditional Access)│
│  - グループ: cs_operator, cs_supervisor, mkt_analyst │
│  - 属性: region, department, mfa_state           │
└────────────┬─────────────────────────────────┘
             │ SAML / OIDC + SCIM
   ┌─────────┼─────────────┬───────────────┐
   ▼         ▼             ▼               ▼
┌────────┐ ┌────────────┐ ┌─────────────┐ ┌─────────┐
│Genesys │ │Databricks  │ │SAP Authorz  │ │ OPA     │
│ Roles  │ │Unity Cat.  │ │(PFCG/Fiori) │ │ (横断)  │
└────────┘ └────────────┘ └─────────────┘ └─────────┘
   │         │             │               │
   └─────────┴─────────────┴───────────────┘
             ▼
   集中監査ログ (Microsoft Sentinel / Purview)
```

統合のキーポイント:

1. **アイデンティティの一元化**: Microsoft Entra ID をマスタにし、Genesys/Databricks/SAP には SCIM 同期。ローカル ID 作成は禁止。
2. **属性の正規化**: `department` `region` `clearance_level` などを Entra ID 側で正規化定義し、各システムは同じキー名で参照する。
3. **権限粒度の役割分担**:
   - Entra ID: 大粒度 (どのシステムに入れるか)
   - 各サービスのネイティブ機能: 中粒度 (テーブル/トランザクション)
   - OPA: 横断粒度 (LLM 利用可否、PII エクスポート可否など)
4. **ポリシー as Code**: Rego/SQL/SAP ロール定義を全て Git 管理し、レビュー必須化。
5. **監査ログ集約**: 全システムの認可判定ログを Sentinel/Purview に集約し、相関分析を可能にする。

### ステップ 5: 実装ライブラリ・サービス候補

| 用途 | 候補 | 備考 |
|------|------|------|
| ポリシーエンジン | Open Policy Agent (OPA) | OSS、Rego言語、CNCF Graduated |
| OPA マネージドサービス | Styra DAS / Permit.io | UI・分散デプロイ・監査が組み込み |
| データ層認可 | Databricks Unity Catalog Row/Column Filter | SQL 関数で属性ベース判定可能 |
| データ層認可 (代替) | Immuta / Privacera | 大規模ガバナンス、外部DBにも適用可 |
| ReBAC | Google Zanzibar 系 (SpiceDB / Authzed) | グラフ関係を高速判定 |
| Microsoft 純正 | Microsoft Entra ID Conditional Access + Continuous Access Evaluation | リスクベース動的アクセス制御 |

### ステップ 6: テスト戦略

ポリシーは「書いた通りに動く」だけでは不十分。**期待されるアクセス可否のテストケース**を網羅する:

| テスト種別 | 実装 | 例 |
|-----------|------|----|
| 単体テスト | OPA `opa test` / Rego テスト | 「営業時間外 → deny」 |
| 統合テスト | Synthetic ユーザでの End-to-End | ダミー SV ユーザで録音DL試行 |
| ネガティブテスト | 権限超過試行を CI で実行 | オペレータが直接 SQL で transcripts を SELECT |
| 退職シナリオ | SCIM 同期失敗時の挙動確認 | Entra ID で disable → 各システム反映遅延の許容時間 |
| カオス | 認可サービスダウン時のフェイルセーフ | OPA 不通時は deny default |

CI パイプラインで毎コミット OPA テスト + ポリシー差分レビューを必須化する。

### ステップ 7: ADR テンプレート (`docs/adr/ADR-xxx-authorization.md`)

```markdown
# ADR-xxx 認可モデル選定

## ステータス
Accepted (YYYY-MM-DD) / Reviewer: <CISO 名>

## コンテキスト
- 対象システム: <データ分析基盤 / VoC 基盤 等>
- 主体・資源・操作の規模: <ユーザ数 / テーブル数 / 操作種別数>
- 既存 IdP: Microsoft Entra ID
- 法規制: 個人情報保護法、業界ガイドライン

## 決定
- **基盤層**: RBAC (Databricks Unity Catalog ロール)
- **PII 層**: ABAC (Row Filter + 環境属性)
- **顧客担当**: ReBAC (担当グラフ)
- **横断ポリシー**: OPA (Rego, Git管理)

## 検討した代替案
1. RBAC のみ → 動的条件不足のため却下
2. ABAC 全面 → ポリシー作成負荷過大、運用破綻リスク
3. 自社内製エンジン → 監査要件・OSS エコシステムから OPA 優位

## 帰結
- ポジティブ: ポリシー as Code, 監査追跡, 段階的拡張
- ネガティブ: Rego 学習コスト、4 系統の同期運用
- 中立: 初期構築 1.5 人月

## 関連
- 品質ゲート: SEC-AUTHZ-DESIGN-001
- 関連 ADR: 暗号化 KMS 設計, ゼロトラスト方針
- 関連リスク: R-AUTH-001 (権限昇格), R-AUTH-002 (退職者の権限残存)
```

## 出力フォーマット

1. **認可要件マトリクス**: 主体 × 資源 × 操作の表 (Excel または Markdown)
2. **モデル比較評価表**: 評価軸×モデルの加重スコア
3. **ポリシー定義 (PoC レベル)**: SQL / Rego / Entra ID Conditional Access JSON
4. **統合シーケンス図**: SCIM 同期、認可判定、監査ログのフロー
5. **テスト計画書**: ユーザペルソナ × 期待結果のマトリクス
6. **ADR**: 上記テンプレートに従い 1 本

## データ分析基盤・コールセンター向け実例ノート

- **Genesys ロール**: コールセンタ ASP は `Agent` `Supervisor` `Quality Manager` 等の組み込みロールを持つが、外部のデータ基盤 (Databricks) と命名規則が一致しない。Entra ID で `cs_operator` `cs_supervisor` のグループ名を統一し、両システムへ SCIM 配信する。
- **Databricks Unity Catalog**: スキーマ階層 `bronze/silver/gold` に対し、`bronze` は ETL サービスプリンシパルのみアクセス可、人間ユーザは `gold` 限定が原則。`silver.transcripts` は ABAC Row Filter で「自部署が担当した案件のみ」に絞る。
- **SAP Authorization**: SAP は権限オブジェクト (Authorization Object) ベースで RBAC が深い。Entra ID グループ ↔ PFCG ロールのマッピング表を CSV 管理し、変更は四半期レビュー。
- **録音 5-10 年保管**: 古い録音ほど業務担当が変わっているため ReBAC の担当グラフから外れる。**「現担当 OR コンプライアンス OR 法務」** という OR 条件をポリシーで明示しないと業務停止する。
- **LLM 利用案件**: LLM API へのプロンプト送信は OPA で「PII を含むかどうか」を判定し、含む場合は同意フラグと purpose_of_use の両方を必須にする。

## STRIDE との関係

- 脅威モデリング (STRIDE) で「Spoofing / Elevation of privilege」と判定された脅威に対し、**本スキルが具体的な認可方式で緩和策を提示する**。
- STRIDE は脅威の網羅性、本スキルは緩和策の具体化と一貫性が役割分担。
- リスクレジスタには `linked_threat_id: TM-S-001` のように脅威 ID を、`linked_authz_decision: ADR-xxx` で本 ADR を相互参照する。

## pm-blueprint 連携

- **Layer 5 既存サブスキル**:
  - 脅威モデリング.md: STRIDE で洗い出した「権限昇格」「なりすまし」脅威を本スキルの ABAC/ReBAC で緩和。
  - リスクレジスタ.md: 認可関連リスク (R-AUTH-xxx) を本スキルの ADR ID と紐付けて登録。
- **Layer 7 法務・規制**: 個人情報保護法の「目的外利用禁止」を OPA Rego で `purpose_of_use` 必須化として実装。
- **Layer 8 LLM ガバナンス**: LLM プロンプトに PII を含める判断を OPA で集中管理し、監査可能化。
- **Layer 9 運用**: 退職時の SCIM 同期、四半期権限レビュー、ポリシー変更レビューを運用設計と連動。
- **品質ゲート対応**:
  - SEC-AUTHZ-DESIGN-001 (本スキル直接対応)
  - SEC-ZEROTRUST-001 (ゼロトラスト方針.md と連携、認可は ZTA の実装手段)
  - LEGAL-THREATMODEL-001 (LINDDUN 観点でも認可粒度がプライバシー保護に直結)
