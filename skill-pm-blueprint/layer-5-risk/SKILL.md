# Layer 5: リスク洗い出し

## このレイヤーの目的

プロジェクト開始前に、定性・定量両面でリスクを洗い出し、各リスクに**Kill基準**(撤退条件)を設定する。青写真段階で見落としたリスクは後続フェーズでは発見困難なため、本レイヤーは pm-blueprint の中で最も重要な位置づけ。

## 推奨実行順序

1. **`事前検死.md`** - 定性的な失敗シナリオ生成 (Gary Klein手法)
2. **`thinking-frameworks/反転思考.md`** - 「どうすれば失敗するか」からの設計
3. **`リスクレジスタ.md`** - 確率×影響で定量化
4. **`脅威モデリング.md`** - セキュリティ観点での STRIDE 分析
5. **`thinking-frameworks/可逆性分析.md`** - Type 1/Type 2決定の分類
6. **`thinking-frameworks/二次効果分析.md`** - 副作用の予測
7. **`thinking-frameworks/ケプナートリゴー.md`** - 高リスク決定の構造化分析

最後に、**すべてのリスクに Kill基準を設定**する。Kill基準は「この閾値を超えたら撤退またはピボットする」という定量/定性条件。

## 入力 (前レイヤーから受け取るもの)

- Layer 2 の前提リスト (特に Viability と Feasibility)
- Layer 2 のステークホルダーマップ
- Layer 3 の ADR (特に可逆性の低い決定)
- Layer 4 の NFR (特に可用性・セキュリティ要件)

## 出力 (次レイヤーへ渡すもの)

- **リスクレジスタ (YAML)** - `templates/リスクレジスタ.yml` 形式
- **Kill基準リスト** - リスクIDと撤退条件の対応表
- **脅威モデル** - STRIDE形式のセキュリティリスクマップ
- **代替案の可逆性分類表** - ADRごとの Type 1/Type 2 判定
- **警告サイン辞書** - 各リスクの早期検知指標

これらは Layer 1 (経営判断)、Layer 6 (WBS) の入力となる。

## 年間1億円規模での軽量化方針

- 事前検死は 60〜90 分のセッション 1 回で実施
- リスクレジスタは Top 15〜25 に絞る (全数列挙しない)
- 脅威モデリングは「信頼境界」と「PII取扱経路」の 2 観点に絞る
- 思考フレームワーク 4 種は全て適用せず、プロジェクト特性に応じて 2〜3 種を選ぶ

## データ分析基盤での典型リスクカテゴリ

| カテゴリ | 代表例 |
|---------|-------|
| 技術 | データソース仕様変更、ベンダー API クォータ制限、LLM モデル変更 |
| 人 | キーパーソン離脱、スキル不足、ユーザー習熟度不足 |
| 外部 | データ提供元の業務停止、ベンダー SaaS 障害 |
| 法的 | 個人情報保護法改正、業種別規制 (金融・医療) |
| 財務 | クラウドコスト超過、追加要件による予算超過 |
| スケジュール | 矢羽①業務要件定義の遅延、矢羽⑦UAT の不合格 |
