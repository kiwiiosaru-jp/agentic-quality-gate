# リスクレジスタ

## 概要

事前検死、ディスカバリー、脅威モデリング、思考フレームワークで洗い出されたリスクを**単一のYAMLファイル**に集約し、プロジェクト期間中を通じて更新する。

リスクレジスタの目的:

1. **可視化** - 全リスクを一箇所で把握
2. **定量化** - 確率×影響で優先順位付け
3. **責任の明確化** - 各リスクにオーナーを割当
4. **撤退判断の客観化** - Kill基準で「なんとなくヤバい」を排除

## 推奨フロー

```
事前検死 → 候補リスト
  ↓
脅威モデリング → セキュリティリスク追加
  ↓
ディスカバリー → 前提リスク追加
  ↓
リスクレジスタに統合 → スコアリング → Kill基準設定
  ↓
月次レビュー (またはフェーズゲート毎)
```

## 確率 × 影響マトリクス

### 確率スケール

| レベル | 定義 | 目安 |
|-------|------|------|
| 低 | 発生確率 10% 未満 | 過去に類似事例少ない |
| 中 | 10% 〜 50% | 過去に類似事例あり |
| 高 | 50% 超 | 既に兆候が見えている |

### 影響スケール

| レベル | 定義 |
|-------|------|
| 軽微 | 工期1〜2週間遅延、予算 5% 超過以内、対応可能 |
| 重大 | 工期1〜2ヶ月遅延、予算 5〜20% 超過、スコープ縮小必要 |
| 致命的 | プロジェクト継続困難、予算 20% 超過、撤退/ピボット検討 |

### 優先度マトリクス

|           | 影響:軽微 | 影響:重大 | 影響:致命的 |
|-----------|----------|----------|------------|
| 確率:低   | L (監視) | M (準備) | H (予防)   |
| 確率:中   | M (準備) | H (予防) | H (予防)   |
| 確率:高   | H (予防) | H (予防) | **K** (Kill基準) |

- **L**: 四半期レビューで再評価
- **M**: 予防策を準備、トリガー監視
- **H**: 予防策を即時実施、オーナー明確化
- **K**: Kill基準を定義し、経営判断へのエスカレーション経路を用意

## カテゴリ

```
technical  - 技術 (アーキ, データ, パフォーマンス, セキュリティ)
people     - 人 (離脱, スキル, チームビルド)
external   - 外部 (ベンダー, 取引先, 市場)
legal      - 法的 (規制, 契約, 知財, 個人情報)
financial  - 財務 (コスト, ROI, 為替)
schedule   - スケジュール (依存, 遅延, クリティカルパス)
```

## Kill基準の定義方法

Kill基準は以下の 4 パターンのいずれかで書く:

### パターン A: 定量イベント型

- 「データ連携先 API の障害が月 3 回以上発生したら撤退」
- 「月次クラウドコストが予算の 130% を超えたらスコープ縮小」

### パターン B: 期限+達成条件型

- 「矢羽①業務要件定義が開始 3ヶ月以内に完了しなければ、仕切り直し判断」
- 「PoC 1ヶ月以内に主要 3 ユースケースが動作しなければ、アーキ再検討」

### パターン C: 定性だが誰でも判定可能型

- 「業務部門責任者が『使えない』と公式に表明したら中断」
- 「法務が PII 取扱について NG 判定を出したら即時停止」

### パターン D: 兆候複数型 (AND/OR)

- 「ユーザー採用率 50% 未満 AND クエリ失敗率 20% 以上 が 2ヶ月連続 → ピボット」

## YAMLテンプレート

詳細は `templates/リスクレジスタ.yml` 参照。基本構造:

```yaml
risks:
  - id: R001
    title: "データソース側の仕様変更"
    category: technical
    probability: medium
    impact: severe
    description: |
      POS システムのスキーマが業務側の都合で変更される可能性。
      事前通知が十分に行われないケースがある。
    triggers:
      - ソース系運用チームからのスキーマ変更連絡
      - ETL パイプラインの NULL 率上昇
    mitigations:
      - スキーマ変更検知のモニタリング導入
      - ソース系運用チームとの月次同期会議
      - Bronze 層でスキーマ変化を吸収する設計
    owner: "データエンジニアリード"
    status: open
    kill_criterion: |
      スキーマ変更が月 3 回以上発生 かつ
      夜間バッチ成功率が 90% を下回った場合、
      データソース再選定を検討する
    warning_signs:
      - Bronze 層取り込みエラー週次集計で 5% 超
      - ソース系からの事前通知リードタイムが 2 週間未満
    linked_use_cases: [UC-003, UC-007]
    linked_adrs: [ADR-0002]
    linked_wbs_l4_tasks: [WBS-3.1.2, WBS-6.2.1]
```

## モニタリングと更新

### 月次レビュー (推奨頻度)

- 全 open リスクのステータス確認
- 確率/影響スコアの再評価 (前月比)
- 警告サインが出ていないか確認
- 新規リスクの追加 (特に前月発生した問題からの類推)
- Kill基準との距離 (どれくらい近づいているか) を記録

### フェーズゲート時の特別レビュー

各矢羽完了時には全リスクを棚卸し:

- クローズできるリスクはクローズ (リスクが消滅または受容済み)
- 新たなフェーズで顕在化しうるリスクを追加
- 次フェーズの WBS にリスク対応タスクを反映
