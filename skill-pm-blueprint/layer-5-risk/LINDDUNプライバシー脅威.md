# LINDDUN プライバシー脅威

> **品質ゲート対応**: LEGAL-THREATMODEL-001 (脅威モデリングが STRIDE 中心でセキュリティ寄り、プライバシー脅威の体系的洗い出しが欠落)

## 概要

STRIDE は「機密性・完全性・可用性」を中心とした**セキュリティ脅威**の枠組みであり、「個人がトラッキングされる」「同意していない第二目的に使われる」といった**プライバシー脅威**を直接モデル化できない。LINDDUN は KU Leuven が提唱したプライバシー脅威モデルで、STRIDE と対照的な 7 観点でプライバシーリスクを洗い出す。

LINDDUN 7 観点:

| 記号 | 脅威 | 対応する性質 |
|-----|------|-------------|
| **L** | Linkability (リンク可能性) | Unlinkability (リンク不能性) |
| **I** | Identifiability (識別可能性) | Anonymity / Pseudonymity (匿名性/仮名性) |
| **N** | Non-repudiation (否認不能) | Plausible deniability (もっともらしい否認可能性) |
| **D** | Detectability (検出可能性) | Undetectability (検出不能性) |
| **D** | Disclosure of information (情報開示) | Confidentiality (機密性) |
| **U** | Unawareness (本人未認識) | Awareness (本人の認識・透明性) |
| **N** | Non-compliance (法令不適合) | Compliance (法令適合) |

> 7番目を NC (Non-compliance) とする版と、Policy and Consent non-compliance とする版があるが、本スキルでは LINDDUN GO の最新整理に従う。

## STRIDE と LINDDUN の使い分け

| 観点 | STRIDE | LINDDUN |
|-----|--------|---------|
| 中心関心事 | セキュリティ (CIA + 認証認可+否認防止) | プライバシー (個人情報主体の権利) |
| 主体 | システム/データ/管理者 | データ主体 (本人) |
| 否認 | "否認 (Repudiation) を防止する" | "否認 (Deniability) を可能にする" ← 真逆 |
| 適用順 | 先に STRIDE で土台、次に LINDDUN で補完 | STRIDE 後 |
| 出力 | 緩和策 = 認証/暗号化/監査 | 緩和策 = 同意/最小化/匿名化/透明化 |

**重要な逆転**: STRIDE の R (Repudiation 防止) と LINDDUN の N (Non-repudiation = データ主体が「私はこの行為をしていない」と主張できなくなる脅威) は概念が真逆。LINDDUN ではむしろ**データ主体に否認の余地を残す**ことが緩和策となる。

## いつ使うか

- STRIDE 脅威モデリングが完了した直後 (補完として実施)
- 個人情報を取り扱う Layer 3 アーキテクチャ確定後
- 録音・文字起こし・チャットログなど個人の発話データを保管する案件
- LLM (生成 AI) を介してユーザ入力を処理するシステム
- データ二次利用 (マーケ分析・モデル学習) を検討する場合
- 法務 (Layer 7) からプライバシー影響評価 (DPIA) を求められたとき
- 個人情報保護法・GDPR・CCPA 対応の DPIA 実施前

## 手順

### ステップ 1: 個人データの分類

LINDDUN の前提として、保護対象を明確化:

| 分類 | 例 | 機密度 |
|-----|-----|-------|
| **直接識別子** | 氏名, マイナンバー, 顔, 声紋 | 最高 |
| **準識別子** | 生年月日, 郵便番号, 性別 (組合せで識別可) | 高 |
| **属性データ** | 購買履歴, 通話履歴, 検索履歴 | 中〜高 |
| **行動・推論データ** | 嗜好, 信用度, 感情推定, AI スコア | 高 (差別リスク) |
| **メタデータ** | 通信時刻, 位置, デバイス ID | 中 |

特に**準識別子の組合せ**による再識別 (k-匿名性違反) と**行動データから推論される属性** (例: 通話パターンから精神疾患を推定) は LINDDUN で重点的に扱う。

### ステップ 2: データフロー図 (DFD) の流用

STRIDE で作成した DFD を流用し、各データ・各ストア・各フローに LINDDUN を適用する。新たに「**プライバシー視点での観察ポイント**」を加える:

- データ主体 (本人) はこの処理を認識しているか?
- 同意の取得経路はどこか?
- このデータは他のデータと結合可能か?
- 削除要求があった時、削除可能か?
- 本人の閲覧 (開示請求) に応えられるか?

### ステップ 3: LINDDUN 適用 (各観点の脅威例)

#### L: Linkability (リンク可能性)

複数のレコードが同一人物のものだと特定できる脅威。

| シナリオ | コールセンター/データ基盤の例 |
|---------|---------------------------|
| 別システムの ID が結合されてしまう | Genesys 顧客 ID + 会員 DB ID + Web 行動 ID が JOIN され、個別行動が紐付く |
| 通話メタデータで人物が再識別 | 通話時刻+電話番号下4桁から特定の家庭に絞り込み |
| ベクトル埋め込み経由でリンク | LLM 埋め込みベクトルから類似発話者をクラスタリング → 同一人物推定 |

**緩和策**:
- システム横断で異なる Pseudonymous ID を使う (Salt 違いハッシュ)
- 結合分析専用ビューを設け、生 ID は隔離
- ベクトル DB は次元削減・差分プライバシー (DP) を適用

#### I: Identifiability (識別可能性)

匿名化したつもりのデータから個人を再識別できてしまう脅威。

| シナリオ | 例 |
|---------|-----|
| 仮名化が外的情報で破られる | "顧客 ID 12345" の購買履歴 + SNS 公開情報で氏名特定 |
| 声紋から個人特定 | 録音を匿名化して保管したつもりが、声紋照合で人物特定 |
| 文体・口癖から個人特定 | LLM 出力ログから書き手の文体パターンが学習されて識別 |

**緩和策**:
- k-匿名性, l-多様性, t-近接性の数値目標設定
- 声紋は録音保管時にボイスチェンジ or 音声特徴量削除
- 文章は要約・抽象化して原文を残さない

#### N: Non-repudiation (否認不能 / 不利に働く文脈)

データ主体が「この行為はしていない」と否認する正当な権利を奪われる脅威。STRIDE の R とは目的が逆。

| シナリオ | 例 |
|---------|-----|
| 通話内容が永久保管され、後に「言った言わない」で本人不利 | 5-10 年前の発言を巻き戻しに使われる |
| デジタル署名で過剰な否認阻止 | 軽微な操作にもタイムスタンプと署名で「本人がやった」を強制 |

**緩和策**:
- 法的・業務的に必要な範囲のみ Non-repudiation を適用
- 通話保管期間の上限を業務正当性ベースで決定 (恣意的な永久保管を避ける)
- 否認可能な処理 (例: 匿名アンケート) では署名を付与しない

#### D: Detectability (検出可能性)

「あるデータが存在すること自体」が漏れる脅威。

| シナリオ | 例 |
|---------|-----|
| 「特定顧客のクレーム件数」を返す API の存在で、その人がクレームしたかが推測される | API 応答時間の差異 (件数 0 vs 1 で異なる) |
| 個別レコードの存在確認攻撃 | "顧客 ID 12345 のデータはありますか?" → エラー差異で存在確認 |

**緩和策**:
- 件数返却 API は「存在しない場合も時間差を一定化」
- レコード存在チェックを禁止し、検索結果として 0/1 件を均質に扱う
- データの「ある/ない」を返す API は最小化

#### D: Disclosure of information (情報開示)

権限のない者にデータが漏洩する脅威 (STRIDE の I と重なるが、データ主体視点)。

| シナリオ | 例 |
|---------|-----|
| 同部署の同僚に閲覧される | 担当でない案件の通話を SV が聞ける |
| LLM 経由の漏洩 | プロンプト履歴がモデル学習に使われ、別ユーザに復元される |
| エクスポート CSV からの漏洩 | 分析者がローカルに保存し、退職時に持ち出し |

**緩和策**:
- ABAC/ReBAC で「担当案件のみ」アクセス
- LLM はオプトアウト or 専用テナント (学習に使われない)
- DLP + 透かし入りエクスポート + 退職プロセスでの回収

#### U: Unawareness (本人未認識)

本人が自分のデータの取扱いを認識・コントロールできない脅威。

| シナリオ | 例 |
|---------|-----|
| 「録音されているとは知らなかった」 | 通話冒頭のアナウンス漏れ |
| AI による感情分析が告知されない | 通話品質改善のためと告知だが、AI が感情スコアを計算 |
| データ二次利用が告知されない | クレーム録音をマーケティング機械学習に流用 |

**緩和策**:
- 通話冒頭の録音通知の徹底 (再生確認)
- AI 利用・推論の明示 (利用規約・コールセンタ案内)
- 二次利用には別途同意取得 (オプトイン)
- 透明性レポートの公開 (年次)

#### N: Non-compliance (法令不適合)

法律・規制・契約に違反する処理の脅威。

| シナリオ | 例 |
|---------|-----|
| 個人情報保護法の利用目的を超過 | 「サービス提供」目的で取得したデータをマーケ用 ML に投入 |
| 越境移転で十分性認定なし国への送信 | LLM API のリージョン制御漏れで EU データが米国へ |
| 保存期間超過 | 業務上不要な録音を 10 年超保管 |
| 開示請求への未対応 | 30 日以内に応答できない |

**緩和策**:
- 利用目的を OPA で技術的強制 (`purpose_of_use` 必須化、認可方式選定.md と連動)
- データ・レジデンシー要件をクラウドリージョン選定で満たす
- 保管期間の自動削除ジョブ + 例外管理
- 開示請求対応 SLA 設計 (15 日以内推奨)

### ステップ 4: 録音 5-10 年保管案件の重点チェック

長期保管案件は LINDDUN リスクが極端に高まる。専用チェックリスト:

- [ ] **保管期間の根拠**: 法令・契約・業務それぞれで根拠文書を保有しているか
- [ ] **段階的アクセス制限**: 1 年経過後の録音は SV のみ、3 年経過後はコンプライアンスのみ等の段階制限
- [ ] **音声特徴量の最小化**: 業務目的が薄れた録音から声紋復元可能性を下げる (圧縮、特徴量抽出後の原本削除)
- [ ] **メタデータの分離**: 通話相手電話番号・時刻・担当者を別ストアにし結合キーを暗号化
- [ ] **削除リクエスト処理**: 本人申し出時の物理削除フロー + 削除証跡
- [ ] **アクセス頻度監視**: 古い録音への突発的アクセスをアラート (内部不正検出)
- [ ] **オフライン保管**: 業務頻度が低い古い録音は WORM/Cold ストレージで Read-only 化
- [ ] **二次利用同意の追跡**: 保管期間中に同意撤回があった場合の処理経路
- [ ] **再識別リスク評価**: 5 年後の AI 技術 (声紋照合精度向上) を前提にした評価
- [ ] **本人開示請求対応**: 録音の文字起こしまで提供するかの方針

### ステップ 5: LLM 利用案件の重点チェック

LLM 経由の処理は LINDDUN 全観点で新たなリスクを発生させる:

| 観点 | LLM 特有のリスク | 緩和策 |
|------|----------------|--------|
| L | プロンプト + 応答が組合さって個人プロファイル化 | プロンプトに ID を含めない、セッション単位で破棄 |
| I | 文体・固有表現で個人特定 | PII レダクション + 一般化 |
| N | LLM 応答が永久ログ化 → 永久に「言ったこと」になる | 応答ログの保管期間設定、否認可能な要約 |
| D | 「ユーザがこの質問をした事実」が漏洩 | プロンプトの暗号化、専用テナント |
| D | プロンプトが学習データに混入し他者に漏洩 | オプトアウト、ベンダー DPA、Microsoft Foundry の "Your data is your data" を確認 |
| U | LLM が裏で動いていることを本人が知らない | 利用案内、AI 介在の明示 |
| N | EU データ → 米国 LLM の越境移転 | リージョン制御、Customer-managed Endpoint |

**LLM プロンプトの PII 除去フロー**:

```
[ユーザ入力 / 通話文字起こし]
        ▼
[PII 検出 (NER + 正規表現)]
   ├─ 氏名 → "[NAME]"
   ├─ 電話番号 → "[PHONE]"
   ├─ 住所 → "[ADDRESS]"
   ├─ メール → "[EMAIL]"
   └─ 口座番号等 → "[ACCOUNT]"
        ▼
[LLM 投入]
        ▼
[応答 PII 検査 (再混入対策)]
        ▼
[ログ保管 (PII 除去版のみ、原本は短期)]
```

### ステップ 6: 緩和策の体系化

LINDDUN の緩和策は STRIDE と異なるカテゴリ:

| カテゴリ | 例 |
|---------|-----|
| **データ最小化** | 必要最小限のフィールドのみ取得・保管 |
| **匿名化・仮名化** | k-匿名, 差分プライバシー, トークン化 |
| **同意管理** | OneTrust 等、目的別オプトイン、撤回経路 |
| **透明性** | 利用案内、AI 介在告知、透明性レポート |
| **本人の権利行使** | 開示・訂正・削除・利用停止のセルフサービス |
| **保管期間管理** | 自動削除、段階的アクセス制限 |
| **技術的隔離** | 専用 VNet, リージョン固定, テナント分離 |

### ステップ 7: DPIA (データ保護影響評価) への接続

LINDDUN の出力を DPIA テンプレートに転記:

1. 処理の概要
2. 必要性・比例性 (なぜこのデータが必要か)
3. プライバシー脅威 (LINDDUN 結果を貼付)
4. 緩和策 (LINDDUN 緩和策一覧)
5. 残存リスクと受容判定
6. データ主体への影響度
7. 監督当局・データ保護責任者 (DPO) のレビュー

### ステップ 8: ADR テンプレート (`docs/adr/ADR-xxx-privacy-threat.md`)

```markdown
# ADR-xxx LINDDUN プライバシー脅威モデル

## ステータス
Accepted (YYYY-MM-DD) / Reviewer: <DPO 名>, <法務責任者>

## 対象
- 録音 (5-10 年保管)
- 文字起こし (LLM 投入対象)
- LLM プロンプト・応答ログ

## 主要なプライバシー脅威 (Top 5)
1. P-LINK-001: Genesys ID と会員 DB の結合による行動プロファイル化
2. P-IDENT-002: 録音音声の声紋による再識別 (将来技術想定)
3. P-AWARE-003: AI 感情分析の告知不足
4. P-COMPL-004: LLM API 越境移転リスク
5. P-DISC-005: 同部署 SV による担当外案件閲覧

## 緩和策
- 各 ID に独立 Pseudonymous キー (Salt 違いハッシュ)
- 録音圧縮+特徴量削除
- AI 利用の利用案内強化
- LLM API リージョン固定 + Customer-managed Endpoint
- ABAC/ReBAC で担当案件のみアクセス

## 関連
- 品質ゲート: LEGAL-THREATMODEL-001
- 関連 ADR: 認可方式選定, 暗号化_KMS設計, ゼロトラスト方針
- 関連リスク: R-PRIV-001 〜 R-PRIV-005
```

## 出力フォーマット

1. **個人データ分類表** (直接識別子 / 準識別子 / 属性 / 推論)
2. **LINDDUN マトリクス** (DFD 要素 × 7観点 × 脅威シナリオ × 緩和策)
3. **録音長期保管チェックリスト進捗**
4. **LLM 利用案件チェックリスト進捗**
5. **DPIA 文書 (法務向け)**
6. **ADR 1 本**

## STRIDE との連携 (具体)

| ステップ | 実施 |
|---------|------|
| 1 | 脅威モデリング.md (STRIDE) で全フロー × STRIDE 6 観点を実施 |
| 2 | 同 DFD に対し本スキル (LINDDUN) を実施 |
| 3 | リスクレジスタに STRIDE 由来 (TM-S-xxx 等) と LINDDUN 由来 (P-xxx-yyy) を別系列で登録 |
| 4 | 重複する脅威 (例: STRIDE I = LINDDUN D-Disclosure) は相互参照 ID を付与し緩和策を共有 |
| 5 | 真逆な観点 (R vs N) は注意深く扱い、業務目的と本人権利のバランスを ADR で記録 |

## pm-blueprint 連携

- **Layer 5 既存サブスキル**:
  - 脅威モデリング.md (STRIDE): 本スキルが補完。STRIDE はセキュリティ、本スキルはプライバシー。
  - リスクレジスタ.md: P-xxx 系統で LINDDUN 由来リスクを登録、`framework: LINDDUN` フィールドで識別。
  - 認可方式選定.md: LINDDUN の Disclosure 緩和策として ABAC/ReBAC が直接対応。
  - 暗号化_KMS設計.md: LINDDUN の Linkability/Identifiability 緩和策としてフィールド暗号化・トークン化。
  - ゼロトラスト方針.md: LINDDUN の Disclosure/Detectability 緩和策として最小権限・JIT 昇格。
- **Layer 7 法務・規制**: 個人情報保護法 / GDPR / CCPA / 業界規制を LINDDUN の Non-compliance で担保。DPIA に直接連携。
- **Layer 8 LLM ガバナンス**: LLM 特有の LINDDUN 脅威を集約し、Foundry / OpenAI / Bedrock 利用ポリシーに反映。
- **Layer 9 運用**: 開示請求対応 SLA、削除フロー、保管期間管理、同意撤回処理を運用設計に組み込む。
- **品質ゲート対応**:
  - LEGAL-THREATMODEL-001 (本スキル直接対応)
  - SEC-AUTHZ-DESIGN-001 (Disclosure 緩和策で連携)
  - SEC-CRYPTO-001 (Linkability/Identifiability 緩和策で連携)
  - SEC-ZEROTRUST-001 (Disclosure/Detectability 緩和策で連携)
