# ケプナートリゴー法 (Kepner-Tregoe Method)

## 概要

Charles Kepner と Benjamin Tregoe が 1965 年に体系化した構造化思考法。
4 つのプロセスを組み合わせて、**複雑な状況を分解し、根拠をもって意思決定**する。

4 つのプロセス:

1. **Situation Appraisal (状況把握)** - 何が起きているかを整理
2. **Problem Analysis (問題分析)** - 原因を特定
3. **Decision Analysis (決定分析)** - 選択肢を評価して決定
4. **Potential Problem Analysis (潜在的問題分析)** - 未来の問題を予防

各プロセスは独立して使えるが、問題発生時は 1→2→3→4 の順で使うと効果的。

## いつ使うか

- リスクが顕在化して原因特定が必要なとき
- ADR でアーキを決める前 (Decision Analysis)
- キックオフ前の全体整理 (Situation Appraisal)
- リスクレジスタ作成時 (Potential Problem Analysis)
- 事前検死で出た失敗シナリオの深掘り

年間 1 億円規模では、毎回 4 プロセス全部はやらず、その時点で必要な 1〜2 プロセスを選んで適用する。

## プロセス 1: 状況把握 (Situation Appraisal)

### 目的
混乱した状況から関心事・懸念事項を洗い出し、優先順位付けする

### 手順
1. **関心事の列挙** (Concerns): 今気になっていること全てを書き出す
2. **分離** (Separate): 大きな関心事を複数の小さなものに分ける
3. **明確化** (Clarify): 各関心事を「○○が X になっている」の形で明文化
4. **優先順位付け** (Prioritize): 以下 3 観点で評価
   - 緊急度 (Timing): いつまでに対処必要か
   - 影響度 (Impact): 放置したときの被害
   - 傾向 (Trend): 悪化しているか安定しているか
5. **次プロセスの選択**: 各関心事に対して「問題分析 / 決定分析 / 潜在的問題分析」のどれが必要かを決定

### データ分析基盤での例

キックオフ 2ヶ月前の状況把握:

| 関心事 | 明確化 | 緊急 | 影響 | 傾向 | 次プロセス |
|-------|-------|------|------|------|-----------|
| データソースオーナーと連絡取れない | POS 側担当者が産休でバックアップ不在 | 高 | 高 | 悪化 | 問題分析 |
| dbt 経験者が採用できない | 市場で供給少なく、年収水準が予算と合わない | 高 | 高 | 横ばい | 決定分析 |
| 矢羽⑥運用体制の初期設計がまだ | 運用部門との調整が未着手 | 中 | 高 | 悪化 | 決定分析 |
| LLM API の価格改定リスク | 下期にベンダーが値上げ示唆 | 低 | 中 | 横ばい | 潜在的問題分析 |

## プロセス 2: 問題分析 (Problem Analysis)

### 目的
「本来あるべき姿」と「現状」のギャップ (= 問題) の根本原因を特定する

### 手順
1. **問題の明文化**: 「○○ が本来 X であるべきなのに Y になっている」
2. **IS / IS NOT マトリクス**: 問題が起きている範囲を特定
   - 何が (What): 起きている / 起きていない
   - どこで (Where): 起きている / 起きていない
   - いつ (When): 起きている / 起きていない
   - どのくらい (Extent): 起きている / 起きていない
3. **特徴の抽出**: IS と IS NOT の違いから仮説を立てる
4. **可能な原因の列挙**: なぜ IS は起きていて IS NOT は起きていないのか
5. **最も可能性の高い原因の特定**: テスト可能な形式で
6. **検証**: 実際にテストして原因を確定

### 簡易版: 5 Whys との関係

時間がないときは 5 Whys (「なぜ?」を 5 回繰り返す) で代用可。ただし IS / IS NOT を併用すると見落としが減る。

## プロセス 3: 決定分析 (Decision Analysis)

### 目的
複数の選択肢を構造化して評価し、最適案を選ぶ

### 手順
1. **決定目的の明文化**: 「何を決めようとしているか」を 1 文で
2. **目標の設定**:
   - MUST (必須): 満たさなければ却下する絶対条件
   - WANT (できれば): 重み付きで評価する相対条件
3. **選択肢の列挙**: 最低 3 つ
4. **MUST スクリーニング**: MUST を 1 つでも満たさない選択肢は即除外
5. **WANT 評価**: 各 WANT に重み (1〜10) を付け、選択肢ごとに評点 (1〜10) を乗じて合計
6. **リスク評価**: 上位候補に対して、採用した場合の副作用・リスクを評価
7. **最終決定**: WANT スコアとリスクを統合して選定

### データ分析基盤での例: DWH 選定

**決定目的**: メインのデータウェアハウスプラットフォームを決定する

**MUST**:
- [ ] AWS 上で動作する
- [ ] SOC2 認証取得済み
- [ ] 日本語サポート窓口あり

**WANT** (重み):
- クエリ性能 TPC-DS 100GB スケール (9)
- 月間コスト 予算内 (8)
- dbt ネイティブサポート (7)
- 社内既存スキルとの親和性 (6)
- LLM / AI 機能 の統合度 (5)

**選択肢評価**:

| 選択肢 | MUST | 性能(9) | コスト(8) | dbt(7) | スキル(6) | AI(5) | 合計 |
|-------|------|---------|----------|--------|-----------|-------|------|
| Snowflake | ○ | 8×9=72 | 6×8=48 | 9×7=63 | 5×6=30 | 7×5=35 | 248 |
| Databricks | ○ | 9×9=81 | 7×8=56 | 8×7=56 | 7×6=42 | 9×5=45 | 280 |
| BigQuery | × (AWS要件で却下) | - | - | - | - | - | - |

**リスク評価**: Databricks 採用時、Notebook中心の開発文化に業務部門が馴染めるか不安 → ハンズオン組み込み

**決定**: Databricks (ADR-0001 として記録)

## プロセス 4: 潜在的問題分析 (Potential Problem Analysis)

### 目的
意思決定後、将来起こりうる問題を予測し予防策を用意する

### 手順
1. **計画の明文化**: 実行計画をステップ化
2. **潜在的問題の列挙**: 各ステップで起こりうる問題をブレスト
3. **確率 × 影響で評価**: 既出のリスクマトリクスと同じ基準
4. **予防策と緊急時対応の設計**:
   - Preventive Action: 起きないようにする
   - Contingent Action: 起きたら即対応する
5. **トリガーの設定**: Contingent Action をいつ発動するかの閾値

### リスクレジスタとの関係

Potential Problem Analysis は**リスクレジスタの材料**。KT法で構造化して出てきたリスクを `リスクレジスタ.md` の形式に転記する。

## 統合フロー例

プロジェクト後半でスケジュール遅延が発生した場合:

```
1. Situation Appraisal
    「矢羽③データ連携Sprintが 3 週遅延している」を関心事として明確化
   ↓
2. Problem Analysis
    IS / IS NOT で「遅延は API 連携部分のみ」「他チームは予定通り」を特定
   ↓
3. Decision Analysis
    選択肢: (A) 期限延長, (B) スコープ縮小, (C) 人員追加 を MUST/WANT で評価
   ↓
4. Potential Problem Analysis
    決定 (B) スコープ縮小が引き起こす副作用 (業務部門の納得感など) を予測し予防
```
