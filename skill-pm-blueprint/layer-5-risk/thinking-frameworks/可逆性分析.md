# 可逆性分析 (Reversibility Analysis)

## 概要

Jeff Bezos が 2015 年 Amazon 株主への年次書簡で提唱した決定分類:

- **Type 1 決定**: 不可逆で重大。片道扉 (one-way door)。慎重に検討すべき
- **Type 2 決定**: 可逆で影響限定的。双方向扉 (two-way door)。素早く試すべき

よくある失敗:

- Type 2 決定を Type 1 として扱い、過剰な検討で時間を浪費
- Type 1 決定を Type 2 と誤認し、軽率に決めて後悔

青写真段階では特に、**どの決定が Type 1 か**を早期に同定し、リソースを集中させる必要がある。

## いつ使うか

- ADR を書く前 (Type 1 であれば ADR を詳細化、代替案の検討を増やす)
- アーキテクチャ決定前 (DWH選定, 認証基盤選定, クラウド選定)
- 契約締結前 (ベンダーロックイン度合いの評価)
- Kill基準設定前 (Type 1 の決定は撤退コストが高いため Kill基準必須)
- 矢羽の並列実行開始前 (並列化により得る柔軟性が Type 2 化につながるか)

## 判定基準

以下の質問に答えて Type 1 か Type 2 かを判定する。

### Type 1 に近い兆候 (多いほど不可逆)

- [ ] 後戻りすると、費用が投資額の 50% 超かかる
- [ ] 後戻りすると、期間が 3ヶ月以上追加で必要
- [ ] 契約上の拘束がある (違約金, 最低利用期間)
- [ ] 顧客/ユーザーへの告知済みで期待形成されている
- [ ] 変更によって依存システムの大幅改修が必要
- [ ] データ構造の根幹に関わる (スキーマ・ID体系など)
- [ ] 組織体制・評価制度の変更を伴う
- [ ] ブランドや信頼に関わる (一度公表すると取り消せない)

### Type 2 に近い兆候 (多いほど可逆)

- [ ] ロールバック手順が数時間以内で完了する
- [ ] A/B テストや限定リリースが可能
- [ ] 契約がスポット・従量課金で切り替えコストが低い
- [ ] 見せ方・UI・色・文言など表層的な変更
- [ ] データは変換で後から調整可能
- [ ] フィーチャーフラグで機能を後から切り替え可能

## データ分析基盤での例示

### Type 1 決定 (不可逆、慎重に)

| 決定 | 理由 |
|-----|------|
| DWH プラットフォーム選定 (Snowflake vs Databricks vs BigQuery) | データ・クエリ・コンピュートが全てロックイン。乗り換えコスト数千万〜 |
| 認証基盤 (IdP) 選定 | 全システム連携に影響、ユーザー体験の根幹 |
| データレイクの ID 体系 (自然キー vs サロゲート) | 全テーブルに波及、後からの変更はマイグレ地獄 |
| 個人情報取扱の方針 (匿名加工 vs 仮名加工 vs 個人情報のまま) | 法的位置づけが変わる、第三者提供の可否に影響 |
| データガバナンスのオーナー組織 | 組織設計・評価制度に組み込まれる |

### Type 2 決定 (可逆、素早く試す)

| 決定 | 理由 |
|-----|------|
| ダッシュボードの色・フォント・グラフ種別 | ユーザー反応を見て変えればよい |
| 個別 dbt モデルのロジック | 次の Sprint で修正可能 |
| テストデータの作り方 | 影響範囲は開発環境内 |
| チケット管理ツールの選定 (Jira vs Linear) | データは ID 付きでエクスポート可能 |
| スプリントの期間 (1 週間 vs 2 週間) | 試して合わなければ変更 |
| LLM プロンプトのテンプレート | バージョン管理して比較可能 |

### グレーゾーン (判定注意)

| 決定 | コメント |
|-----|---------|
| dbt vs 手書き SQL パイプライン | 数ヶ月分のコードが書かれると実質 Type 1 化 |
| BI ツール選定 | 最初は Type 2 に見えるが、ダッシュボード資産が増えると Type 1 化 |
| マイクロサービス vs モノリス | 初期は可逆だが、コード量と運用チーム分離で不可逆化 |

## 実践的ルール

### ルール 1: Type 1 を見抜いたら ADR を書く

Type 1 決定は必ず ADR (`layer-3-architecture/ADR作成.md` 参照) に記録。
- 代替案を最低 3 つ検討
- 選定の根拠を定量化
- 可逆性の低さを明示

### ルール 2: Type 2 は ADR ではなく Sprint チケットで

Type 2 決定をいちいち ADR にすると意思決定が重くなる。Sprint チケットで「こう決めた、ダメなら次で変える」と軽く扱う。

### ルール 3: Type 1 には Kill基準必須

ADR で書いた Type 1 決定には、リスクレジスタで対応する Kill基準を設ける。
例: ADR-0001 で Snowflake を選定 → Kill基準「月次クエリコストが予算 150% を 3ヶ月連続で超えたら Databricks 移行検討を開始」

### ルール 4: 組織能力で判定基準は変わる

同じ「DWH 変更」でも、組織にデータ基盤チームがあれば Type 1 寄り、外部ベンダー依存なら Type 2 になることも (契約解除で済む場合)。

### ルール 5: 「実は Type 2 にできないか」を疑う

Type 1 に見える決定でも、以下で Type 2 化できる場合がある:

- **抽象化**: インターフェース層を挟んで実装を差し替え可能にする
- **分割**: 一括決定を複数の小さな決定に分解
- **段階導入**: 限定的なユースケースから始める (カナリア、パイロット)

例: 「LLM ベンダー選定」を Type 1 と思いきや、プロンプトを抽象化レイヤー経由にすれば Type 2 化できる場合がある。
