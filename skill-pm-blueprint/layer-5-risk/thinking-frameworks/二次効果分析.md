# 二次効果分析 (Second-Order Effects Analysis)

## 概要

あらゆる意思決定には**直接効果の先に副作用**がある。典型的な失敗は「意図した効果のみ」を評価し、副作用を見落とすこと。

二次効果分析は効果を 3 階層に分けて予測する:

- **1 次効果 (First-order)**: 直接的・意図された効果。短期的・可視
- **2 次効果 (Second-order)**: 波及的な副作用。中期的・他部門やユーザーへの影響
- **3 次効果 (Third-order)**: 構造的変化。長期的・組織文化や市場動態への影響

ほとんどの意思決定は 1 次効果で評価されるが、プロジェクトを壊すのは 2 次・3 次効果であることが多い。

## いつ使うか

- 大きなアーキ決定後 (決定分析のリスク評価とセット)
- 組織変更・役割再定義の前
- ツール導入・業務変更の前
- Kill基準検討時 (Kill した後にどんな波及があるか)
- 成功シナリオの検証時 (成功したらどう組織が変わるかを読む)

## 手順

### ステップ 1: 意思決定を明文化

「○○を X から Y に変更する」の形式で書く。

例:

- 「データ分析を Excel から BIツール に変更する」
- 「SQL書きを DWH チームから各部門に移管する (セルフサービス化)」
- 「dbt の所有権を開発チームからデータスチュワードに移す」

### ステップ 2: 1 次効果の列挙 (5〜10分)

直接的・意図された効果をブレスト:

- 「レポート作成時間が半減する」
- 「分析の属人化が減る」
- 「最新データが即時反映される」

### ステップ 3: 2 次効果の予測 (15〜20分)

以下の問いで波及効果を広げる:

- **誰が**新たに恩恵 / 被害を受けるか? (部門, 役職, 業務)
- **何が**新たに必要になるか? (スキル, ツール, 体制)
- **何が**不要になるか? (既存業務, 既存ツール)
- **どのような行動変化**が起きるか? (ユーザー, 管理者, 経営)
- **依存関係**はどう変わるか?
- **インセンティブ構造**はどう変わるか?

### ステップ 4: 3 次効果の予測 (15〜20分)

2 次効果が定着した後の構造変化を想像:

- 組織構造にどう影響するか
- 文化・評価制度にどう影響するか
- 市場・取引先との関係はどう変わるか
- 技術的負債・政治的負債はどう蓄積するか
- 次の大きな意思決定にどう影響するか

### ステップ 5: 望ましくない効果の緩和策設計

2 次・3 次効果の中で望ましくないものに対して緩和策を用意:

- 教育・トレーニング
- 段階的導入
- インセンティブ設計の調整
- コミュニケーション戦略
- 段階的撤回余地の確保

## データ分析基盤での具体例

### 例 1: 「分析を Excel から BIツール に全面移行」

**1 次効果 (意図)**:
- 集計時間の短縮
- データソース統一
- リアルタイム性の向上

**2 次効果**:
- Excel で独自計算をしていた部門が、自部門の定義が失われて混乱
- BIツール学習コストが現場に発生し、一時的に生産性低下
- 「Excel でできたあの機能が BI にない」という不満
- 部門間で同じ KPI を別定義で計算していたことが可視化され、政治的摩擦
- BIツールライセンス費が増加、予算増額交渉が必要

**3 次効果**:
- 「データで会話する文化」が定着し、経営会議のアジェンダが変化
- 新卒社員が BIツール前提のキャリアパスになる
- 競合他社も類似投資を開始し、データリテラシーが業界標準化
- 一部の「Excel 職人」が評価されにくくなり、離職/移動が発生
- ベンダー交渉力が一括化され、調達コストを下げられる交渉余地が生まれる

**緩和策**:
- Excel との並行利用期間を 6ヶ月設ける
- 部門独自定義を「ユーザー関数/計算フィールド」として BI に移植する仕組み
- BIツール熟練者を各部門に 1 名配置 (エヴァンジェリスト)
- KPI 定義の公式統一プロジェクトを別途立ち上げ

### 例 2: 「dbt 導入でデータ変換を IT から業務部門に移管」

**1 次効果**:
- 業務部門の要望への対応速度 UP
- IT チームの定型作業が減る

**2 次効果**:
- 業務部門が dbt を学ぶ工数を確保できず、一部の人に負荷集中
- 業務部門ごとに命名規則が乱立
- dbt モデルの重複・競合が発生
- IT チームのレビュー負荷が逆に増える (秩序維持のため)
- PR 運用・Git 理解の社員が増え、エンジニアリング文化が浸透

**3 次効果**:
- 「データスチュワード」という新しい役割が生まれ、キャリアパスに組み込まれる
- 業務部門からデータエンジニアへの転身者が出現
- dbt の使い方で部門対立が起こり、CoE 組織が新設される
- 内部のデータ利用規約が強化され、ガバナンス組織が事実上誕生

**緩和策**:
- dbt 導入前にデータスチュワードを各部門に任命
- 命名規則・レイヤー構造を先に合意
- IT チームに「PR レビュアー」役割を正式に組み込む (工数確保)
- 部門別 dbt プロジェクトに分割し、共通層のみを CoE 管理に

### 例 3: 「LLM による要約・分類を本番投入」

**1 次効果**:
- 手作業のテキスト分類が自動化
- アナリストの工数削減

**2 次効果**:
- LLM 出力の揺らぎで下流処理が時々壊れる
- プロンプトの属人化 (一部の人しかチューニングできない)
- 「LLM が出力したから正しい」と過信し、検証プロセスが形骸化
- LLM API コストが月次予算を圧迫
- プロンプトインジェクション事案が発生し、セキュリティ部門が対応に追われる

**3 次効果**:
- 「プロンプトエンジニア」という役割が組織に誕生
- LLM ベンダーへのロックインが進み、交渉力低下
- モデル更新時の re-evaluation が定常業務化
- アナリストのスキルが「LLM 出力の検証」にシフト
- LLM ガバナンス委員会が設置され、全社の AI 利用を統括

**緩和策**:
- LLM 抽象化レイヤー (LangChain 等) でベンダー切替可能性を維持
- プロンプトテンプレートを Git 管理
- 定期的な出力アライメントテスト
- コスト閾値でのアラート設定
- プロンプトインジェクション対策ガイドラインの策定

## 適用のコツ

### コツ 1: 各階層で「誰に」の視点を変える

- 1 次: 当事者 (直接の利用者)
- 2 次: 周辺ステークホルダー (他部門, 管理者)
- 3 次: 組織全体 / 市場 / 社会

### コツ 2: 「期待される副作用」と「期待されない副作用」を分ける

期待される副作用 (例: データ文化の定着) は積極的に強化。
期待されない副作用 (例: 政治的摩擦) は緩和策を設計。

### コツ 3: 3 次効果まで書けない場合は 2 次まででも十分

3 次効果は予測精度が落ちるため、2 次効果を網羅することの方が実利が大きい。

### コツ 4: 5〜10 年のタイムスパンで 3 次を想像

短期 (1〜2年) だと 3 次効果は見えづらい。5〜10年の期間で想像してみる。

## 出力フォーマット

```markdown
## 意思決定: [内容]

### 1 次効果
- [効果1]
- [効果2]

### 2 次効果
- [ポジティブ] [効果1]
- [ネガティブ] [効果2]
  - 緩和策: [策]

### 3 次効果
- [構造変化1]
- [構造変化2]
  - 緩和策: [策]
```
