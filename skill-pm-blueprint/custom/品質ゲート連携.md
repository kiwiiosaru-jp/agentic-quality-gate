# 品質ゲート連携

## 概要

`agentic-quality-gate` (v4以降, /Users/shigeru.abe/agentic-quality-gate/) は計画書を 176 件のナレッジで評価する品質ゲートツール。pm-blueprint 出力に対して評価を実行し、Fail/Conditional の指摘を受けて該当レイヤーを再生成する **往復ループ** を構築する。

## 連携フロー

```
[1] pm-blueprint 統合オーケストレーター実行 (10ステップ)
        ↓ 出力
[2] blueprint/ 配下に 計画書 + phase-outputs/ + リスクレジスタ.yml 等
        ↓ 入力
[3] agentic-quality-gate plan_review モードで評価
        ↓ 出力
[4] reports/YYYY-MM-DD_blueprint_review.md (49項目判定結果)
        ↓ Fail/Conditional 抽出
[5] 該当 Layer/サブスキル特定 (eval-mapping.md 参照)
        ↓ 再生成
[6] pm-blueprint 該当レイヤーで補強文書を作成
        ↓ 反映
[7] 計画書再構築
        → [3] へ戻り Pass まで反復
```

## 評価モード

### plan_review モード (主用途)
- 計画書中心レビュー (P0/P1/cross-cutting)
- 実装前の青写真段階で利用
- 約 49 件の P0/P1 critical+high エントリ

### code_review モード (実装後)
- コード判定 (P2-P6)
- 矢羽③④⑤ 開発 Sprint 中以降に利用

## target_phases と Layer 対応

| 品質ゲート phase | 主担当 pm-blueprint Layer |
|----------------|------------------------|
| P0 (戦略/法務) | Layer 1 (経営) / Layer 7 (法務) |
| P1 (設計/データ/API/環境) | Layer 3 (アーキ) / Layer 4 (要件) |
| P2-P6 (実装/運用) | Layer 6 (WBS実行) / Layer 9 (運用) |
| cross-cutting (LLM/セキュリティ/ガバナンス) | Layer 5 (リスク) / Layer 8 (LLM) / Layer 9 (運用) |

詳細は `quality-gates/eval-mapping.md` を参照。

## 評価実行手順

### Step 1: 評価実行
```bash
# agentic-quality-gate のディレクトリへ
cd /Users/shigeru.abe/agentic-quality-gate

# 評価対象 = pm-blueprint 出力ディレクトリ
TARGET=/Users/shigeru.abe/Documents/.../blueprint
MODE=plan_review

# 実行 (gate-evaluator)
python eval.py --target $TARGET --mode $MODE --severity critical,high
```

### Step 2: レポート確認
- 出力先: `reports/YYYY-MM-DD_blueprint_review.md`
- 集計: Pass / Conditional / Fail / N/A
- Critical Fail Top N の即時対応リスト

### Step 3: Fail/Conditional 解消フロー

優先順位:
1. **Critical Fail** (即時対応必須) — 計画書 Go/No-Go の前提
2. **High Conditional** (リリース前対応) — 矢羽② 完了前に解消
3. **Medium 以下** (継続改善) — 矢羽③④⑤ 開発中に逐次

### Step 4: 該当レイヤー特定 (eval-mapping.md)

例: `LEGAL-PII-001 Fail` → Layer 7 個人情報取扱台帳.md を実行
例: `API-IDEMPOTENCY-001 Fail` → Layer 3 API設計.md を実行
例: `LLM-AGENCY-001 Fail` → Layer 8 エージェント権限境界書.md を実行

### Step 5: 補強文書作成

該当 Layer のサブスキルを起動し、template/ から雛形を引いて埋める。

### Step 6: 計画書再構築

補強文書を計画書のセクション 13/14/15 に反映、または `phase-outputs/` に追記。

### Step 7: 再評価
[3] へ戻り、Fail が 0 になるまで反復。

## 期待される改善サイクル

### 第1ラウンド (初回評価)
- 典型: Fail 10-15件、Conditional 25-30件
- 主因: 法務・データ設計・API・LLM ガバナンス文書未作成

### 第2ラウンド (Layer 7-9 補強後)
- 期待: Fail 3-5件、Conditional 15-20件
- 主因: ARB承認記録、専門家レビュー記録の不在 (これらは人間レビュー必須)

### 第3ラウンド (人間レビュー後)
- 期待: Fail 0件、Conditional 5-10件
- Pass 率 80%超

## 自動化

### スクリプト (例)
```bash
# pm-blueprint/scripts/run_quality_gate.sh
#!/bin/bash
TARGET=$1
cd /Users/shigeru.abe/agentic-quality-gate
python eval.py --target $TARGET --mode plan_review > /tmp/result.md
critical_fail=$(grep -c "🔴.*Fail" /tmp/result.md)
echo "Critical Fail: $critical_fail"
exit $critical_fail
```

CI で `exit code != 0` なら Fail とみなし、PR ブロック。

## ループの停止条件

- Critical Fail = 0
- High Conditional ≤ 5
- 残 Conditional は人間レビュー (ARB/CISO/法務承認) 待ち

## 適用例: ある案件 (匿名化済)

第1ラウンド結果:
- Fail 13件 / Conditional 27件 / Pass 4件

→ Layer 7 / Layer 3 強化 / Layer 8 で補強 (本改訂で対応)

予測される第2ラウンド結果:
- Fail 1-3件 (ARB承認記録は人間アクション必須)
- Conditional 10-15件
- Pass 30件超
