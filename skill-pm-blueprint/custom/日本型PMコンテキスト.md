# 日本型PMコンテキスト

## このスキルの目的

Layer 1〜6 の方法論は**国際標準**に準拠しているが、日本の組織で運用する場合、**文化的な差異**を考慮する必要がある。
本スキルは、日本の組織で pm-blueprint を運用する際の**補助的ガイド**。

注意: 本スキルは一般論にとどめ、特定の企業・業界・制度名への依存を避ける。

## 日本の組織文化の特徴 (PM 観点)

### 特徴 1: 合意形成が重視される
- 意思決定は会議室ではなく、会議室**以外**で進む
- 「会議で決める」のではなく「会議は既に決まったことの承認の場」
- 根回しを怠ると、会議で通らない

### 特徴 2: 責任の明確化を避ける傾向
- 「誰が決めた」より「みんなで決めた」が好まれる
- 結果責任より**プロセス責任**が問われる
- 個人の失敗より組織の失敗で処理される

### 特徴 3: ドキュメント主義
- 稟議書、議事録、要件定義書などの書面が重要
- 口頭合意は後で「そんなこと言ってない」の対象
- ドキュメントの体裁 (フォーマット、印鑑、承認者) も重要

### 特徴 4: 序列と関係性の重視
- 発言権は役職と在籍年数に比例
- 「誰が言ったか」が「何を言ったか」より優先
- 外部ベンダーは内部関係者より発言重みが低い

### 特徴 5: 長期視点
- 短期 ROI より長期的な関係性・信用を重視
- 「次の案件も」を見越した振る舞いが期待される
- 目先の効率化より「うまく回す」

これらは良い悪いではなく、前提として理解する。

## 1. 根回しフェーズの設計

### 根回しとは
主要な意思決定の前に、**関係者個別に説明して事前同意を得る**活動。
日本の組織では、会議で初めて提案を聞くことはほぼ「失礼」と見なされる。

### 根回しのタイミング

| タイミング | 内容 | 対象 |
|----------|------|------|
| RFP 提出前 | スコープ・概算の事前共有 | 情シス部長、業務部門長 |
| キックオフ前 | 体制・役割・成果物の擦り合わせ | 全主要ステークホルダー |
| 投資判断会議前 | 判断材料の事前共有と Q&A | 経営層、稟議ルート上の役職 |
| 矢羽①完了時 | 承認前の事前合意 | 業務部門長、情シス部長 |
| 重大変更時 | 変更の背景と影響の個別説明 | 影響を受ける関係者 |

### 根回しの進め方

```
1. ステークホルダーマップ (Layer 2) を見て根回し優先度を決める
   - 「密接に管理」象限 → 最優先
   - 「満足させる」象限 → 次優先
2. 個別に 30 分程度の時間を取って説明
   - 資料は A4 1〜3 枚の短いもの
   - 質問・懸念を引き出す姿勢で
3. 懸念事項を持ち帰り、必要なら資料修正
4. 次の根回し先へ
5. 全員が概ね同意した後で正式会議へ
```

### 根回し時間の確保

年間 1 億円規模の案件で、主要ステークホルダー 10 名に根回し:
- 各 30 分 × 10 名 = 5 時間
- 調整時間込みで **3 日〜1 週間** は確保する必要がある

WBS の矢羽①に「根回し時間」を明示的に組み込むことを推奨。

## 2. ステークホルダーマッピング 日本版

国際標準の 4 象限 (影響度×関心度) に加え、**3 層構造**で見る:

### 3 層: 経営層 / 部門長 / 現場

```
[経営層 (役員)]
   │ 関心: ROI, 戦略整合, 外向き
   │ 主要資料: 投資判断書 (A4 3 枚)
   ▼
[部門長 (課長, 部長)]
   │ 関心: 部門目標達成, 実行リスク, 内向き
   │ 主要資料: 矢羽WBS, 体制図
   ▼
[現場 (担当者)]
   │ 関心: 自分の業務負荷, 使いやすさ, 学習コスト
   │ 主要資料: ユースケース, ハンズオン
```

各層で**情報の粒度と頻度が異なる**。同じ情報を全員に届けるのは非効率。

### 3 層別コミュニケーション

| 層 | 頻度 | 形式 | 主要トピック |
|---|------|------|------------|
| 経営層 | 月次 or 矢羽完了時 | 1 枚サマリー + 対面 | 予算、進捗、主要リスク |
| 部門長 | 週次 or 隔週 | 進捗報告書 + 会議 30 分 | 部門への影響、依頼事項 |
| 現場 | 週次 | Slack + Sprint Review | 使い方、フィードバック依頼 |

## 3. 合意形成プロセス (稟議想定)

### 稟議制度
日本の組織では、重要な意思決定は**稟議書**で承認経路を回ることが多い。

典型的な稟議ルート:

```
起案者 → 課長 → 部長 → 部門長 → 経営層 → 承認
```

各段階で「承認 / 差し戻し / 保留」が発生し、1 段戻るとやり直し。

### 稟議を通すためのポイント

1. **根回し済み**: 稟議提出時には全承認者に個別説明済み
2. **フォーマット遵守**: 組織の稟議書フォーマットに完全準拠
3. **前例参照**: 類似の過去稟議を引用 (「前回の X 案件では…」)
4. **リスクへの備え**: 想定質問と回答をセットで準備
5. **金額の刻み**: 決裁権限に合わせて金額を設計 (例: 課長権限 300 万円以下)

### 稟議にかかる時間

| 規模 | 想定期間 |
|------|---------|
| 100 万円以下 | 1〜2 週間 |
| 100〜1000 万円 | 2〜4 週間 |
| 1000 万円超 | 1〜2ヶ月 |
| 1 億円超 (役員稟議) | 2〜3ヶ月 |

稟議期間は WBS に組み込む。矢羽①完了から矢羽②開始までに 1〜2ヶ月のバッファを設ける。

## 4. 報連相のリズム設計

日本の組織で好まれる報告パターン:

### 定例報告

| 頻度 | 形式 | 内容 |
|-----|------|------|
| 日次 (非公式) | Slack / Teams | ブロッカーのみ |
| 週次 | 進捗メール or 会議 | 先週/今週/来週、リスク |
| 月次 | 月次レビュー会議 | KPI, 予算、変更 |
| 矢羽完了時 | フェーズゲートレビュー | 成果物承認 |

### 報連相の質

- **事実と判断を分離** (「売上 120%」「これを順調と判断」)
- **悪い報告を早く** (「問題が発生しました。対応策を検討中」)
- **相談と報告を分離** (「相談ですが、A と B の比較です。意見ください」)

### 「順調です」は要注意

プロジェクト中盤で「順調です」が 3 回続いたら、以下のいずれか:
1. 本当に順調 (稀)
2. 問題を伝えていない
3. 問題を把握していない

PM は「具体的に何が順調ですか?」「懸念はゼロですか?」と掘り下げる。

## 5. キックオフミーティング 日本向けアジェンダ

### 典型的アジェンダ (2 時間)

| 時間 | 内容 | 担当 |
|------|------|------|
| 5 分 | 開会挨拶 | スポンサー (役員) |
| 10 分 | 自己紹介 (全員) | 全員 |
| 15 分 | プロジェクト背景・目的 | PM |
| 20 分 | ステークホルダー | PM |
| 20 分 | スコープと成果物 | PM |
| 15 分 | 矢羽 WBS | PM |
| 10 分 | 体制と役割 | PM |
| 10 分 | 主要リスクと Kill 基準 | PM |
| 10 分 | 次ステップ | PM |
| 5 分 | 質疑 | 全員 |

### 日本式の留意点

- 役員は**冒頭のみ参加**することが多い (10〜15 分で離席)
- 質疑は**個別後日**が多い (その場で反論しない文化)
- 懇親会 (オンラインでも) が効果的な関係構築の場

## 6. プロジェクトオーナー / スポンサーの明確化

### 混同されがちな役割

| 役割 | 責任 | 日本で混同されやすい |
|-----|------|---------------------|
| プロジェクトオーナー | 意思決定、成果責任 | 業務部門長 |
| プロジェクトスポンサー | 予算、経営との接続 | 役員 |
| プロジェクトマネジャー | 実行責任 | PM |
| ステークホルダー | 影響を受ける | 関連部署 |

### 明確化の仕方

キックオフ時に「このプロジェクトの意思決定最終権者は○○さん」と明文化。
逃げ道を作らない。

「相談します」と言っても最終的に誰が決めるかを明示する。

## 7. ドキュメント主義への対応

### 必須ドキュメント (日本向け)

| 成果物 | 用途 |
|-------|------|
| プロジェクト憲章 | 稟議・合意形成の基礎 |
| 矢羽WBS (ガントチャート付き) | 進捗管理の共通言語 |
| 要件定義書 | 業務部門承認の形式的材料 |
| 議事録 | 後日の「言った言わない」対策 |
| 報告書 (週次/月次) | 稟議履歴の残存 |
| 最終報告書 | プロジェクト完了の証跡 |

### フォーマットの重要性

- 組織の既存フォーマットに合わせる (独自フォーマットは通りにくい)
- 印鑑・承認欄・日付の形式を遵守
- PDF 化、改ざん防止

### バランス

ドキュメント主義は過剰になると形式主義に陥る。以下を目指す:

- **必要最小限の必須ドキュメント**
- **Notion / Confluence 等で Living Document 化**
- **コードに近いもの (ADR, 仕様) は Git 管理**

## 8. 矢羽パターン完了時の顧客承認ゲート

### 各矢羽完了時の承認

| 矢羽 | 承認者 | 承認材料 |
|------|-------|---------|
| 矢羽① 業務要件定義 | 業務部門長 | 業務要件定義書、ユースケース一覧 |
| 矢羽② PoC+システム要件 | 情シス部長、業務部門長 | PoC報告書、システム要件定義書、アーキ図 |
| 矢羽③ データ連携 | 情シス部長 | 結合テスト結果 |
| 矢羽④ AI分析 | 業務部門長 | データマート仕様、AI出力サンプル |
| 矢羽⑤ 検索・BI | 業務部門長 | ダッシュボード仕様、デモ |
| 矢羽⑥ 非機能・運用 | 情シス部長、運用部門長 | 運用手順書、障害対応フロー |
| 矢羽⑦ UAT・パイロット | 業務部門長、経営層 | UAT結果、パイロット評価 |

### 承認ゲートを通すコツ

- 承認会議の**前日まで**に資料を配布
- 質問リストを事前に共有
- 反対意見を会議外で個別対応
- 「承認率」を上げる事前調整

### ゲート不合格時の対応

- 差し戻し内容を明文化
- 再提出期限を設定 (通常 1〜2 週間)
- 差し戻し原因の根本分析 (前の根回し不足 / 要件認識違い)

## 適用時の留意点

### 適用すべき案件

- 日本の伝統的大企業・金融・製造業
- 官公庁、公共事業
- 親会社が日本企業の子会社

### 適用を控える案件

- 外資系企業 (本社が海外)
- スタートアップ
- 純粋 IT 企業 (Web 系)
- グローバルチーム

### バランスの取り方

本スキルの慣習は**過剰に従うと進捗を阻害**する。以下を見極めて適用:

- 根回しは「必要十分」に (全員根回しだと時間尽きる)
- 稟議期間はバッファで吸収、無駄な待ち時間は作らない
- ドキュメントは「意思決定の記録」に絞り、形式主義は避ける
- 国際標準の手法を諦めない (エビデンスで説得)
