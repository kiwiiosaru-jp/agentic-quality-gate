# 統合オーケストレーター (2026-04-30 改訂版)

## 改訂履歴
- **2026-04-30**: 品質評価 (`agentic-quality-gate v4` 49項目評価) を受け、7→**10ステップ**に拡張。Layer 7 (法務)、Layer 8 (LLM ガバナンス)、Layer 9 (運用) を新規統合。
- 2026-04-24: 初版 (7ステップ、6レイヤー)

## このスキルの位置づけ

pm-blueprint 全体の **起点となるメタスキル**。
全 9 レイヤーを **10 ステップフロー** で順次呼び出し、最終的に **15 セクション構成**のプロジェクト計画書を生成する。

## 標準実行フロー (10 ステップ)

```
[Step 1-2: 仮説構築]
  ステップ1: Layer 2 ディスカバリー
  ステップ2: Layer 2 前提抽出 (4軸 V/U/Vi/F)

[Step 3-5: 設計詳細化 — 拡張]
  ステップ3: Layer 3 ADR + データ設計詳細 + API設計 + アーキスタイル選定 + 環境分離_IaC
  ステップ4: Layer 4 ユースケース + SMART-NFR + EARS + SLI/SLO + DR + 可観測性
  ステップ5: Layer 5 リスク + 認可方式 + 暗号化_KMS + ゼロトラスト + LINDDUN

[Step 6-7: 実行計画]
  ステップ6: Layer 1 経営判断 (投資/ピボット/撤退)
  ステップ7: Layer 6 WBS + トレーサビリティ

[Step 8-10: 品質ゲート対応 — 新規]
  ステップ8: Layer 7 法務・コンプライアンス文書化
  ステップ9: Layer 8 LLM ガバナンス文書化
  ステップ10: Layer 9 運用ドキュメント + 品質ゲートチェック
```

## 各ステップ詳細

### ステップ 1: ディスカバリー (Layer 2)
**入力**: RFP、要件概要、ヒアリング記録
**実行**: JTBD / ステークホルダーマップ / OST / 多視点ブレスト
**出力**: `step-1-discovery.md` (JTBD、SH 4象限、OST、不足情報リスト)

### ステップ 2: 前提抽出 (Layer 2)
**実行**: V/U/Vi/F 4軸前提抽出、不確実性マトリクス、検証実験設計
**出力**: `step-2-assumptions.md`

### ステップ 3: アーキ設計 詳細 (Layer 3 強化版)
**実行**:
- 3a. ADR (技術選定の Type 1 決定 5-8件)
- 3b. **データ設計詳細** (ERD / Tx境界 / Outbox / SoT / 整合性レベル / 履歴保持)
- 3c. **API設計** (REST/GraphQL/gRPC選定 / Idempotency-Key / バージョニング / OpenAPI)
- 3d. **アーキスタイル選定** (IaaS/PaaS/SaaS / モノリス vs MS / レイヤード/ヘキサ/オニオン)
- 3e. **環境分離_IaC** (dev/stg/prod / Terraform / データフロー禁則)

**出力**: `step-3-adrs.md`、`step-3-data-design.md`、`step-3-api-design.md`、`step-3-arch-style.md`、`step-3-env-iac.md`

### ステップ 4: 要件定義 (Layer 4 強化版)
**実行**:
- 4a. ユースケース (10-20件、Use Case 2.0 slim)
- 4b. SMART NFR (FURPS+ ランディングゾーン)
- 4c. EARS 形式要件文 (30文)
- 4d. **SLI/SLO/エラーバジェット** (29ppm計算、バーンダウン速度)
- 4e. **RTO/RPO/DR設計** (Active-Active等、演習頻度)
- 4f. **可観測性三本柱** (log/metric/trace、分散トレースID)

**出力**: `step-4-requirements.md`、`step-4-slo.md`、`step-4-dr.md`、`step-4-observability.md`

### ステップ 5: リスク・セキュリティ (Layer 5 強化版)
**実行**:
- 5a. 事前検死 (Pre-mortem 失敗シナリオ)
- 5b. 反転思考 (Inversion 確実失敗パターン)
- 5c. リスクレジスタ (確率×影響、Kill基準)
- 5d. STRIDE 脅威モデリング
- 5e. **LINDDUN プライバシー脅威** (録音・LLM 案件で必須)
- 5f. **認可方式選定** (RBAC/ABAC/ReBAC/OPA)
- 5g. **暗号化_KMS設計** (Key Vault、鍵ローテーション、BYOK)
- 5h. **ゼロトラスト方針** (内部例外撤廃、JIT昇格、CISO承認)

**出力**: `step-5-risks.md`、`step-5-authz.md`、`step-5-crypto.md`、`step-5-zerotrust.md`、`step-5-linddun.md`、`リスクレジスタ.yml`

### ステップ 6: 経営判断 (Layer 1)
**実行**: 投資判断書 / ピボット判断 / 撤退判断
**出力**: `step-6-executive.md`

### ステップ 7: WBS (Layer 6)
**実行**: 矢羽パターンWBS (L2/L3/L4)、トレーサビリティマトリクス
**出力**: `step-7-wbs.md`、`トレーサビリティマトリクス.md`

### ステップ 8: 法務・コンプライアンス (Layer 7 — 新規)
**実行 (9サブスキル)**:
- 8a. 個人情報取扱台帳 (LEGAL-PII-001)
- 8b. 個情法対応マトリクス (COMPLY-PIPA-JP-001)
- 8c. 業法該当性判定書 (LEGAL-INDUSTRY-001)
- 8d. データ分類台帳 (LEGAL-DATACLASS-001)
- 8e. AI生成物著作権ポリシー (LEGAL-AIGEN-001)
- 8f. 越境データ移転書 (LEGAL-CROSSBORDER-001)
- 8g. OSS ライセンス管理 (LEGAL-LICENSE-001)
- 8h. データ保持_削除戦略 (DATA-DELETE-001 / COMPLY-RETAIN-001)
- 8i. 法務承認記録 (法務 + DPO + CISO)

**出力**: `step-8-legal/` 配下 各文書

### ステップ 9: LLM ガバナンス (Layer 8 — 新規)
**実行 (8サブスキル、AI/LLM 利用案件のみ)**:
- 9a. エージェント権限境界書 (LLM-AGENCY-001)
- 9b. プロンプトインジェクション評価セット (LLM-INJECT-DIR/INDIR-001)
- 9c. ハルシネーション対策 (LLM-HALLUCIN-001)
- 9d. PII境界_DLP (LLM-PII-001)
- 9e. LLM出力検証_許可リスト (LLM-OUTPUT-TRUST-001)
- 9f. モデルドリフト検知 (LLM-DRIFT-001)
- 9g. 評価セット_回帰検知 (LLM-EVAL-REGR-001)
- 9h. AI データ境界ガイド (GOV-AI-DATAFLOW-001)

**出力**: `step-9-llm/` 配下 各文書

### ステップ 10: 運用ドキュメント + 品質ゲート (Layer 9 + quality-gates/ — 新規)
**実行**:
- 10a. ランブック規約 (DOC-RUNBOOK-001)
- 10b. 監査証跡設計 (GOV-AUDIT-TRAIL-001)
- 10c. シャドー AI 禁止ガバナンス (GOV-SHADOW-AI-001)
- 10d. **品質ゲートチェック実行** (`quality-gates/checklist.yml` の 49 項目を出力ファイルに突合)

**出力**: `step-10-ops/` 配下 + `quality-gate-result.md`


## 情報不足時の質問リスト (18項目に拡張)

旧 12項目 (予算/期限/対象ユーザー/データソース/既存システム/クラウド方針/既存スキル/法規制/成功基準/制約/失敗経験/ベンダーロックイン) に加え:

13. **個人情報取扱の同意取得方法** (PII 法務承認の前提)
14. **業法該当性** (業種・取引先での該当法令確認)
15. **AI生成物の責任所在** (生成物を顧客に提供するか)
16. **SLO目標** (稼働率/応答時間の経営合意水準)
17. **DR要件** (RTO/RPO の経営合意、演習可能か)
18. **認可方式の社内標準** (Entra ID / OPA 等の既存方針)
19. **シャドーAI 禁止ルール** (4,500名向けポリシー有無)

[要確認] タグで明示する。

## 実行時の留意事項

### 仮説は仮説と明示する
すべての前提・未検証事項に `[仮定]` `[要確認]` タグを付ける。

### Layer 7-9 の適用判断
- **Layer 7 (法務)**: 個人情報・業法該当性のある案件で必須
- **Layer 8 (LLM)**: LLM/AI を業務で利用する案件で必須
- **Layer 9 (運用)**: 24h365日稼働・規制対象で必須

### 品質ゲート連携
ステップ10完了後、`custom/品質ゲート連携.md` の手順で `agentic-quality-gate` (v4以降) と連携可能。Fail/Conditional 結果に基づき該当レイヤー再生成のループを回せる。

### スキップ条件 (情報不足時)
- ステップ3 ADR → 「技術選定はステップ後半 PoC で最終化」と明記
- ステップ4 EARS → 「詳細要件文はシステム要件定義段階で作成」
- ステップ8-10 → AI/LLM 不使用案件はステップ9をスキップ可
