# AI駆動開発 特有リスク チェックリスト

## 位置づけ

Layer 5 `事前検死.md` の**ドメイン特化版**。
以下のいずれかに該当する案件で**追加適用**する:

- LLM を業務処理に組み込む (要約、分類、生成)
- AI エージェント (Claude Code, Copilot 等) を開発フローに組み込む
- RAG / Embedding / Vector DB を使う
- Text2SQL / Text2BI 機能を持つ
- プロンプト中心の開発パターンを採用

通常のリスクレジスタに加えて、本チェックリストの 13 項目を追加する。

## 13 項目チェックリスト

### R-AI-01: コンテキストドリフト

**兆候:**
- チャットセッションが長くなるにつれ、AI の回答精度が低下
- 過去のプロンプトで決めたルールが忘れられている
- AI エージェントが指示に反する動作を始める

**予防策:**
- システムプロンプトで重要ルールを毎回再注入
- 長時間セッションは定期的にリセット
- Context Window の最大トークンを監視
- Claude Code なら `/memory` や CLAUDE.md で安定ルールを定着

**検知方法:**
- 出力ログに「指示に反する動作」が出た回数を計測
- ユーザーフィードバック (「さっき言ったのに」) の頻度

### R-AI-02: 意思決定疲労 (Decision Fatigue)

**兆候:**
- 開発者が毎日 AI の提案に Yes/No を 100 回判断
- 疲れて「とりあえず承認」が増える
- レビュー品質が低下

**予防策:**
- 自動承認できるパターンを事前に合意 (テスト追加、リファクタリング等)
- 日次の AI 判断数を制限 (上限 50 回等)
- ペアレビュー (人間 2 名でローテーション)
- チェックリストで「何に同意して何に反対するか」を明確化

**検知方法:**
- PR マージ速度の急上昇 (= 碌にレビューしていない兆候)
- 本番障害の増加

### R-AI-03: 便益分配の偏在

**兆候:**
- 一部のメンバだけ AI を使いこなして生産性 2 倍
- 他メンバは従来通りで相対的に「遅い」とされる
- 評価制度が追いついていない

**予防策:**
- AI 活用のベストプラクティスを全員で共有 (週次勉強会)
- メンバ別の AI 活用度を可視化
- 評価制度に「AI を使ったアウトプット」も含める
- スキル格差を個人責任にしない文化

**検知方法:**
- メンバ別の PR 生産量ばらつき
- 退職面談で「評価に納得できない」コメント

### R-AI-04: 仕様曖昧起因のやり直し

**兆候:**
- AI に曖昧な指示を出して、期待と違うアウトプット
- 修正指示の往復で時間を浪費
- 「最初から自分で書いた方が早かった」

**予防策:**
- プロンプトに入力・出力仕様を明示 (スキーマ、例、制約)
- 小さく試して確認してから大きく生成
- 複雑なタスクは分解して逐次確認
- EARS 形式で要件を事前に詰める

**検知方法:**
- プロンプト往復回数の計測
- タスク完了までの所要時間が想定の 2 倍超

### R-AI-05: AI 出力の過信によるレビュー品質低下

**兆候:**
- 「AI が出したから正しい」とレビューが甘くなる
- テスト通過で OK とし、設計レビューが省略される
- 誤った出力がそのまま本番投入

**予防策:**
- レビューチェックリストを明示 (ロジック、セキュリティ、性能、エッジケース)
- AI 出力には **人間サインオフ**を必須化
- クリティカルパス (決済、認証、データ変更) は複数名レビュー
- AI 出力にも単体テスト必須

**検知方法:**
- 本番バグ率の増減
- レビュー時間の激減 (表面的レビューの兆候)

### R-AI-06: プロンプトの属人化

**兆候:**
- 特定メンバしか書けないプロンプトが本番で稼働
- そのメンバが休むと問題対応できない
- プロンプトの意図が後から分からない

**予防策:**
- プロンプトを Git 管理 (バージョニング、レビュー)
- プロンプトにコメントで意図を記載
- プロンプトテストを自動化
- ペアプロンプティング (共同作成・共同理解)

**検知方法:**
- プロンプト修正の Git 履歴に特定名が集中していないか
- 「プロンプト修正できる人」のバス係数チェック

### R-AI-07: モデル変更による再現性喪失

**兆候:**
- ベンダーが LLM モデル更新 → 出力フォーマットが微妙に変わる
- 下流処理がエラー
- 過去のテスト結果と一致しない

**予防策:**
- モデルバージョン固定 (API バージョン指定)
- モデル変更時の回帰テスト必須
- アライメントテスト (期待出力との乖離度合い) を定期実行
- 出力パースに堅牢性 (スキーマ検証、リトライ)

**検知方法:**
- アライメントテストのスコア低下
- 下流処理のエラー率増加

### R-AI-08: テストカバレッジの表面的拡張

**兆候:**
- AI が大量のテストを生成 → カバレッジ 95%
- でも実は自明なケースばかり
- 実際のバグはエッジケース (テストされていない)

**予防策:**
- カバレッジの「量」だけでなく「質」を評価
- Mutation Testing でテスト有効性を検証
- エッジケース・エラーケースのテストを人間が追加
- AI 生成テストのレビュー基準を明示

**検知方法:**
- カバレッジ vs 本番バグ率の乖離
- Mutation Score の確認

### R-AI-09: ドキュメント生成と実態の乖離

**兆候:**
- AI がドキュメントを生成 → 実装とズレていく
- README の手順通りにやると動かない
- ドキュメントを誰も信じなくなる

**予防策:**
- ドキュメント → コード生成ではなく、コード → ドキュメント生成
- ドキュメント検証テスト (コード例が実行可能か等)
- 変更時に関連ドキュメントを必ず更新 (PR ルール)
- ドキュメントの「Last Updated」をチェック

**検知方法:**
- ドキュメント記載の手順で実行成功率
- ユーザーからの「ドキュメント間違い」報告件数

### R-AI-10: マルチエージェント協調失敗

**兆候:**
- 複数の AI エージェントを並列起動
- 同時に同じファイルを修正してコンフリクト
- エージェント間で意図が噛み合わない

**予防策:**
- エージェント間の責任範囲を明確化
- 共有リソースへのロック機構
- 各エージェントの出力を順次統合 (並列ではなく直列)
- オーケストレーター役の人間が全体把握

**検知方法:**
- エージェント間コンフリクト数
- 統合後の手戻り量

### R-AI-11: トークンコスト膨張

**兆候:**
- LLM API 利用料が月次予算の 200% 超
- 長文入力、繰り返し呼び出しでコスト爆発
- プロンプトキャッシュが効いていない

**予防策:**
- プロンプトキャッシュを活用 (Claude API なら `cache_control`)
- Context Window を必要最小限に
- コスト上限アラート (80% / 100% / 150%)
- モデル使い分け (Haiku/Sonnet/Opus)

**検知方法:**
- 日次コスト集計
- トークン数 / タスク のメトリクス

### R-AI-12: プロンプトインジェクション懸念

**兆候:**
- ユーザー入力が LLM プロンプトに直接混入
- 悪意ある入力で指示を奪われる可能性
- システムプロンプトを上書きされる

**予防策:**
- ユーザー入力と指示を明確に分離 (区切り子、フォーマット)
- ユーザー入力の無害化 (サニタイゼーション)
- LLM 出力のバリデーション (期待外の内容を検出)
- 機微な操作には人間承認必須

**検知方法:**
- 異常なプロンプトパターン検出
- LLM 出力の監査ログ

### R-AI-13: 知的貢献の評価困難

**兆候:**
- AI が書いたコードが PR の大半を占める
- 「誰の成果」が不明瞭
- 人事評価で混乱

**予防策:**
- AI 生成部分と人間貢献部分をコミットで分ける (または PR テンプレートで明示)
- 評価軸を「最終品質・設計判断・レビュー深度」にシフト
- AI 使いこなしも評価対象に含める
- チーム全体の生産性を評価軸に (個人単位ではなく)

**検知方法:**
- 人事評価での議論時間増
- メンバのモチベーション低下

## 適用手順

### ステップ 1: 該当判定

プロジェクトが AI 駆動開発に該当するか確認。該当しなければスキップ。

### ステップ 2: 13 項目の個別評価

各項目を以下で評価:

| ID | 項目 | 該当するか | 確率 | 影響 | 警告サイン | 予防策採用 |
|----|------|-----------|------|------|-----------|-----------|
| R-AI-01 | コンテキストドリフト | ○ | 中 | 中 | [...] | [...] |
| R-AI-02 | 意思決定疲労 | ○ | 中 | 中 | [...] | [...] |
| ... | ... | ... | ... | ... | ... | ... |

### ステップ 3: リスクレジスタへ転記

該当する項目を Layer 5 の `リスクレジスタ.md` の YAML 形式に転記。
既存の R001〜ID 体系に組み込む (例: R025 から R037)。

### ステップ 4: Kill基準の設定

特に R-AI-11 (コスト膨張), R-AI-12 (プロンプトインジェクション) には**必ず Kill基準**を設定:

- R-AI-11: 「月次 LLM コストが予算 200% を 2ヶ月連続で超えたらモデル選定見直し」
- R-AI-12: 「プロンプトインジェクション事案発生時は 24h 以内にパイプライン停止」
