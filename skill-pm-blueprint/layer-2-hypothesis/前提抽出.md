# 前提抽出 (Assumption Mapping)

## 概要

プロジェクトの失敗は「やってみたら前提が間違っていた」に帰着することが多い。
前提抽出は、**暗黙の前提を明示化**し、**最もリスキーなものから検証**する活動。

David Bland "Testing Business Ideas" が提唱した 4 軸:

- **Value (価値)**: ユーザーが本当にそれを欲しがるか?
- **Usability (使いやすさ)**: ユーザーは実際に使えるか?
- **Viability (事業性)**: 事業として成立するか?
- **Feasibility (実現可能性)**: 技術的・組織的に作れるか?

## いつ使うか

- ディスカバリー直後、ユースケース仮説を固める前
- 矢羽①業務要件定義の前 (検証するべき前提を明示)
- PoC 計画時 (何を検証するかを決める)
- 仕様変更が頻発した時 (根本の前提を再点検)

## 手順

### ステップ 1: 4 軸で前提を洗い出し

各軸について、**質問リスト**を投げかけて前提を抽出する。

### ステップ 2: 不確実性マトリクスで優先順位付け

抽出した前提を以下の 2 軸で評価:

- **影響度**: この前提が間違っていたら、プロジェクトにどれくらい影響するか (低/中/高)
- **不確実性**: この前提が正しいという確信度は低いか (確信度高 = 不確実性低)

```
          影響度 高
            │
[最優先検証] │ [優先検証]
  不確実性高 │ 不確実性低
   ─────────┼───────── 不確実性
[あとで検証] │ [放置可]
            │
          影響度 低
```

### ステップ 3: 検証計画の作成

最優先検証象限の前提に対して、**最小コストで検証する実験**を設計:

- ユーザーインタビュー
- プロトタイプ / Wizard of Oz
- データ分析 (既存データでの検証)
- PoC (限定スコープの実装)
- 競合ベンチマーク

### ステップ 4: 検証結果のフィードバック

検証結果に応じて:

- 前提が正しかった → 次フェーズへ
- 前提が間違っていた → ユースケース、スコープ、アーキを修正
- まだ分からない → さらなる実験

## Value 軸: ユーザーはこれを欲しがるか?

### 質問リスト

- [ ] このシステムで解決したいジョブは、ユーザーにとって実際に痛みか?
- [ ] ユーザーは現状どのように代替しているか?
- [ ] 代替手段ではなくこのシステムを選ぶインセンティブは?
- [ ] 価値を感じる頻度は? (毎日 / 月次 / 四半期)
- [ ] 価値の大きさを定量化できるか? (時間削減, 売上増)
- [ ] 提供価値はユーザー層で均一か? 一部のパワーユーザーに偏るか?
- [ ] この価値は 3 年後も残るか?
- [ ] ユーザーが「これは要らない」と気づく兆候は?

### データ分析基盤での典型的 Value 前提

- [仮定] 経営層は月次 KPI ダッシュボードを閲覧する (実際には CFO が Excel で作った PDF を読むだけかも)
- [仮定] マーケ部はセグメント抽出を日常的に行いたい (年に数回のキャンペーン時のみかも)
- [仮定] 店舗運営者は店舗別 KPI を見たい (本部からの依頼で受動的に見ているだけかも)
- [仮定] データエンジニアは dbt の導入を歓迎する (既存 SQL 運用に慣れているかも)

## Usability 軸: ユーザーは使えるか?

### 質問リスト

- [ ] ユーザーの現在のツールスキルは? (SQL, BI, Excel)
- [ ] 学習コストはどれくらい許容されるか?
- [ ] 初回ログインから価値獲得までの時間は?
- [ ] 多言語対応は必要か?
- [ ] アクセス方法は? (デスクトップ / モバイル / オフライン)
- [ ] ユーザーは支援なしで使えるか? (オンボーディング前提か)
- [ ] アクセシビリティ要件は?

### データ分析基盤での典型的 Usability 前提

- [仮定] 経営層は BIダッシュボードを自分でフィルタリングする (実際には秘書が代行するかも)
- [仮定] 業務部門は SQL を書ける (Excel 関数が限界かも)
- [仮定] セルフサービス分析が機能する (質問を投げる先が無くて使われないかも)
- [仮定] BI ツールの UI は現行 Excel よりわかりやすい (逆に混乱するかも)

## Viability 軸: 事業として成立するか?

### 質問リスト

- [ ] 投資額に対する ROI は? (定量・定性)
- [ ] 損益分岐点はどこか?
- [ ] 運用コスト (クラウド, ライセンス) の継続性は?
- [ ] 組織内の予算承認プロセスをクリアできるか?
- [ ] 競合優位性は?
- [ ] 3 年後にこの投資は評価されるか?
- [ ] 関連部門の評価基準と整合するか?
- [ ] 全社戦略との整合性は?

### データ分析基盤での典型的 Viability 前提

- [仮定] 経営層はデータドリブン経営にコミットしている (単なる流行追従かも)
- [仮定] 運用コストは予算内に収まる (データ量増加で 3 年後には 2 倍になるかも)
- [仮定] ライセンス更新は承認される (他投資優先で打ち切られるかも)
- [仮定] 内製できるリソースがある (3 年後にキーパーソンが離脱するかも)

## Feasibility 軸: 実現可能か?

### 質問リスト

- [ ] 技術的に実装可能か? (前例あるか, 標準ツールで作れるか)
- [ ] データは実際に取得できるか? (契約, 品質, 鮮度)
- [ ] チームのスキルで作れるか?
- [ ] 期限内に完成可能か?
- [ ] 外部依存 (ベンダー, 取引先) はコントロール可能か?
- [ ] 組織的な障害 (部門間調整) は越えられるか?
- [ ] 法的・規制上の障害は?
- [ ] セキュリティ要件を満たせるか?

### データ分析基盤での典型的 Feasibility 前提

- [仮定] POS データは API で日次取得できる (実際にはファイル連携でタイムラグありかも)
- [仮定] 会員DBの PII はマスキング処理後に分析利用できる (同意範囲外かも)
- [仮定] dbt 経験者を採用可能 (市場で供給少なく年収水準が予算と合わないかも)
- [仮定] セキュリティ部門は方針 X を承認する (実は別方針を推しているかも)
- [仮定] ソース系運用チームは協力的 (業務繁忙期で対応不可かも)

## 不確実性マトリクスの例

データ分析基盤プロジェクトの抽出結果例:

| 前提 | 軸 | 影響度 | 不確実性 | 象限 | 検証方法 |
|------|---|-------|---------|------|---------|
| 経営層が月次KPIを実際に閲覧する | Value | 高 | 高 | 最優先 | 役員ヒアリング、現行PDF配布の開封率分析 |
| POS API で日次連携可能 | Feasibility | 高 | 高 | 最優先 | IT側にAPI仕様書入手、PoC接続 |
| dbt 経験者採用可能 | Feasibility | 中 | 高 | 優先 | 採用市場調査、外部パートナー打診 |
| 業務部門はSQL 書ける | Usability | 高 | 中 | 優先 | ヒアリング、既存 Excel 高度関数利用度調査 |
| クラウドコスト予算内 | Viability | 高 | 中 | 優先 | 類似ベンチマーク、ベンダー試算 |
| セキュリティが方針Xを承認 | Feasibility | 高 | 低 | あとで検証 | 既存関連プロジェクト参照 |
| 経営層はデータドリブン経営にコミット | Viability | 中 | 低 | あとで検証 | ToP 会議議事録参照 |

## 検証実験の設計例

### 最優先検証 1: 「経営層が月次KPIを実際に閲覧する」

**仮説**: 経営層は月次KPIダッシュボードを週 1 回以上閲覧する

**検証方法**:
- 役員 3 名にインタビュー (30分)
- 現状 PDF の開封率をメール送信ログで確認
- モックダッシュボード (Figma) を見せて利用頻度を予測してもらう

**結果判定基準**:
- 週 1 回以上閲覧意図あり: 仮説支持 → 継続
- 月 1 回程度: 仮説部分支持 → スコープ縮小 (月次 Email 配信に変更)
- 閲覧しない意向: 仮説棄却 → ピボット (別ステークホルダー向けに再設計)

**期間**: 2 週間

### 最優先検証 2: 「POS API で日次連携可能」

**仮説**: 既存 POS システムから日次で API 経由でデータ取得可能

**検証方法**:
- IT部門に API 仕様書入手依頼
- PoC 環境から POS API に接続テスト
- 1 週間分のデータ取得実験

**結果判定基準**:
- 接続成功、日次連携可: 仮説支持
- API はあるが同期が遅い/データ欠損: 部分支持 → ストリーム対応検討
- API 未整備: 仮説棄却 → 代替方式 (ファイル連携, CDC) 検討

**期間**: 2 週間
