# Layer 2: 仮説・ディスカバリー

## このレイヤーの目的

プロジェクト開始前、情報が十分に揃っていない段階で、**仮説ベースで**骨格を描く。

- 何を作るのか (ユースケース仮説)
- なぜ作るのか (Jobs to be Done)
- 誰が使うのか (ステークホルダーマップ)
- どういう順序で作るのか (ロードマップ)
- 何が不確かか (前提と仮説の明示)

仮説は仮説であることを明示し、検証計画とセットで扱う。未検証の前提には `[仮定]` タグを付ける。

## 推奨実行順序

1. **ディスカバリー.md** - JTBD、ステークホルダーマップ、Opportunity Solution Tree で全体像把握
2. **前提抽出.md** - 4 軸 (Value / Usability / Viability / Feasibility) で前提を整理
3. **ロードマップ逆算.md** - 期限・マイルストーンからクリティカルパスを定義

## 入力 (前レイヤー or RFP から)

- RFP、提案依頼書
- 顧客ヒアリング記録
- 業界の既往事例 (公開情報)
- ステークホルダーの発言録

## 出力 (次レイヤーへ渡すもの)

- **ユースケース仮説一覧** (10〜30 件)
- **JTBD (Jobs to be Done) 分析**
- **ステークホルダーマップ** (影響度 × 関心度)
- **前提リスト** (4 軸分類、不確実性スコア付き)
- **不確実性マトリクス** (優先検証順)
- **ロードマップ初版** (Gantt形式、矢羽パターン)
- **キックオフ前確認事項リスト** (不足情報の洗い出し)

これらは Layer 3 (アーキ), Layer 4 (要件), Layer 5 (リスク) の入力となる。

## 年間1億円規模での軽量化方針

- ユースケース仮説は 10〜20 件で開始、矢羽①で精緻化
- JTBD は主要 3〜5 ジョブに絞る
- ステークホルダーは 10〜20 人まで
- 前提リストは 4 軸合計 20〜40 項目
- ロードマップは矢羽レベル (L2) でまず 1 枚に収める
