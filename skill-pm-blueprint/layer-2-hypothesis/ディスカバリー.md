# ディスカバリー (Discovery)

## 概要

プロジェクト開始前、「何を作るべきか」を**ユーザー・業務・経営視点で掘り下げる**活動。
実装に入る前に以下を揃える:

- 何のために作るのか (ジョブ)
- 誰に影響するのか (ステークホルダー)
- どんな解決策があるのか (機会と選択肢)
- 複数視点での仮説 (PM / エンジニア / ユーザー / 経営)

Teresa Torres の Continuous Discovery, Anthony Ulwick の Outcome-Driven Innovation 等を参考に、本スキルでは**軽量版**を提供。

## いつ使うか

- RFP 受領直後、提案作成前
- キックオフ前、ユースケース仮説を固める時
- 仕様変更要望が多発した時 (本質的ジョブに立ち戻る)
- スコープ交渉時

## 4 つのフレームワーク

### A. Jobs to be Done (JTBD)

「ユーザーは**製品を雇う**のではなく、**ジョブを完了するために製品を雇う**」(Clayton Christensen)。

ユースケースやペルソナよりも **1 段抽象度が高い**問い:

- このユーザーは、どんな進歩 (progress) を求めているのか
- どんな状況 (circumstance) で、それを求めるのか
- どんな力 (force) がユーザーを動かすのか

#### JTBD 記述テンプレート

```
[状況] の時、[動機] したい
なぜなら [期待される成果] を得たいから
```

#### データ分析基盤での JTBD 例

- 経営層: 「月次経営会議の準備時に、業績の異常兆候を早期に察知したい、なぜなら意思決定のタイミングを逃したくないから」
- マーケ分析: 「キャンペーン企画の初動時に、類似セグメントの過去反応を確認したい、なぜなら仮説を早く検証したいから」
- 店舗運営: 「朝礼前に、昨日の店舗KPIを一目で確認したい、なぜならスタッフに当日目標を即座に共有したいから」
- データエンジニア: 「新規データソース追加時に、既存の類似モデルを参照したい、なぜならゼロから作らず効率化したいから」

### B. ステークホルダーマッピング

プロジェクトに関わる**全ての人・組織**を洗い出し、影響度と関心度で 4 象限に分類。

```
              影響度 高
                 │
   [満足させる]  │  [密接に管理]
                 │
   ──────────────┼────────────── 関心度
                 │
   [監視]        │  [情報共有]
                 │
              影響度 低
```

| 象限 | 対応 |
|------|------|
| 密接に管理 (影響度高・関心度高) | 定期ミーティング、意思決定パートナー |
| 満足させる (影響度高・関心度低) | 定期レポート、要点説明 |
| 情報共有 (影響度低・関心度高) | 進捗メール、デモ招待 |
| 監視 (影響度低・関心度低) | 重大変更時のみ通知 |

#### データ分析基盤での典型ステークホルダー

| ステークホルダー | 影響度 | 関心度 | 象限 |
|---------------|-------|-------|------|
| 経営層 (役員) | 高 | 中 | 満足させる〜密接に管理 |
| 業務部門長 (マーケ, 営業) | 高 | 高 | 密接に管理 |
| 業務担当者 | 中 | 高 | 情報共有 |
| 情シス部長 | 高 | 高 | 密接に管理 |
| セキュリティ部門 | 高 | 中 | 満足させる |
| 法務部門 | 高 | 低 | 満足させる |
| 経理部門 (予算承認) | 高 | 低 | 満足させる |
| ソース系運用チーム | 中 | 中 | 情報共有 |
| データ分析基盤ベンダー | 中 | 高 | 情報共有 |
| エンドユーザー (店舗等) | 低 | 高 | 情報共有 |

### C. Opportunity Solution Tree (OST)

Teresa Torres の手法。目標 → 機会 → 解決策 → 実験 の 4 階層で構造化:

```
[Desired Outcome 目標]
    │
    ├── [Opportunity 1 機会]
    │       ├── [Solution 1-A 解決策]
    │       │       └── [Experiment 1-A-1 実験]
    │       └── [Solution 1-B]
    │               └── [Experiment 1-B-1]
    └── [Opportunity 2]
            └── [Solution 2-A]
                    └── [Experiment 2-A-1]
```

#### データ分析基盤での OST 例

```
目標: 経営会議の意思決定速度を 2 倍にする

├── 機会 A: 月次KPIの手動集計に 2 日かかる
│       ├── 解決策 A-1: BIダッシュボードで自動集計
│       │       └── 実験: PoC でダミーデータ表示
│       └── 解決策 A-2: 週次配信メールで代替
│               └── 実験: 1ヶ月間既存データで試験配信
│
├── 機会 B: 担当者により定義が異なる
│       ├── 解決策 B-1: Semantic Layer で定義統一
│       │       └── 実験: dbt Metrics で主要5指標を定義
│       └── 解決策 B-2: KPI辞書の整備
│               └── 実験: 簡易版を Notion で運用
│
└── 機会 C: 会議資料の準備が前夜から朝まで
        └── 解決策 C-1: ライブダッシュボード
                └── 実験: 役員向けに Power BI 導入
```

### D. 複数視点ブレスト

同じテーマを、**異なる役割の仮想ブレイン**で検討する。

推奨視点:

- **PM 視点**: スコープ、期限、予算、ステークホルダー管理
- **エンジニア視点**: 実装難度、技術的負債、保守性
- **ユーザー視点**: 使いやすさ、業務への適合、学習コスト
- **経営視点**: ROI、機会損失、競合優位性
- **運用視点**: 監視、障害対応、コスト
- **セキュリティ視点**: リスク、コンプライアンス

各視点で「これで良いか?」「懸念は?」「代替案は?」を問う。

#### 実施例: 「BIツールとして Power BI を選定」について

| 視点 | 観点 |
|-----|------|
| PM | ライセンス交渉の見通し、既存契約との整合性 |
| エンジニア | Databricks との連携、Semantic Layer 統合の困難度 |
| ユーザー | Excel に慣れた業務部門のオンボード時間 |
| 経営 | 競合他社の採用状況、ROI |
| 運用 | パッチ更新頻度、ダウンタイム実績 |
| セキュリティ | RLS の実装難度、監査ログの粒度 |

## 推奨実施フロー (半日〜1 日セッション)

```
09:00-10:00  JTBD: 主要ユーザーごとにジョブを洗い出し (付箋ブレスト)
10:00-11:00  ステークホルダーマップ: 影響度×関心度で分類
11:00-12:00  Opportunity Solution Tree: 目標 → 機会 までブレスト
  (昼休み)
13:00-14:30  OST: 解決策と実験を掘り下げ
14:30-16:00  複数視点ブレスト: 主要解決策に対して6視点評価
16:00-17:00  合意形成: Top 3 解決策、検証優先順位、次アクション
```

## ディスカバリーで引き出すべき情報

以下は次レイヤーの入力として必要:

- [ ] 主要ジョブ 3〜5 個
- [ ] キーステークホルダー 10〜20 人 + 象限分類
- [ ] 優先解決策 Top 3
- [ ] 各解決策の検証実験
- [ ] 業務部門が持っている前提 (明示化されていないもの)
- [ ] 技術的な制約 (既存システム、スキル、契約)
- [ ] 予算・期限の確度

## よくある失敗

### 失敗 1: 「何を作るか」から始める

- NG: 「BIツール入れましょう」から始まる会話
- OK: 「何のジョブを解決するのか」→ そのためにどんな仮説があるか → 手段の候補

### 失敗 2: ステークホルダーを技術的役割のみで整理

- NG: 「開発者」「運用者」だけ
- OK: 経営層、業務部門、法務、経理、外部ベンダーまで含める

### 失敗 3: 解決策をすぐ決めすぎる

- NG: OST で Solution を 1 つしか出さない
- OK: 各機会に最低 2〜3 の解決策、それぞれに実験

### 失敗 4: 検証計画がない

- NG: 「JTBD に合意しました、以上」
- OK: 「この JTBD は矢羽①の業務部門ヒアリングで検証する」を明示
