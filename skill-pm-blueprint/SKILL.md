# pm-blueprint: プロジェクト開始前青写真作成スキル

## このスキルの目的

プロジェクト開始前に、限られた情報から仮説ベースで以下を描ききる 9 階層統合スキル:

1. 経営層視点での投資判断
2. ユースケースと前提の仮説化
3. 最適アーキテクチャの構想
4. 機能・非機能要件への落とし込み
5. 各要素のリスク洗い出しと緩和策
6. 矢羽パターンのハイブリッド型 WBS への展開
7. 法務・コンプライアンス対応マトリクス
8. LLM ガバナンス（PII 境界 / 出力検証 / 評価セット）
9. 運用設計（監査証跡 / シャドー AI / ランブック）

## いつ使うか

- RFPを受領した直後
- 新規プロジェクトの提案作成時
- 既存プロジェクトの仕切り直し
- 予算・スコープ確定前のフィジビリティスタディ
- キックオフ前の青写真詳細化

## 推奨フロー

起点は `custom/統合オーケストレーター.md`。
この1本のオーケストレーターが Layer 2 → 3 → 4 → 5 → 1 → 6 の順で各レイヤーを呼び出し、最終的に12セクション構成のプロジェクト計画書を生成する。

## 各レイヤーの役割

| レイヤー | 役割 | 主要スキル | 成果物 |
|---|---|---|---|
| 1. 経営 | 投資判断・ピボット判断・撤退判断 | 意思決定テンプレート | Go/No-Go 判断資料 |
| 2. 仮説 | 前提抽出・ディスカバリー・ロードマップ | 4 軸前提分析 (V/U/Vi/F) | 前提リスト、JTBD、ロードマップ |
| 3. アーキ | 構造化・決定記録 | ADR / C4 / DDD / データ設計 / 環境分離 IaC | ADR、C4 図、コンテキストマップ、ERD、IaC スケルトン |
| 4. 要件 | 機能・非機能の具体化 | ユースケース / SMART-NFR / EARS / SLI/SLO / RTO/RPO | ユースケース、NFR ランディングゾーン、EARS 要件 |
| 5. リスク | 事前リスク洗い出しと Kill 基準 | 事前検死 / リスクレジスタ / 脅威モデリング / LINDDUN / 認可方式 / 暗号化-KMS / ゼロトラスト | リスクレジスタ YAML、Kill 基準、STRIDE/LINDDUN マップ |

| 7. 法務 | 法務・コンプラ対応 | 個情法対応マトリクス / 業法該当性判定書 / OSS ライセンス管理 / 越境データ移転書 / AI 生成物著作権 | 法務 8 文書 |

| 9. 運用 | 運用設計 | AI データ境界ガイド / 監査証跡設計 / シャドー AI 禁止 / ランブック規約 | 運用設計文書群 |

## 対象規模

- **年間予算**: 1億円〜1.5億円
- **チーム規模**: 3〜8名
- **期間**: 6〜18ヶ月
- **開発手法**: ハイブリッド (全体ウォーターフォール + 開発フェーズ並列アジャイルSprint)

## 矢羽パターン標準構成 (データ分析基盤)

```
矢羽① 業務要件定義     ──┐ (直列)
矢羽② PoC+システム要件   │→ 矢羽③ データ連携 Sprint       ┐ (並列)
                         │→ 矢羽④ AI分析・dbt Sprint       │
                         │→ 矢羽⑤ 検索・BI Sprint          │→ 矢羽⑦ 総合テスト
矢羽⑥ 非機能・運用 ───────────────────────────────────────── (横断)
```

## 注意事項

- このスキルは「机上で青写真を描く」ことが目的。実装フェーズのスクラム運用は別スキルで補完する
- 仮説は仮説であることを明示する (未確定事項には `[仮定]` タグを付ける)
- 年間1億円規模を想定しているため、より大規模な案件では各レイヤーの重厚さを追加する必要がある
- 矢羽パターンは本スキルのデフォルトだが、純粋ウォーターフォールや純粋アジャイルにも適用可能
- 日本の組織文化への配慮は `custom/日本型PMコンテキスト.md` を参照

## 初回利用ガイド

1. `README.md` を読む
2. `custom/統合オーケストレーター.md` を開く
3. 「RFP・要件概要」をユーザーから受け取る
4. オーケストレーターの7ステップを順次実行
5. 最終プロジェクト計画書を生成
