# エージェント権限境界書

## 対応する品質評価ID

- **LLM-AGENCY-001** (Fail → Pass): エージェント権限の天井が文書化されておらず、Human承認フローも未定義
- **OWASP LLM06** (Excessive Agency): LLM が自律的に過剰な操作権限を持つ脅威
- **GOV-AI-DATAFLOW-001** (Conditional → Pass): AI データフロー全体の権限ガバナンス

## 概要

LLM/AI エージェント (Claude Code、Copilot、AutoGen、LangGraph、独自エージェント) は、**自律的に道具 (tools) を呼び出し、業務システムを操作する**。RPA や従来のバッチ処理と異なり、LLM は判断ロジックがブラックボックスであり、想定外の操作を実行する可能性がある。

そこで、エージェントごとに以下を**事前**に文書化し、運用時に強制する:

1. **権限スコープ** : 読み取り専用 / DML (INSERT/UPDATE/DELETE) / DDL (CREATE/DROP)
2. **対象リソース** : どのテーブル/API/ファイル/コマンドを触れてよいか
3. **ハイリスク操作の Human承認フロー** : 承認者・SLA・記録方式
4. **Sandbox 実行環境** : 失敗時の影響範囲を物理的に隔離
5. **Just-In-Time (JIT) 権限** : 必要時のみ短時間で権限を付与

## いつ使うか

- LLM が業務システム (DB/API/メール/CRM) を操作するユースケースを設計する直前
- AI エージェントを本番環境にデプロイする前
- 既存エージェントの権限を見直すとき (定期レビュー)
- インシデント発生後の権限再設計
- 監査対応 (ISO/IEC 42001、SOC2、内部統制)

## 手順

### ステップ 1: エージェントのインベントリ作成

プロジェクト内の全エージェントを洗い出す。

| エージェント ID | 名称 | 主要ユースケース | 利用 LLM |
|---|---|---|---|
| AGT-001 | Voice Bot | UC-003 ボイスボット (顧客電話応対) | Azure OpenAI GPT-4o |
| AGT-002 | Inquiry Classifier | UC-008 LLM 分類 (問合せ自動カテゴライズ) | Snowflake Cortex Llama 3 |
| AGT-003 | Text2SQL Agent | UC-011 自然言語 → SQL 変換 | Claude Sonnet (Bedrock) |
| AGT-004 | Report Generator | UC-012 経営層向け報告書生成 | Azure OpenAI GPT-4 |
| AGT-005 | Loyal Policy FAQ | UC-013 FAQ ボット | Azure OpenAI GPT-4o-mini |

### ステップ 2: 操作カテゴリの分類

各エージェントが行う操作を 5 段階に分類する:

| 段階 | 操作カテゴリ | 例 |
|---|---|---|
| L0 | Read-only (公開情報) | FAQ 検索、製品カタログ参照 |
| L1 | Read-only (機微情報) | 顧客プロファイル参照、購買履歴閲覧 |
| L2 | DML 軽微 | チケットメモ追記、タグ付け |
| L3 | DML 重要 | 顧客情報更新、注文ステータス変更 |
| L4 | DDL / 破壊的 | テーブル作成・削除、本番デプロイ、メール送信 |

### ステップ 3: 権限境界書の YAML 化

各エージェントごとに以下を記述する。

```yaml
# 例: AGT-003 Text2SQL Agent
agent_id: AGT-003
name: Text2SQL Agent
use_case: UC-011 (自然言語 → SQL 変換 → BI 表示)
llm: claude-sonnet-4-5-bedrock
max_token_per_call: 8000
permissions:
  data_access:
    read_only:
      - schema: analytics_gold.*  # Gold層 集計済みのみ
      - schema: marketing_silver.cohort_summary
    forbidden:
      - schema: bronze.*    # 生 PII を含む
      - schema: silver.pii_*
  database_operations:
    allowed: [SELECT]
    forbidden: [INSERT, UPDATE, DELETE, MERGE, CREATE, DROP, ALTER, TRUNCATE]
  external_api:
    allowed: []
    forbidden: [HTTP_OUT_*]
  filesystem:
    allowed: [/tmp/agent-workspace]
    forbidden: [/, /etc, /home, /var]
high_risk_operations:
  - operation: SELECT_PII_COLUMN  # PII カラムへのアクセス
    requires_approval: true
    approver: [data_steward]
    sla_minutes: 30
sandbox:
  type: snowflake_role_isolated
  role: AGENT_TEXT2SQL_ROLE
  warehouse: AGENT_WH_XSMALL
  network_policy: deny_all_egress
just_in_time:
  enabled: true
  default_duration: 0    # 通常は権限なし
  request_workflow: ServiceNow_AccessRequest
audit:
  log_destination: snowflake.audit_db.agent_logs
  retention_days: 365
  pii_redaction: true
kill_switch:
  trigger: |
    異常検知 (権限外 SQL 試行 / 同一ユーザー 100req/分超 / DDL 試行)
  action: agent_disable_immediately
```

### ステップ 4: ハイリスク操作の Human承認フロー設計

L3 以上の操作は **必ず** Human承認を経由する。Slack / Teams / ServiceNow 等のワークフローを使う。

```yaml
approval_flow:
  trigger_operations: [DML_重要, DDL, 外部送信, 大量データ移動]
  approval_method: slack_modal  # またはServiceNow チケット
  approvers:
    primary: [tech_lead]
    fallback: [data_steward, security_officer]
  approval_timeout: 60_minutes
  on_timeout: reject  # 自動承認は禁止
  audit:
    record_request: true
    record_decision: true
    record_executor: true
    immutable_log: true  # WORM ストレージ
```

**重要:** タイムアウト時の自動承認 (auto-approve) は厳禁。LLM が「承認待ち中」を装って実行を試みるパターンを Layer 5 R-AI-12 と組み合わせて検知する。

### ステップ 5: Sandbox 実行環境の選定

| Sandbox 種別 | 用途 | 例 |
|---|---|---|
| Container Sandbox | コード実行 (Text2SQL の SQL 検証等) | Docker / gVisor / Firecracker |
| DB Role Sandbox | DB クエリ | Snowflake Role 分離、Postgres ROLE |
| Network Sandbox | API 呼び出し | egress NAT で許可 URL のみ通過 |
| Filesystem Sandbox | ファイル R/W | bind mount / tmpfs / chroot |
| Tool-call Sandbox | エージェントの tool 呼び出し | Claude Agent SDK の `permission_mode=ask` |

### ステップ 6: Just-In-Time (JIT) 権限の運用

通常は権限ゼロ、必要時に**最小限・短時間**で付与する。

```yaml
# 例: 月次レポート生成時のみ Gold 層書き込み権限を付与
jit_policy:
  agent_id: AGT-004
  default_role: NO_ACCESS
  elevation_request:
    role: REPORT_WRITER_ROLE
    scope: [analytics_gold.monthly_reports]
    duration_minutes: 30
    requestor: AGT-004_service_principal
    approver: data_steward
    approval_record: ServiceNow_CR_xxxxx
  auto_revoke: true
```

### ステップ 7: 監査ログとキルスイッチ

全エージェント呼び出しを **改ざん不能** な監査ログに記録する。

```yaml
audit_log_schema:
  - timestamp: ISO8601
  - agent_id: string
  - session_id: uuid
  - user_id: string  # 起動ユーザー (人間 or 上位エージェント)
  - operation: enum [READ, DML, DDL, EXTERNAL]
  - target_resource: string
  - input_prompt_hash: sha256  # 全文ではなくハッシュ (PII保護)
  - output_summary: string  # 50 文字程度
  - permission_check: enum [PASS, FAIL, ELEVATED]
  - approver: string  # Human承認時
  - approval_record: string

kill_switch:
  triggers:
    - permission_check == FAIL の連続発生 (5 回 / 5 分)
    - 異常な tool 呼び出し回数 (100 回 / 分超)
    - DDL or 外部送信の試行
    - PII 含有レスポンスの大量出力
  actions:
    - immediate: agent_endpoint disable
    - notify: [security_officer, tech_lead]
    - investigation: 24h以内に根本原因分析
```

## 例示: コールセンター UC-003 ボイスボット

### 権限境界書 YAML

```yaml
agent_id: AGT-001
name: Voice Bot
use_case: UC-003 (顧客電話応対 / 一次受付)
llm: azure-openai-gpt4o
permissions:
  data_access:
    read_only:
      - resource: kb.faq_articles  # ナレッジベース
      - resource: customer.profile_summary  # マスキング済プロファイル
    forbidden:
      - resource: customer.full_pii  # フル PII
      - resource: payment.*          # 決済情報
  business_operations:
    allowed:
      - create_inquiry_ticket  # チケット起票のみ可
      - update_inquiry_status_to: [received, escalated]
    forbidden:
      - close_inquiry         # 完了ステータスは人間のみ
      - issue_refund          # 返金は絶対禁止
      - send_external_email   # 外部メール送信禁止
  human_handoff:
    trigger:
      - 顧客が「人と話したい」と発話
      - LLM 信頼度スコア < 0.6
      - PII 含有応答が必要なシナリオ
    target: human_operator_queue
high_risk_operations:
  - operation: 顧客住所変更
    requires_approval: true
    approver: human_operator
    note: ボットでは絶対に実行しない (情報案内のみ)
sandbox:
  type: azure_apim_aigateway
  egress: deny_all_except_kb
audit:
  retention_days: 1825  # 5 年保持 (個人情報保護法対応)
  pii_redaction: true   # 録音テキストの PII を redaction 後に保存
kill_switch:
  trigger: 顧客クレーム閾値超過 / ハルシネーション検出器スコア > 0.3
  action: ボット停止 → 全件人間転送
```

### Read-only vs DDL/DML の段階的制限例 (UC-011 Text2SQL)

| 操作種別 | 許可 | 備考 |
|---|---|---|
| SELECT (Gold 層) | OK | Row-level Security 経由のみ |
| SELECT (Silver 層) | NG | 機微情報直接参照禁止 |
| SELECT (Bronze 層) | NG | 生 PII 含有のため絶対禁止 |
| INSERT | NG | 全 DML 禁止 |
| UPDATE | NG | 全 DML 禁止 |
| DELETE | NG | 全 DML 禁止 |
| CREATE TABLE | NG | DDL 完全禁止 |
| DROP TABLE | NG | DDL 完全禁止 |
| GRANT/REVOKE | NG | 権限操作禁止 |
| EXPLAIN | OK | クエリプラン参照のみ |

## 出力フォーマット

権限境界書は以下 3 形式で管理する:

1. **YAML マスター** (`agents/agent-permissions.yaml`) : Git 管理・PR レビュー必須
2. **IaC への注入** (Terraform / Bicep / Snowflake Role) : 上記 YAML から生成
3. **ランタイム検証** (API Gateway ポリシー、DB ROLE) : YAML と IaC が一致することを CI で検証

## 検証方法

- [ ] 全エージェントが本書に記載されているか (インベントリ抜け漏れチェック)
- [ ] YAML が Git 管理されレビュー履歴があるか
- [ ] IaC とランタイム ROLE が一致するか (drift detection)
- [ ] L3 以上操作の Human承認フローが Slack/Teams/ServiceNow で動作確認済か
- [ ] Sandbox の egress 制限が ON か
- [ ] 監査ログの WORM 保存が有効か
- [ ] Kill switch のトリガーがテスト環境で発火するか (机上ではなく実テスト)
