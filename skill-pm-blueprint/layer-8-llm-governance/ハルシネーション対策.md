# ハルシネーション対策

## 対応する品質評価ID

- **LLM-HALLUCIN-001** (Conditional → Pass): ハルシネーション検出と封じ込め体制が不十分
- **OWASP LLM09** (Misinformation): LLM が誤情報を拡散する脅威
- **GOV-AI-DATAFLOW-001** (Conditional → Pass): 業務情報の正確性ガバナンス

## 概要

ハルシネーション (Hallucination) とは、LLM が**事実に基づかない情報を確信を持って生成する**現象。コールセンター・データ分析基盤・経営報告書のような**正確性が求められる業務**では、ハルシネーションを 0% にすることは現実的でない代わりに、以下の組み合わせで**封じ込める**:

1. **引用必須化** : 全ての事実言明に出典を付与
2. **RAG retrieved-context 利用** : 「自分の知識から答えるな、与えられた文書のみから答えよ」
3. **出典検証層** : 引用された出典が実在し、内容が一致するかを後処理で検証
4. **信頼度スコア** : LLM の自己申告 + 検出器スコアで「この回答は不確実」を可視化
5. **検出率閾値** : 評価セットでハルシネーション率 X% 以下を品質ゲートに設定

## いつ使うか

- LLM が事実情報を提供する全ユースケース (FAQ、報告書、説明文)
- RAG ベースの問合せ応答システム
- 経営層向けレポート生成 (UC-012)
- 顧客向け FAQ ボット (UC-013)
- ボイスボット (UC-003) で会社規定や手続きを案内する場合
- 監査対応・法務応答に LLM を使う場合 (絶対に避けるべきだが必要なら厳格適用)

## 手順

### ステップ 1: ハルシネーション類型の分類

ハルシネーションには複数のタイプがある。

| 類型 | 説明 | 例 |
|---|---|---|
| 事実捏造 (Fabrication) | 存在しない事実を生成 | 「新規キャンペーン X (架空) があります」 |
| 出典捏造 (Source Fabrication) | 存在しない出典を引用 | 「弊社規定 第 XX 条によると...」(条文不在) |
| 文脈逸脱 (Off-context) | 質問と無関係な内容で長く答える | 「キャンセル方法」を聞かれて配送方針を答える |
| 過剰確信 (Overconfidence) | 不確実な内容を断定 | 「100% 返金されます」(規定上は条件付) |
| 数値誤り (Numerical Error) | 計算/数値の取り違え | 売上を 100 倍誤記 |
| 時間軸誤認 (Temporal Drift) | 古い情報を最新と誤認 | 「現在の代表取締役は前社長」 |

### ステップ 2: 引用必須化のシステムプロンプト設計

```
あなたは弊社の業務サポート AI です。回答は以下のルールに**完全に**従うこと:

1. すべての事実言明には出典を `[出典: 文書ID, セクション]` 形式で付与する
2. 与えられた文書 (retrieved_context) に記載されていない事項には「該当情報がありません」と答える
3. 推測・一般論・自分の知識からの補完を**絶対に**しない
4. 数値・日付・固有名詞は文書から逐語引用する
5. 不確実な場合は「確認が必要です」と明示し、人間オペレータへエスカレーションする

<retrieved_context>
{{rag_documents}}
</retrieved_context>

ユーザーの質問: {{user_query}}
```

### ステップ 3: 出典検証層 (Post-hoc Verification)

LLM 出力を後処理で検証する。

```python
def verify_citations(llm_output: str, retrieved_docs: list[Document]) -> VerificationResult:
    citations = extract_citations(llm_output)  # [出典: DOC-001, セクション 2] を抽出
    issues = []

    for citation in citations:
        # 1. 出典が存在するか
        doc = find_doc_by_id(citation.doc_id, retrieved_docs)
        if not doc:
            issues.append(f"FABRICATED_SOURCE: {citation.doc_id} 不在")
            continue

        # 2. 引用内容が文書に存在するか
        cited_text = extract_cited_text(llm_output, citation)
        similarity = compute_semantic_similarity(cited_text, doc.section_text)
        if similarity < 0.7:
            issues.append(f"MISMATCHED_CONTENT: {citation.doc_id} で類似度 {similarity:.2f}")

        # 3. 数値・日付の逐語一致
        numbers_in_output = extract_numbers(cited_text)
        numbers_in_doc = extract_numbers(doc.section_text)
        if not all(n in numbers_in_doc for n in numbers_in_output):
            issues.append(f"NUMBER_DRIFT: {citation.doc_id}")

    return VerificationResult(issues=issues, pass_rate=1 - len(issues) / max(len(citations), 1))
```

### ステップ 4: 検出率閾値 (品質ゲート)

評価セットを使って自動測定する。

| メトリクス | 最低 (Pass) | 推奨 |
|---|---|---|
| 出典付与率 | ≥ 95% | ≥ 99% |
| 出典実在率 | ≥ 99% | 100% |
| 引用内容一致率 (semantic similarity ≥ 0.7) | ≥ 90% | ≥ 95% |
| 数値一致率 (逐語) | ≥ 99% | 100% |
| 「該当なし」適切応答率 | ≥ 90% | ≥ 95% |
| 過剰確信率 (推測を断定する率) | ≤ 5% | ≤ 1% |

**Conditional → Pass の閾値**: 全メトリクス最低ライン充足。出典実在率と数値一致率は本番ブロック条件。

### ステップ 5: ハルシネーション検出器の選定

| 検出器 | 仕組み | 特徴 |
|---|---|---|
| Self-CheckGPT | 同一クエリを複数回投げ一貫性検証 | OSS、簡易、誤検出多 |
| RAGAS Faithfulness | retrieved_context に対する忠実度 | RAG 専用、定量化容易 |
| Lynx (Patronus AI) | LLM-judge 型 | 高精度、商用 |
| TruLens Groundedness | OSS、Faithfulness 計測 | LangChain と統合 |
| Azure AI Evaluation | Azure 統合、Groundedness/Relevance | Foundry 連携 |

検出器を**冗長**に組み合わせる (例: RAGAS + LLM-judge) ことで誤検知/見逃しを減らす。

### ステップ 6: 信頼度スコアの提示

エンドユーザーに「この回答はどれくらい信頼できるか」を示す。

```yaml
response:
  answer: "返金は購入後 30 日以内に申請いただけます [出典: REFUND-POLICY-2026, 第 3 条]"
  confidence_score: 0.92
  confidence_breakdown:
    citation_present: 1.0
    citation_verified: 1.0
    semantic_match: 0.85
    numerical_consistency: 1.0
  warning: null
  fallback_to_human: false

# 低信頼度の例
response:
  answer: "確認が必要です。担当オペレータへお繋ぎします。"
  confidence_score: 0.42
  warning: 関連文書の信頼度低
  fallback_to_human: true
```

UI 上では信頼度 < 0.7 の場合に「この回答は要確認です」を表示し、オペレータ転送ボタンを出す。

### ステップ 7: フォールバック設計

ハルシネーション疑い検出時の対応:

```
信頼度 ≥ 0.8           → そのまま回答
信頼度 0.5〜0.8         → 「確認が必要です」を付記、ログに記録
信頼度 < 0.5           → ボット回答停止 → 人間オペレータへ転送
出典実在率 < 100%       → 即時 STOP、回答せず人間転送
数値一致率 < 100%       → 即時 STOP (UC-012 報告書では特に厳格)
```

## 例示: UC-012 経営層向け報告書生成

### 課題

経営層レポートには「2026 年 Q1 売上は 12.3 億円、前年同期比 +5.7%」のような数値が並ぶ。LLM が数値を誤ると経営判断が歪む。

### 防御フロー

```
1. SQL で確定値を取得 → JSON ヘッダで LLM に渡す
   { "q1_revenue": 1230000000, "yoy_growth": 0.057 }

2. システムプロンプトで「JSON 内の数値以外は出力禁止」を明示

3. 出力後、抽出した数値を JSON ソースと逐語比較
   - 完全一致 → OK
   - 不一致 → BLOCK、人間レビュー

4. 解釈文章は別 LLM が生成し、retrieved_context で「経営計画書」を参照
   - Faithfulness スコア < 0.85 → BLOCK
```

### 出力例 (検証済)

```markdown
# 2026 Q1 業績レポート

## サマリ
2026 年 Q1 売上は **12.3 億円** [出典: financials_q1.json] となり、前年同期比 +5.7%
[出典: financials_q1.json] の伸長を達成しました。

これは経営計画書 (2025-12 版) で示された通期目標 +6.0% [出典: BIZPLAN-2026, 第 2 章] に
ほぼ沿った進捗です。

## 信頼度
- 数値出典率: 100% (financials_q1.json と一致)
- 解釈文 Faithfulness: 0.91 (BIZPLAN-2026 と整合)
```

## 例示: UC-013 ロイヤル方針 FAQ

### システムプロンプト (引用必須化)

```
あなたはロイヤルカスタマー向け FAQ ボットです。

【絶対ルール】
- 質問に対し、与えられた FAQ 文書 (retrieved_context) のみから回答する
- 一般論・推測・他社事例での補完は禁止
- 該当 FAQ がない場合は "申し訳ございませんが、該当する案内がございません。
  オペレータへお繋ぎします" と返答
- 回答末尾に [FAQ-ID] を必ず記載
```

### 検証ステップ

```
INPUT: "ロイヤル特典のポイント有効期限は?"
RAG 取得: [FAQ-RYL-007: ポイント有効期限は最終獲得日から 24 ヶ月]
LLM 出力: "ポイント有効期限は最終獲得日から 24 ヶ月です [FAQ-RYL-007]"

検証:
- 出典 ID 付与: OK
- 出典実在: OK (FAQ-RYL-007 は文書セットに存在)
- semantic similarity: 0.97
- 数値一致: 24 ヶ月 → 24 ヶ月 OK
信頼度スコア: 0.96 → そのまま回答
```

## 検証方法

- [ ] 全ユースケースで引用必須化のシステムプロンプトが導入されているか
- [ ] retrieved_context の trust_level マーキング (high/medium/low) があるか
- [ ] 出典検証コードが本番デプロイされているか
- [ ] 評価セット (50〜200 件) が整備されているか
- [ ] 数値一致率・出典実在率が品質ゲート閾値を満たすか
- [ ] 信頼度スコア < 0.7 で人間転送する UI があるか
- [ ] 監査ログにハルシネーション検出件数が日次集計されているか
