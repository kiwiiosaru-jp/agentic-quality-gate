# モデルドリフト検知

## 対応する品質評価ID

- **LLM-DRIFT-001** (Conditional → Pass): モデル更新時の品質ドリフト検知体制が不十分
- **OWASP LLM03** (Supply Chain): ベンダー側変更によるモデルサプライチェーンリスク
- **GOV-AI-DATAFLOW-001** (Conditional → Pass): AI データフロー全体の品質モニタリング

## 概要

LLM はベンダー (Azure OpenAI, Anthropic, Snowflake Cortex 等) が**暗黙的に**モデルを更新することがある。たとえ「同じ `gpt-4o` を呼び出しても、内部の重みやトレーニングデータが入れ替わり**回答品質が変化**することがある。これを「モデルドリフト」と呼ぶ。

`custom/AI駆動開発リスク.md` R-AI-07 (モデル変更による再現性喪失) のチェックリストレベルでは「気づいたら回帰テスト」程度の対応だが、本書では:

1. **固定メトリクスの定常測定** : 同じ評価セットを毎日叩き、スコア時系列を保存
2. **トレンドダッシュボード** : Grafana / Power BI で日次・週次の劣化を可視化
3. **A/B 比較** : 旧バージョン (固定スナップショット) と最新バージョンを並行測定
4. **自動アラート** : 閾値超過で即時通知 → 評価セット拡充判断
5. **ロールバック手順** : 旧バージョンへ戻す経路の事前確保

## いつ使うか

- LLM をモデルバージョン非固定で本番運用している全ユースケース
- ベンダー側の auto-update 設定を OFF にできない場合
- LLM 出力の品質が業務に直結するケース (UC-008 LLM 分類、UC-013 FAQ ボット)
- ユーザーから「最近回答が変わった」フィードバックを受けた時
- ベンダーがモデルアップデートを announce した時 (例: gpt-4o → gpt-4o-2026-XX)

## 手順

### ステップ 1: モデルバージョン管理戦略の決定

まず**バージョンを固定するか変動を許容するか**を決める。

| 戦略 | メリット | デメリット | 推奨ユースケース |
|---|---|---|---|
| 固定 (Pinned) | 再現性 100% | 自動改善を逃す、API 仕様変更で死ぬ | UC-012 経営報告書 |
| 最新追従 (Latest) | 自動改善受益 | ドリフトリスク高 | 内部利用ツール |
| ステージドリリース | 検証期間あり | 運用負荷 | UC-008/013 (本番品質要件高) |

`custom/AI駆動開発リスク.md` R-AI-07 ではバージョン固定を推奨するが、ベンダーによっては固定不可なため、**ドリフト検知が必須**。

### ステップ 2: 固定評価セットの整備

ドリフト測定用の**金標準 (Gold Standard)** 評価セットを 100〜500 件用意する。

```yaml
# evals/drift/gold-standard.yaml
description: "ドリフト検知用 固定評価セット (毎日同じ入力で測定)"
version: 1.0
created: 2026-04-30
locked: true  # 内容を勝手に変更しない
items:
  - id: DRIFT-CLAS-001
    use_case: UC-008
    input: "返金についての質問です。商品が届かないのですが..."
    expected_label: "billing_inquiry"
    expected_severity: "high"
  - id: DRIFT-CLAS-002
    use_case: UC-008
    input: "アプリの調子が悪くてログインできません"
    expected_label: "technical_issue"
    expected_severity: "medium"
  - id: DRIFT-FAQ-001
    use_case: UC-013
    input: "ロイヤル特典のポイント有効期限は?"
    expected_substring: "24ヶ月"
    expected_citation: "FAQ-RYL-007"
  - id: DRIFT-T2S-001
    use_case: UC-011
    input: "先月の売上トップ 10 顧客"
    expected_sql_pattern: |
      SELECT.*FROM analytics_gold.fact_sales.*
      WHERE sale_date.*
      GROUP BY.*
      ORDER BY.*DESC LIMIT 10
```

### ステップ 3: 測定メトリクスの定義

ユースケースごとに**主要メトリクス**を 3〜5 個定義する。

| ユースケース | 主要メトリクス | 閾値 |
|---|---|---|
| UC-008 LLM 分類 | accuracy, F1 (per class), severity_match_rate | accuracy ≥ 0.85 |
| UC-011 Text2SQL | exact_match, execution_correctness, semantic_match | execution ≥ 0.90 |
| UC-012 報告書生成 | numerical_accuracy, citation_correctness, faithfulness | numerical = 1.00 |
| UC-013 FAQ ボット | answer_correctness, citation_present, refusal_rate | correctness ≥ 0.92 |
| UC-003 ボイスボット | intent_accuracy, escalation_rate, hallucination_rate | hallucination ≤ 0.02 |

副次メトリクスも測定:

- レイテンシ (p50, p95, p99)
- トークン消費量 (input/output)
- エラー率 (timeout / 5xx)
- コスト (USD / 100 件)

### ステップ 4: 定常測定パイプライン

毎日 (または毎時) 同じ評価セットを叩く。

```yaml
# .github/workflows/daily-drift-check.yml
name: Daily LLM Drift Check
on:
  schedule:
    - cron: "0 1 * * *"  # 毎日 01:00 UTC
  workflow_dispatch:
jobs:
  drift_check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run gold-standard eval
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          python -m evals.drift.runner \
            --suite evals/drift/gold-standard.yaml \
            --providers azure_openai,snowflake_cortex,anthropic \
            --output reports/drift/$(date +%Y%m%d).json
      - name: Upload metrics to time-series DB
        run: |
          python -m evals.drift.uploader \
            --input reports/drift/$(date +%Y%m%d).json \
            --destination snowflake://drift_metrics
      - name: Trigger alert if degraded
        run: |
          python -m evals.drift.alert \
            --report reports/drift/$(date +%Y%m%d).json \
            --baseline reports/drift/baseline.json \
            --threshold 0.05  # 5% 劣化でアラート
```

### ステップ 5: トレンドダッシュボード設計

Grafana / Power BI / Snowsight で以下を可視化:

| パネル | 内容 |
|---|---|
| 時系列 accuracy | 過去 90 日の日次 accuracy 推移 |
| 主要メトリクス推移 | 各 UC のメトリクス推移 (折れ線) |
| ベースライン乖離 | 初回測定値からの相対変化率 |
| プロバイダ別比較 | OpenAI vs Anthropic vs Snowflake の同タスク比較 |
| エラーケース増減 | 失敗した評価項目の差分 (DRIFT-FAQ-001 が今日失敗) |
| トークンコスト | 日次・月次のコスト推移 |

```sql
-- ベースライン乖離の閾値超過アラート
SELECT
  metric_date,
  use_case,
  metric_name,
  metric_value,
  baseline_value,
  (metric_value - baseline_value) / baseline_value AS relative_change
FROM drift_metrics
WHERE metric_date >= DATEADD(day, -7, CURRENT_DATE())
  AND ABS(relative_change) > 0.05  -- 5% 以上変動
ORDER BY relative_change ASC;
```

### ステップ 6: A/B 比較 (バージョン pin vs latest)

旧モデル (固定) を「カナリア」として並走させ、最新モデルとの差分を測る。

```yaml
ab_test_config:
  baseline:
    model: gpt-4o-2024-11-20  # スナップショット版 (固定)
    purpose: 比較ベースライン
  candidate:
    model: gpt-4o  # ベンダーの最新
    purpose: 本番運用
  comparison:
    eval_set: evals/drift/gold-standard.yaml
    schedule: 毎日
    metrics: [accuracy, latency_p95, cost]
  alert_conditions:
    - metric: accuracy
      degradation_threshold: 0.03  # 3% 以上の劣化で警告
    - metric: latency_p95
      degradation_threshold: 0.20  # 20% 以上の遅延で警告
```

### ステップ 7: アラートとエスカレーション

```yaml
alert_policies:
  - id: DRIFT-CRITICAL
    condition: |
      日次 accuracy が 5% 以上劣化 AND 連続 3 日
    severity: P1
    notify: [tech_lead, product_owner, security_officer]
    action: 評価セット拡充 + ベンダー問合せ + ロールバック検討
  - id: DRIFT-WARNING
    condition: 日次 accuracy が 2% 以上劣化
    severity: P3
    notify: [ml_engineer]
    action: 監視強化 (頻度を 1h 毎へ)
  - id: COST-SPIKE
    condition: トークン消費量が前週比 30% 増
    severity: P2
    notify: [tech_lead, finance]
    action: プロンプト見直し / モデル変更検討
  - id: HALLUCINATION-INCREASE
    condition: ハルシネーション検知率が 1% → 3% へ増加
    severity: P1
    notify: [tech_lead, product_owner]
    action: ハルシネーション対策強化、本番一時停止検討
```

### ステップ 8: ロールバック手順

ドリフト発覚時に旧バージョンへ戻す経路を事前確立。

```yaml
rollback_procedure:
  trigger: DRIFT-CRITICAL アラート
  pre_conditions:
    - 旧モデルバージョン (例: gpt-4o-2024-11-20) が API で呼べる
    - feature flag で切替可能
  steps:
    1. feature flag `llm.model_version=stable` に切替 (15 分以内)
    2. 監視ダッシュボードで accuracy 復帰確認 (60 分)
    3. ベンダーへ問合せ送信
    4. 評価セット拡充 → 新モデルの fix 待ち
    5. fix 後の再評価 (新評価セット込) → 再 cutover
  rollback_time_objective: 30 分以内
  records:
    - rollback_id: uuid
    - reason: text
    - duration: minutes
    - business_impact: text
```

## 例示: UC-008 LLM 分類のドリフト

### シナリオ

問合せ自動分類 (10 カテゴリ) を Snowflake Cortex Llama 3 で運用中。
ある日、Snowflake が暗黙的に Cortex Llama 3 のサーバ重みを更新。

### 検知フロー

```
Day 1: accuracy = 0.91 (baseline)
Day 30: accuracy = 0.91
Day 60: accuracy = 0.90
Day 65: accuracy = 0.86  ← DRIFT-WARNING (-5%)
Day 66: accuracy = 0.85  ← DRIFT-CRITICAL (-6%)

→ 連続 3 日 DRIFT-WARNING で P1 アラート
→ 失敗ケース分析: "billing_inquiry" の F1 が 0.95 → 0.78 に低下
→ Snowflake サポートへ問合せ + Azure OpenAI へ並走切替検討
```

### ダッシュボード例 (Grafana)

```
[時系列パネル: accuracy by day]
  X 軸: 日付
  Y 軸: accuracy
  ライン: candidate (gpt-4o), baseline (pinned)
  注釈: ベンダー更新日にマーカー

[ヒートマップ: per-category F1]
  行: 10 カテゴリ
  列: 過去 30 日
  色: F1 値 (赤 = 低、緑 = 高)
  ドリフト発生時に赤いブロックが拡散
```

## 例示: UC-011 Text2SQL のドリフト

### シナリオ

Claude Sonnet (Bedrock) で Text2SQL。ある週末に Anthropic が学習データを更新。
SQL 構文の細かな書き方 (たとえば `DATEADD` 関数の引数順) が変化。

### 検知

```
exact_match: 0.85 → 0.82
execution_correctness: 0.94 → 0.95   ← 実行は通る
semantic_match: 0.92 → 0.90

→ exact_match のみ劣化、execution は維持
→ 「動作はするがクエリの書き方が変わった」と判定
→ DRIFT-WARNING、テスト基準 (exact_match → semantic_match へ移行)
```

## 検証方法

- [ ] 固定評価セットが 100 件以上整備されているか
- [ ] 毎日自動実行されているか (GitHub Actions / Azure DevOps)
- [ ] メトリクス時系列が DB に蓄積されているか
- [ ] ダッシュボードでベースライン乖離が見えるか
- [ ] A/B 比較 (固定 vs latest) が動いているか
- [ ] アラート閾値が業務影響に応じて妥当か
- [ ] ロールバック手順が実行可能か (机上ではなく訓練済か)
