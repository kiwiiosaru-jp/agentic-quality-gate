# EARS 形式要件 (Easy Approach to Requirements Syntax)

## 概要

Alistair Mavin らが 2009 年に提唱。自然言語要件にありがちな曖昧さを排除するため、
要件を **6 種類の構文テンプレート**のいずれかに当てはめて書く方法。

利点:

- 要件を読んだ人が必ず同じ解釈になる
- Given-When-Then 形式の受入基準に変換しやすい
- テストケースへの落とし込みが直接的
- ユースケースの補完として使える

## EARS の 6 パターン (日本語化)

### パターン 1: Ubiquitous (恒常的要件)

**構文:** 「システムは <機能> すること」

システムが常に満たすべき性質。トリガーや条件がない。

例:
- システムはすべての PII を保存時に暗号化すること
- システムは操作履歴を記録すること
- システムは UTC の時刻を基準とすること

### パターン 2: Event-driven (イベント駆動)

**構文:** 「<イベント> が発生した時、システムは <機能> すること」

特定のトリガー発生時のみ動作。

例:
- ユーザーがログインした時、システムはロールを判定すること
- 夜間バッチが完了した時、システムは完了通知を Slack に送信すること
- エラーが発生した時、システムはアラートをメール送信すること

### パターン 3: State-driven (状態駆動)

**構文:** 「<状態> の間、システムは <機能> すること」

特定の状態にある間だけ動作する。

例:
- 業務時間帯 (8:00-20:00 JST) の間、システムは BI ダッシュボードに応答すること
- メンテナンスモードの間、システムは利用者にメンテ画面を表示すること
- 認証トークンが有効な間、システムはセッションを維持すること

### パターン 4: Optional (オプション)

**構文:** 「<条件> の場合、システムは <機能> すること」

条件が成立したら動作する。条件が成立しなければ何もしない。

例:
- データ量が 1TB を超える場合、システムは日次バックアップに加えて週次のフルバックアップを実行すること
- ユーザーが閲覧権限を持つ場合、システムはドリルダウン機能を有効にすること
- プライベートネットワークからのアクセスの場合、システムは MFA をスキップすること

### パターン 5: Unwanted behavior (望ましくない振る舞いの抑止)

**構文:** 「<望ましくない条件> の場合、システムは <対応> すること」

エラー・異常ケースの対応を明示。

例:
- 認証失敗が 5 回連続した場合、システムはアカウントを 30 分ロックすること
- Gold 層へのクエリが 30 秒を超えた場合、システムはタイムアウトしてエラーを返すこと
- PII がクエリ結果に含まれる場合、システムは自動的にマスキングすること

### パターン 6: Complex (複合)

**構文:** 上記パターンの組み合わせ

例:
- 業務時間帯 (State) であり、かつユーザーが認証済み (Condition) の場合、
  ユーザーがクエリを発行した時 (Event)、
  システムは 3 秒以内に結果を返すこと (Function)

複合パターンは読みにくくなりがちなので、可能なら単純パターンに分割する。

## EARS テンプレートの使い分け指針

| パターン | いつ使うか |
|---------|-----------|
| Ubiquitous | システム全体に関わる性質 (暗号化, 監査) |
| Event-driven | 非同期処理、通知、ワークフロー |
| State-driven | 業務時間帯、メンテナンス、認証状態 |
| Optional | 条件付き機能、ロール依存の機能 |
| Unwanted behavior | エラー処理、セキュリティ対応 |
| Complex | 他パターンで書けない場合のみ |

## データ分析基盤での例示

### Ubiquitous の例 (各5個)

1. システムはすべての PII カラムを AES-256 で暗号化すること
2. システムは全利用者の操作履歴を 1 年間保持すること
3. システムは Gold 層への全クエリを Query History に記録すること
4. システムは JST タイムゾーンで日次バッチをスケジュールすること
5. システムは SSL/TLS 1.2 以上で通信を行うこと

### Event-driven の例 (各5個)

1. ユーザーがダッシュボードURLにアクセスした時、システムは Entra ID での認証を要求すること
2. dbt テストが失敗した時、システムは デプロイを中断して Slack に通知すること
3. バッチ処理時間が目標を超過した時、システムは 運用チームにメールで警告すること
4. 新規データソースが登録された時、システムは データカタログに自動登録すること
5. Gold 層のスキーマが変更された時、システムは 依存 BI レポートのオーナーに通知すること

### State-driven の例 (各5個)

1. 業務時間帯 (平日 8:00-20:00 JST) の間、システムは BIダッシュボードの可用性 99.5% を維持すること
2. 夜間バッチ実行中 (22:00-06:00 JST) の間、システムは BIダッシュボードの応答速度を優先度 2 に落とすこと
3. 認証トークン有効期間 (8 時間) の間、システムは 再認証を要求しないこと
4. メンテナンスウィンドウの間、システムは API 応答で HTTP 503 を返すこと
5. データインポート実行中の間、システムは Bronze 層への書込を他ジョブから排他すること

### Optional の例 (各5個)

1. ユーザーが Executive ロールを持つ場合、システムは 経営層限定ダッシュボードを表示すること
2. データ量が 100GB を超えるクエリの場合、システムは 非同期実行モードを提案すること
3. プライベートネットワーク (社内VPN) からのアクセスの場合、システムは MFA をスキップしてもよい
4. 営業日が月末の場合、システムは 月次集計バッチを実行すること
5. 利用者が請求先責任者として登録されている場合、システムは コスト分析ダッシュボードを表示すること

### Unwanted behavior の例 (各5個)

1. 認証失敗が 5 回連続した場合、システムは アカウントを 30 分間ロックすること
2. Gold 層クエリが 60 秒を超過した場合、システムは クエリをキャンセルしてタイムアウトエラーを返すこと
3. PII フィールドを含むデータがCSVダウンロードされる場合、システムは 該当カラムをマスキングして出力すること
4. LLM API のクォータ上限に達した場合、システムは 処理をキューに退避し、翌日再試行すること
5. dbt モデルのテスト (not_null, unique) が失敗した場合、システムは デプロイを中断し、前回バージョンを維持すること

## ユースケースとの使い分け

| 観点 | ユースケース | EARS |
|-----|------------|------|
| 粒度 | 業務フロー全体 (1 画面 or 1 プロセス) | 個別の要件 (1 文) |
| 形式 | シナリオ (ステップの順序) | テンプレート (構文) |
| 読者 | 業務部門とアーキ | 開発者とテスター |
| 用途 | 合意形成 | 実装・テストへの落とし込み |

推奨フロー:

1. ユースケースで業務フローを合意
2. 各ステップを EARS で具体的要件に分解
3. EARS 文を Given-When-Then (受入基準) に変換
4. テストケースへ

## Given-When-Then への変換

EARS Event-driven → Given-When-Then:

- EARS: 「ユーザーがダッシュボードURLにアクセスした時、システムはEntra ID認証を要求すること」
- Given-When-Then:
  - Given: 未認証ユーザー
  - When: ダッシュボードURLにアクセス
  - Then: Entra ID ログイン画面へリダイレクト

EARS Unwanted behavior → Given-When-Then:

- EARS: 「認証失敗が 5 回連続した場合、システムはアカウントを 30 分ロックすること」
- Given-When-Then:
  - Given: 認証失敗 4 回のユーザー
  - When: 5 回目の認証失敗
  - Then: アカウントが 30 分間ロックされる、かつアラートが SOC に送信される

## 作成時のチェックリスト

- [ ] 「速く」「使いやすく」等の曖昧語を使っていない
- [ ] 数値化可能な部分は数値化されている (NFRから数値を引く)
- [ ] パターンのいずれかに明確に当てはまる
- [ ] 主語は「システム」(アクターが主語になっていない)
- [ ] テスト可能 (受入基準に変換できる)
- [ ] Complex を使わざるを得ないか、単純パターンに分解できるか検討済み
