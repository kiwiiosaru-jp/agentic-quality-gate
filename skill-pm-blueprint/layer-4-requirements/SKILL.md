# Layer 4: 要件定義

## このレイヤーの目的

Layer 2 の仮説、Layer 3 のアーキテクチャを、**実装可能な粒度**の要件に落とし込む。
ただし、青写真段階では詳細設計書レベルまでは降りず、**開発チームが Sprint チケットに分解できる粒度**までで止める。

## 推奨実行順序

1. **ユースケース.md** - Use Case 2.0 slim 形式で機能要件の骨格を作る
2. **SMART非機能要件.md** - FURPS+ 観点で NFR を定量化 (ランディングゾーン)
3. **EARS形式要件.md** - ユースケースと NFR を Easy Approach to Requirements Syntax で文章化

## 機能要件と非機能要件の紐付けルール

**機能要件 1 つに対して最低 3 つの NFR を紐付ける** (本スキルのルール)。
典型的には: 性能 / 可用性 / セキュリティ が各機能に紐付く。

これにより、機能だけを追うのではなく、常に「いつまでに / どれくらいの精度で / どんな条件下で」を意識する要件が書ける。

## 入力 (前レイヤーから受け取るもの)

- Layer 2 のユースケース仮説一覧
- Layer 2 の前提 (Usability 軸)
- Layer 3 の ADR (実装可能性の確認材料)
- Layer 3 の C2 Container 図 (実行単位ごとの NFR 検討材料)

## 出力 (次レイヤーへ渡すもの)

- **機能要件一覧** (ユースケース形式 10〜30 件)
- **NFR ランディングゾーン** (カテゴリ別の最小/目標/卓越 3 水準)
- **EARS 要件文** (受入基準に転用可能な形式)
- **機能 × NFR マトリクス** (紐付け表)

これらは Layer 5 (リスク: 実装困難リスクの洗い出し), Layer 6 (WBS: 工数見積の入力) の入力となる。

## 年間1億円規模での軽量化方針

- ユースケースは主要 10〜20 件に絞る (全業務カバーしようとしない)
- NFR ランディングゾーンは 5〜7 カテゴリ (FURPS+の主要部分)
- EARS 文は、機能要件 1 つあたり 3〜5 文
- 網羅性より**主要な合意形成**を重視
