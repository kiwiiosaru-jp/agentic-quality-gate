# 可観測性 三本柱 (Logs / Metrics / Traces)

## 対応する品質ゲート

| ID | 観点 | 既定の Conditional 条件 |
|----|------|------|
| NFR-OBS-001 | 可観測性 | Log/Metric/Trace の三本柱、Trace ID 規約、保持期間、SaaS との統合戦略が宣言されていない |

## 概要

「監視」と「可観測性 (Observability)」は別概念:

- **Monitoring**: 既知の問題に対する事前定義のしきい値で検知
- **Observability**: 未知の問題でも「内部状態を外部から推論できる」性質

可観測性は **三本柱 (Three Pillars)** で実現する:

1. **Logs** : 個別イベントの記録 (構造化推奨)
2. **Metrics** : 集約された数値 (時系列)
3. **Traces** : リクエストの全行程 (Span のツリー)

そして 4 番目の柱として **Profiles / Continuous Profiling** が追加されつつある (CPU / Memory プロファイルの常時計測)。

本スキルでは OpenTelemetry を中心に、Genesys / Databricks / SAP / Azure Monitor を統合する戦略を設計する。

## いつ使うか

- 本番リリース前の運用設計
- 「障害発生時に原因がわからない」が頻発した時
- マイクロサービス化 / モジュール分割で系全体の追跡が困難になった時
- LLM / RAG パイプラインの推論レイテンシ・品質を追う時

## 手順 (5 ステップ)

### ① 三本柱を OpenTelemetry で統合する

#### 推奨: OpenTelemetry (OTel) 標準化

```
[アプリ]
   ├── OTel SDK (Python / TypeScript / .NET / Java)
   |
   ↓ OTLP (gRPC / HTTP)
[OTel Collector]
   ├── Batching, Sampling, Enrichment
   |
   ↓
[Backend (複数)]
   - Logs   → Azure Log Analytics Workspace
   - Metrics → Azure Monitor Metrics
   - Traces  → Azure Application Insights / Tempo / Jaeger
```

**鉄則:**

- アプリは OTel に書く。Backend は後で取り替えられる
- ベンダー固有 SDK (Application Insights, Datadog Agent) は使わない (OTel Collector 経由)
- Local Dev も OTel Collector 経由で Console 出力

### ② 構造化ログ規約 (Logs)

#### 必須フィールド

```json
{
  "timestamp": "2026-04-30T14:30:15.123+09:00",
  "level": "INFO",
  "logger": "inquiry_intake.handler",
  "message": "interaction created",

  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",
  "request_id": "req_2026043014301512345",

  "service.name": "inquiry-intake-api",
  "service.version": "1.4.2",
  "service.environment": "prod",
  "service.instance.id": "ca-app-prod-rev-7-bc4xyz",

  "user.id": "u_abc123",
  "tenant.id": "tenant_xyz",

  "interaction.id": "i_2026043012345",
  "channel": "voice",

  "duration_ms": 142,
  "http.status_code": 201,
  "http.method": "POST",
  "http.route": "/v1/interactions"
}
```

#### ログレベル運用ガイド

| Level | 用途 | サンプリング |
|------|-----|-----------|
| ERROR | エラー、対応必須 | 100% |
| WARN | 異常傾向、要注意 | 100% |
| INFO | 業務イベント | 100% (重要操作のみ) |
| DEBUG | 開発診断 | 0% (prod) / 100% (dev) |
| TRACE | 詳細追跡 | 0% (prod) / on-demand |

#### PII / 機密情報の取扱

```yaml
log_pii_policy:
  forbidden_in_logs:
    - 氏名、電話、メール、住所 (生)
    - クレジットカード番号
    - パスワード、API Key
    - 通話音声ファイルのバイナリ
  allowed_after_masking:
    - email → SHA-256 hash
    - phone → 下 4 桁のみ (XXXX-1234)
    - 顧客 ID → そのまま (内部 ID のみ)
  redaction_at_collector:
    - OTel Collector の `attributes/redact` プロセッサで再チェック
```

### ③ メトリクス設計 (Metrics)

#### 推奨メトリクス階層 (RED + USE)

**RED Method (リクエスト系):**

- **R**ate (req/sec)
- **E**rrors (失敗率)
- **D**uration (応答時間 P50/P95/P99)

**USE Method (リソース系):**

- **U**tilization (CPU/Memory 使用率)
- **S**aturation (キュー深さ、待ち時間)
- **E**rrors (resource error)

**Four Golden Signals (Google SRE):**

- Latency / Traffic / Errors / Saturation

#### Metric 命名規則 (OTel Semantic Convention)

```yaml
metric_naming:
  pattern: <namespace>.<entity>.<measurement>
  examples:
    - http.server.request.duration  (histogram, ms)
    - http.server.request.count     (counter)
    - inquiry.intake.processed.count (counter, attributes: status, channel)
    - llm.token.usage.count         (counter, attributes: model, prompt_type)
    - db.connection.pool.active     (gauge)
```

#### High Cardinality の注意

- ラベルに UUID / 顧客 ID を入れると爆発
- ラベル候補: status, env, region, channel, model_name (低 cardinality)
- ラベルに不可: trace_id, user_id, request_id (Log で扱う)

#### LLM / RAG パイプライン推奨メトリクス

```yaml
llm_metrics:
  - llm.request.latency_ms (histogram, attributes: model, prompt_type)
  - llm.token.input.count (counter)
  - llm.token.output.count (counter)
  - llm.cost.usd (counter, attributes: model)
  - llm.error.rate (counter, attributes: error_type)
  - rag.retrieval.latency_ms (histogram)
  - rag.retrieval.hit_rate (gauge, 0..1)
  - rag.context.relevance_score (gauge)
  - llm.eval.accuracy (gauge, 評価データセット由来)
```

### ④ 分散トレース (Traces) 設計

#### Trace / Span / Trace ID の伝播

```
Trace (1 リクエストの全行程)
└── Root Span (API Gateway)
    └── Span (auth check)
    └── Span (intake-api)
        └── Span (postgres write)
        └── Span (outbox publish)
            └── Span (eventhub send)
                └── Span (databricks ingest)  ← downstream まで貫通
```

#### Trace ID 規約 (W3C Trace Context)

```
HTTP Header:
  traceparent: 00-{trace_id}-{parent_span_id}-{flags}
  tracestate: vendor=value
```

- trace_id: 32 hex char (128bit)
- span_id: 16 hex char (64bit)
- 全サービスでこのヘッダを必ず伝播
- ログに `trace_id` を必ず埋める

#### サンプリング戦略

```yaml
sampling:
  ingress:
    rule: "Tail-based"
    rate:
      success_2xx: 1%
      error_5xx: 100%
      error_4xx: 10%
      slow_p99: 100%  (P99 超過は全 Trace 保存)
  collector_config:
    type: tail_sampling
    decision_wait: 30s
```

#### Genesys Webhook の Trace 連携

```yaml
genesys_webhook_trace:
  inbound:
    - Genesys Webhook → 自社 API
    - 自社で trace_id 生成、interaction.id を attributes に
  store_in_db:
    - INTERACTION テーブルに trace_id カラム
  downstream:
    - Postgres → Outbox → Event Hubs (trace_id を property に伝播)
    - Databricks Auto Loader (trace_id を保持して Bronze へ)
  query_pattern:
    - 「この通話 (interaction_id) の全処理を見たい」→ trace_id で全 backend を横断検索
```

#### Databricks Job の Trace

```python
# Databricks Notebook の擬似例
from opentelemetry import trace
tracer = trace.get_tracer("databricks.notebook")

with tracer.start_as_current_span("bronze_to_silver") as span:
    span.set_attribute("partition", target_partition)
    df = spark.read.format("delta").load(bronze_path)
    df_silver = transform(df)
    df_silver.write.format("delta").mode("append").save(silver_path)
    span.set_attribute("rows_written", df_silver.count())
```

### ⑤ 保持期間とコスト最適化

#### 推奨保持期間

| データ | Hot (即時検索可) | Cool (低速) | Cold (Archive) | 法定/監査 |
|-------|---------------|-----------|--------------|---------|
| Application Logs (DEBUG) | 7 days | 30 days | 削除 | - |
| Application Logs (INFO/WARN) | 30 days | 90 days | 1 year | - |
| Audit Logs (Security) | 90 days | 1 year | 7 years | J-SOX 7 年 |
| Metrics | 30 days | 13 months | (集約のみ) | - |
| Traces | 7 days | 30 days | (Sampled のみ) | - |
| LLM 推論結果 (品質 RAG 評価用) | 30 days | 90 days | 削除 | - |

#### Azure Monitor / Log Analytics コスト最適化

```yaml
cost_optimization:
  workspace_tier: "Pay-As-You-Go" (大きくなれば Commitment Tier)
  data_collection_rules:
    - tableName: "AppTraces"
      filter: "Severity != 'Verbose'"  # DEBUG 除外
    - tableName: "AppRequests"
      sample: 100% (全件)
  archival:
    - 90 days 経過したログを Cold Tier (1/8 コスト)
  retention_per_table:
    AppTraces: 30
    AppRequests: 90
    SecurityEvent: 730 (2 年)
```

### ⑥ 統合ダッシュボード設計

#### 推奨レイヤ

```
Top Level: Service Health (1 画面)
  ├── SLO 達成度
  ├── アクティブインシデント
  └── トラフィック概況

Drill-down 1: Service Detail
  ├── RED Metrics (Rate/Errors/Duration)
  ├── ログ最新 100 件 (filter: ERROR)
  └── 進行中 Trace のサマリ

Drill-down 2: Trace Search
  ├── trace_id / interaction_id で検索
  └── Span Tree 表示

Drill-down 3: Resource (USE)
  ├── CPU / Memory / Disk
  └── Container Apps Replica 状態
```

#### Genesys / Databricks / SAP の統合戦略

```yaml
saas_integration:
  - vendor: Genesys Cloud
    available_data:
      - API Audit Trail (X-CorrelationId)
      - Webhook delivery log
    integration:
      - Webhook 受信時に X-CorrelationId を Span Attribute へ
      - 自社の trace_id をマッピングログに記録
  - vendor: Databricks
    available_data:
      - System Tables (system.access.audit)
      - Job Run Logs
    integration:
      - Notebook 内で OpenTelemetry SDK 利用
      - System Tables を Log Analytics に Cross-Workspace Query
  - vendor: SAP S/4HANA Cloud
    available_data:
      - IDOC ログ, AIF (Application Interface Framework)
    integration:
      - 中継キュー (Service Bus) に trace_id を property
      - SAP 側エラーは IDOC 番号 → trace_id 紐付け
  - vendor: Azure OpenAI
    available_data:
      - Diagnostic Settings (Token usage, Latency)
    integration:
      - Diagnostic を Log Analytics へ送信
      - request-id を OTel Attribute と紐付け
```

## 例示: コールセンター VoC 基盤の Observability 構成

| 柱 | 採用 | 保持 | 備考 |
|----|------|----|------|
| Logs | Log Analytics Workspace + OTel | INFO 30d / Audit 7y | 環境別 Workspace |
| Metrics | Azure Monitor Metrics + Application Insights | 13 months | RED + USE |
| Traces | Application Insights (OTel 経由) | Sampled 30d | tail-based sampling |
| Profiles | Application Insights Profiler (Daily) | 30d | 本番 0.1% |

```yaml
example_obs_stack:
  application_insights:
    resources:
      - app-insights-prod-app
      - app-insights-prod-databricks
    connection: Workspace-Linked (Log Analytics)
  log_analytics:
    workspaces:
      - law-prod-app  (Pay-As-You-Go, 90d)
      - law-prod-sec  (Pay-As-You-Go, 730d)
  otel_collector:
    deployment: AKS / Container Apps Sidecar
    pipeline:
      receivers: [otlp, fluentforward]
      processors: [batch, attributes/redact, tail_sampling]
      exporters: [azuremonitor, otlphttp]
  trace_id_pipeline:
    Genesys Webhook (X-CorrelationId)
      ↓ map
    自社 API (trace_id 採番)
      ↓ propagate
    Postgres write (DB column)
      ↓ outbox
    Event Hubs (Property)
      ↓
    Databricks (Notebook OTel Span)
      ↓
    Gold table (audit row)
```

## 作成時のチェックリスト

- [ ] OpenTelemetry を採用 (ベンダーロックイン回避)
- [ ] OTel Collector を中継 (バックエンド切替容易性)
- [ ] 構造化ログのスキーマ (必須フィールド) が定義済み
- [ ] PII を含めないルールがあり、Collector で再チェックされる
- [ ] ログレベル運用ガイドが文書化されている
- [ ] メトリクスが RED + USE で揃っている
- [ ] LLM / RAG パイプライン用の専用メトリクスがある
- [ ] W3C Trace Context (traceparent) で Trace ID が伝播される
- [ ] Tail-based サンプリングで重要 Trace は 100% 保存
- [ ] Genesys / Databricks / SAP / Azure OpenAI と Trace ID マッピングが設計済み
- [ ] 保持期間が法定要件 (J-SOX 7y, 個人情報) を満たす
- [ ] コスト最適化 (DEBUG 除外、Archive Tier) が設定されている
- [ ] 統合ダッシュボードに SLO / RED / USE が一覧表示される
- [ ] ADR (OTel 採用、保持ポリシー) が起票済み
