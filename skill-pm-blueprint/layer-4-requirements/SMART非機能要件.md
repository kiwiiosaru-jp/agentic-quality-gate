# SMART 非機能要件 (SMART NFR Elicitation)

## 概要

非機能要件 (NFR) は機能要件より軽視されがちだが、プロジェクト失敗の半数以上は NFR の未定義・未達成が原因。

Olaf Zimmermann が Design Practice Repository (DPR) で提唱した SMART NFR Elicitation は、以下を特徴とする:

- **SMART**: Specific / Measurable / Achievable / Relevant / Time-bound
- **ランディングゾーン**: 最小 (Minimum) / 目標 (Target) / 卓越 (Outstanding) の 3 水準
- **カテゴリ別質問リスト**: FURPS+ に沿って漏れを防ぐ

「良い NFR」は **測定可能** で、**ランディングゾーンで柔軟性を持つ** (最小さえ満たせば失敗ではない)。

## いつ使うか

- ユースケース抽出後、各機能に NFR を紐付ける時
- アーキ決定時に性能・可用性要件を具体化する時
- コスト見積時に性能要件から規模試算する時
- SLA/SLO 交渉時

## FURPS+ カテゴリ

- **F**unctionality — 機能 (本来 FR だが一部の機能品質はここに)
- **U**sability — 使いやすさ、UX
- **R**eliability — 信頼性 (可用性、障害復旧、正確性)
- **P**erformance — 性能
- **S**upportability — 保守性、拡張性、運用性、移植性
- **+** — 法的 (Legal)、文化的 (Cultural)、組織的 (Organizational)

## ランディングゾーンの考え方

1 つの NFR につき 3 水準を定義:

| 水準 | 意味 |
|-----|------|
| 最小 (Minimum) | これを下回ると失敗。撤退レベル |
| 目標 (Target) | このプロジェクトで達成したい水準 |
| 卓越 (Outstanding) | これ以上は投資対効果が悪くなる上限 |

利点:

- 要件交渉で「最小は譲れない」「目標との差は予算次第」と議論可能
- トレードオフ分析が楽になる (性能 vs コスト等)
- プロジェクト中盤でスコープ調整する際の指針になる

## カテゴリ別 質問リストとランディングゾーン例

### 1. Usability (使いやすさ)

**質問リスト:**
- [ ] 利用者の前提スキル (SQL, BI, Excel) は?
- [ ] 初回ログインから意思決定までの想定ステップ数は?
- [ ] UI 言語は?
- [ ] オンボーディング時間の上限は?
- [ ] 多デバイス対応 (デスクトップ / タブレット / モバイル)?

**例:** マーケ分析担当者のセグメント抽出時間

| 水準 | 定義 |
|-----|------|
| 最小 | 初回実行 60分 / 2回目以降 20分 以内 |
| 目標 | 初回実行 30分 / 2回目以降 10分 以内 |
| 卓越 | 初回実行 10分 / 2回目以降 5分 以内 |

### 2. Reliability (信頼性)

**質問リスト:**
- [ ] 可用性の目標 (パーセント、業務時間帯限定 or 24h)?
- [ ] 復旧時間目標 (RTO) と復旧ポイント目標 (RPO)?
- [ ] データ品質 (完全性、一意性、整合性) の目標値?
- [ ] 障害時のフォールバック経路?
- [ ] バックアップ戦略と保持期間?

**例:** Gold 層への日次バッチの成功率

| 水準 | 定義 |
|-----|------|
| 最小 | 月次成功率 95% 以上 |
| 目標 | 月次成功率 99% 以上 |
| 卓越 | 月次成功率 99.9% 以上 |

**例:** BIダッシュボード可用性 (業務時間帯)

| 水準 | 定義 |
|-----|------|
| 最小 | 98.0% (月間ダウンタイム 3.6h 以内) |
| 目標 | 99.5% (月間ダウンタイム 55min 以内) |
| 卓越 | 99.9% (月間ダウンタイム 6min 以内) |

### 3. Performance (性能)

**質問リスト:**
- [ ] BIクエリの応答時間 (中央値、95/99 パーセンタイル)?
- [ ] 同時利用ユーザ数の想定?
- [ ] バッチジョブの完了時間 (日次、月次)?
- [ ] スループット (秒間トランザクション)?
- [ ] データ量成長時の劣化許容度?

**例:** BIダッシュボード初期表示時間 (95パーセンタイル)

| 水準 | 定義 |
|-----|------|
| 最小 | 10秒以内 |
| 目標 | 3秒以内 |
| 卓越 | 1秒以内 |

**例:** 夜間バッチ (全Gold層)

| 水準 | 定義 |
|-----|------|
| 最小 | 朝7時までに完了 (オープンに間に合う) |
| 目標 | 朝5時までに完了 (運用チームの点検時間確保) |
| 卓越 | 朝3時までに完了 (障害時のリトライ余裕あり) |

### 4. Security (セキュリティ)

**質問リスト:**
- [ ] 認証方式 (SSO, MFA)?
- [ ] 認可モデル (RBAC, ABAC, RLS)?
- [ ] データ暗号化 (保存時, 転送時)?
- [ ] 監査ログ保持期間?
- [ ] 特権アクセス管理 (PAM)?
- [ ] 脆弱性対応 SLA?
- [ ] データ損失時の検知と対応?

**例:** 監査ログの保持期間

| 水準 | 定義 |
|-----|------|
| 最小 | 90日 (社内ポリシー最低ライン) |
| 目標 | 1年 |
| 卓越 | 7年 (金融・医療相当) |

### 5. Maintainability (保守性)

**質問リスト:**
- [ ] コードカバレッジ?
- [ ] ドキュメント整備率?
- [ ] 平均故障修理時間 (MTTR)?
- [ ] 新規参入者のオンボーディング期間?
- [ ] 技術的負債の許容率?

**例:** dbt モデルのテストカバレッジ

| 水準 | 定義 |
|-----|------|
| 最小 | Silver層以降 60% 以上 |
| 目標 | Silver層以降 80% 以上、Gold層 100% |
| 卓越 | 全層 95% 以上 + カスタムテスト |

### 6. Scalability (拡張性)

**質問リスト:**
- [ ] 3年後のデータ量/クエリ量の想定倍率?
- [ ] 新データソース追加時のリードタイム?
- [ ] スケールアウト/アップ戦略?
- [ ] ピーク時対応方針?

**例:** 新データソース追加のリードタイム

| 水準 | 定義 |
|-----|------|
| 最小 | 3ヶ月 (要件〜本番) |
| 目標 | 1ヶ月 |
| 卓越 | 2週間 |

### 7. Operability (運用性)

**質問リスト:**
- [ ] 監視・アラート設計?
- [ ] ランブック整備?
- [ ] 自動復旧の範囲?
- [ ] オンコール体制?
- [ ] 障害対応の平均時間?

**例:** アラート検知から一次対応着手までの時間

| 水準 | 定義 |
|-----|------|
| 最小 | 業務時間内 30分以内 / 業務時間外 翌朝9時 |
| 目標 | 業務時間内 15分以内 / 業務時間外 4時間以内 |
| 卓越 | 24時間 15分以内 |

### 8. Portability (移植性)

**質問リスト:**
- [ ] クラウドベンダー変更の可能性?
- [ ] モデル/ライブラリ変更の必要性?
- [ ] データ形式の標準化?

**例:** DWH 移行作業の許容期間

| 水準 | 定義 |
|-----|------|
| 最小 | 12ヶ月 |
| 目標 | 6ヶ月 |
| 卓越 | 3ヶ月 (抽象化層あり) |

### + 法的 (Legal)

**質問リスト:**
- [ ] 個人情報保護法への準拠?
- [ ] 業種別規制 (金融, 医療)?
- [ ] GDPR / CCPA の適用?
- [ ] 電子帳簿保存法?
- [ ] データの国内保管要否?

**例:** データの国内保管

| 水準 | 定義 |
|-----|------|
| 最小 | PII のみ国内 |
| 目標 | 全データ国内 |
| 卓越 | 全データ国内 + 地理冗長 |

## 出力フォーマット

NFR 一覧は以下の Markdown 表で保存:

| NFR ID | カテゴリ | 内容 | 最小 | 目標 | 卓越 | 測定方法 | 関連UC |
|--------|---------|-----|------|------|------|---------|-------|
| NFR-001 | Performance | BIダッシュボード初期表示 (P95) | 10秒 | 3秒 | 1秒 | Application Insights | UC-001, UC-003 |
| NFR-005 | Reliability | BI可用性 (業務時間帯) | 98% | 99.5% | 99.9% | SLA監視 | UC-001〜005 |
| NFR-010 | Security | 認証 | SSO | SSO + MFA | SSO + MFA + 特権PAM | 監査 | 全UC |

## 作成時のチェックリスト

- [ ] FURPS+ の 8 カテゴリを漏らしていないか
- [ ] 各NFRに測定方法が明記されているか
- [ ] 各NFRにランディングゾーン 3 水準があるか
- [ ] 各ユースケースに最低 3 つのNFRが紐付いているか
- [ ] 最小水準を下回った場合の対応 (Kill基準とリンク) があるか
- [ ] ステークホルダー合意が得られているか
