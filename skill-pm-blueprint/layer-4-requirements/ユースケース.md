# ユースケース抽出 (Use Case 2.0 slim)

## 概要

Ivar Jacobson らが 2011 年に提唱した Use Case 2.0 は、従来の重厚なユースケース記法を**軽量化**した形式。
スライス (Slice) という概念で、1 ユースケースを複数の独立実装可能な断片に分割できる。

本スキルではさらに簡略化した **slim 版**を採用:

- 1 ユースケース = 主シナリオ + 代替シナリオ + 例外シナリオ
- 1 ユースケース = 1 ページ以内
- 詳細設計書ではなく **「何が実現されるか」を合意する文書**

## いつ使うか

- Layer 2 のディスカバリー結果を機能に落とし込むとき
- Layer 5 の脅威モデリング前 (データフローの入力として)
- Layer 6 の WBS L3 アクティビティを決める前
- スプリントチケット (ユーザーストーリー) への分解前

## 標準構造 (slim版)

```markdown
# UC-NNN: [ユースケース名]

- アクター: [主アクター / 副アクター]
- 前提条件: [開始時点で成立している状態]
- 事後条件: [正常終了時点で成立している状態]
- トリガー: [ユースケースを開始させるイベント]

## 主シナリオ (Happy Path)
1. [アクターが何をする]
2. [システムが何を返す]
3. ...

## 代替シナリオ (Alternative Flow)
2a. [代替経路1]
   - 2a-1. 処理
   - 2a-2. 主シナリオのステップ3へ戻る

## 例外シナリオ (Exception Flow)
2E. [エラー条件が発生]
   - 2E-1. エラー表示
   - 2E-2. ユースケース終了

## 関連NFR
- NFR-001 (性能): 応答時間 3秒以内
- NFR-005 (セキュリティ): 認証必須
- NFR-010 (監査): 全操作をログ記録

## 関連ADR
- ADR-0001 (DWH選定)

## 関連リスク
- R003 (PII漏洩)

## 備考
- [前提、制約、補足]
```

## ユースケーススライスの概念

1 つのユースケースは複数のスライスに分割可能:

```
UC-003: 売上KPIダッシュボード閲覧
├── Slice 1: 月次KPIの表示 (最小実装 = MVP)
├── Slice 2: 店舗別比較の表示
├── Slice 3: ドリルダウン (日次への掘り下げ)
└── Slice 4: 前年同月比の色分け表示
```

スライスは**独立して実装・リリース可能**な単位。Sprint チケットの粒度に相当する。

## 書き方のコツ

### コツ 1: 主アクターは「役割」で書く

- 悪い: 山田さん、A さん
- 良い: 経営層、マーケ分析担当、店舗運営担当

### コツ 2: ステップは 5〜9 個に収める

10 を超えたら複数ユースケースに分割するサイン。

### コツ 3: 「システムは」で始めて受動的に書かない

- 悪い: 「データが Gold 層に格納される」 (主語が曖昧)
- 良い: 「システムが Gold 層に集計結果を格納する」

### コツ 4: 代替と例外を分ける

- 代替: 業務上の別経路 (例: 権限違い、データ有無)
- 例外: エラー (例: システム障害、入力不正)

### コツ 5: NFR と紐付ける

主要 NFR を必ずリストアップする。紐付かない機能要件は、NFR の見落としのサイン。

## データ分析基盤での例示

### 例 1: UC-001 経営層向けKPIダッシュボード閲覧

```markdown
# UC-001: 経営層向けKPIダッシュボード閲覧

- アクター: 経営層 (役員)
- 前提条件: Entra ID で認証済み
- 事後条件: 最新のKPIデータが画面に表示されている
- トリガー: 経営層がダッシュボードURLにアクセス

## 主シナリオ
1. 経営層がダッシュボードURLにアクセス
2. システムがEntra ID認証を確認
3. システムがロールを判定 (Executive ロール)
4. システムが当月のKPI (売上/粗利/会員数) を Gold 層から取得
5. システムがKPIカードと前年同月比トレンドを表示
6. 経営層が前月比・前年比を確認して離脱

## 代替シナリオ
3a. 経営層でないロール判定
   - 3a-1. システムが閲覧権限なしメッセージを表示
   - 3a-2. ユースケース終了

## 例外シナリオ
4E. Gold 層への接続エラー
   - 4E-1. システムがキャッシュ (前日版) を表示
   - 4E-2. 画面上部にデータ鮮度表示 (「前日データを表示中」)

## 関連NFR
- NFR-001 (性能): 初期表示 3秒以内、95パーセンタイル
- NFR-005 (可用性): 99.5% 以上 (業務時間帯)
- NFR-010 (セキュリティ): Entra ID認証必須、RLS適用
- NFR-015 (監査): 閲覧ログを90日保持

## 関連ADR
- ADR-0001 (DWH選定)
- ADR-0005 (認証基盤選定)

## 関連リスク
- R003 (PII漏洩 - 会員数表示時)

## 備考
- Slice 1 (MVP): 当月KPIの表示のみ
- Slice 2: 前年同月比の色分け
- Slice 3: 店舗別ドリルダウン
```

### 例 2: UC-003 マーケ分析担当のセグメント抽出

```markdown
# UC-003: マーケ分析担当のセグメント抽出

- アクター: マーケ分析担当
- 前提条件: BIツールにアクセス可能、Analyst ロール付与済み
- 事後条件: 抽出結果が CSV ダウンロード可能な状態
- トリガー: キャンペーン対象顧客の抽出依頼が発生

## 主シナリオ
1. 担当者がセグメント定義画面を開く
2. システムが利用可能な属性一覧を表示
3. 担当者が属性条件を複数設定 (年齢、直近購入、会員ランク等)
4. 担当者が「プレビュー」をクリック
5. システムがGold層の顧客マートから該当件数を返す
6. 担当者が件数を確認し「CSV出力」をクリック
7. システムが同意取得済み顧客のみフィルタしてCSVを生成
8. 担当者がCSVをダウンロード

## 代替シナリオ
3a. 担当者が保存済みセグメントを呼び出し
   - 3a-1. システムが過去のセグメント一覧を表示
   - 3a-2. 担当者が選択して主シナリオのステップ4へ

## 例外シナリオ
5E. 該当件数が0件
   - 5E-1. システムが「該当顧客がいません」を表示
   - 5E-2. ユースケース終了

7E. 同意未取得顧客が含まれる
   - 7E-1. システムが対象外件数を警告表示
   - 7E-2. 担当者は続行 or キャンセル

## 関連NFR
- NFR-001 (性能): プレビュー 10秒以内、CSV生成 5分以内
- NFR-010 (セキュリティ): Analyst ロール必須
- NFR-020 (法的): 同意取得済み顧客のみ出力
- NFR-025 (監査): CSV出力を監査ログに記録

## 関連ADR
- ADR-0001, ADR-0003

## 関連リスク
- R003 (PII漏洩)
- R021 (セグメント定義の誤り)

## 備考
- Slice 1 (MVP): 基本属性5種のみ
- Slice 2: 購買履歴ベース条件の追加
- Slice 3: 保存済みセグメントの呼び出し
```

### 例 3: UC-007 データエンジニアによる dbt モデル本番リリース

```markdown
# UC-007: データエンジニアによる dbt モデル本番リリース

- アクター: データエンジニア
- 前提条件: Pull Request がレビュー承認済み、CI が全て合格
- 事後条件: 本番環境で新モデルが稼働、lineage が更新されている
- トリガー: PR が main ブランチにマージされる

## 主シナリオ
1. データエンジニアが PR をマージ
2. CI/CDパイプラインが起動
3. システムがdbt-core を実行 (本番ターゲット)
4. システムがテスト (not_null, unique, referential integrity) を実行
5. システムが lineage を更新
6. システムが Slack へ成功通知を送る

## 代替シナリオ
4a. テスト失敗
   - 4a-1. システムがデプロイを中断
   - 4a-2. システムが Slack へ失敗通知
   - 4a-3. 担当者が修正PR作成

## 例外シナリオ
3E. dbt 実行タイムアウト (2時間超)
   - 3E-1. システムが実行停止
   - 3E-2. 運用チームにエスカレーション

## 関連NFR
- NFR-003 (性能): 全モデル実行 2時間以内
- NFR-015 (監査): 全実行履歴を 1年保持
- NFR-030 (運用): 失敗時は自動ロールバック

## 関連ADR
- ADR-0002 (dbt 選定)

## 関連リスク
- R009 (dbt 実行失敗による BI 停止)

## 備考
- Slice 1: 手動実行
- Slice 2: CI/CD 自動化
- Slice 3: 失敗時の自動ロールバック
```

## 合意形成のヒント

ユースケースはアーキテクトではなく**業務部門と一緒に書く**ものであり、単独で書き上げてレビューを依頼するのは反パターン。

推奨進め方:

1. PM + 業務部門で「業務の流れ」をホワイトボードに書く
2. その場で主シナリオを文章化
3. 「他にこんなケースありますか?」で代替・例外を追加
4. テクニカルな部分 (NFR, ADR, リスク) は後から追記
